## Context

The current Mermaid experience is split across two frontend paths: `frontend/src/components/MarkdownRenderer.vue` renders Mermaid immediately after Markdown conversion, while `frontend/src/composables/useMermaid.ts` scans the DOM and renders `.mermaid` nodes through a separate queue-based pipeline. These paths use different Mermaid themes, different font stacks, different syntax-repair logic, and only one path applies SVG post-processing to prevent clipped labels. The backend also instructs the LLM to wrap every label in double quotes, which increases the chance of invalid nested quoting in math-heavy node text.

This change is cross-cutting because the visible bug is a product of three layers working against each other: Markdown-to-HTML conversion, Mermaid rendering and SVG adjustment in the frontend, and Mermaid authoring guidance in the backend prompt. The imported regression case in `tests/test_md_import.md` demonstrates all of these failure modes together: raw Markdown headings leaking into the rendered view, formula labels rewritten into incorrect quoted text, and Chinese or mixed-width labels clipping at the node boundary.

## Goals / Non-Goals

**Goals:**
- Establish one canonical Mermaid sanitization and rendering pipeline for Markdown-rendered content.
- Ensure Mermaid blocks do not break surrounding Markdown parsing or cause heading/source text to leak into the UI.
- Preserve formula intent when repairing invalid Mermaid label syntax, especially for nested quoting around math expressions.
- Apply consistent SVG safety adjustments so multilingual labels render without clipping in both immediate and deferred render paths.
- Add regression tests that cover the known imported Markdown document and representative Mermaid edge cases.
- Reduce invalid Mermaid output from the backend by refining prompt guidance for quoted labels and mathematical expressions.

**Non-Goals:**
- Redesign the visual theme of Mermaid diagrams beyond what is required for consistency and readability.
- Add support for new Mermaid diagram families or custom diagram extensions.
- Rewrite the entire Markdown rendering stack or imported-document pipeline.
- Guarantee perfect recovery for every malformed Mermaid snippet generated by users or LLMs; the goal is safe, predictable repair for known patterns.

## Decisions

### Decision: Centralize Mermaid code cleanup and SVG post-processing in a shared frontend utility
The frontend currently duplicates label-repair logic in `MarkdownRenderer.vue` and `useMermaid.ts`, which causes the two paths to diverge over time. We will extract a shared Mermaid utility module responsible for code normalization, syntax repair, and SVG post-processing. Both direct renderer and queue-based scanning will call the same helpers.

Why this over keeping both implementations in sync manually:
- Shared logic prevents the current drift in cleanup behavior and safety margins.
- The bug is specifically caused by subtle behavioral mismatches, so duplication is the wrong abstraction boundary.
- A utility can be tested directly without mounting full components for every case.

Alternative considered: remove `useMermaid.ts` entirely and rely only on `MarkdownRenderer.vue`.
- Rejected because deferred scan/queue behavior may still be needed for lazily inserted or intersected content elsewhere in the app.
- Unifying behavior is lower-risk than collapsing the rendering lifecycle in one step.

### Decision: Replace destructive quote substitution with targeted syntax repair rules
Current repair logic converts inner double quotes to single quotes, which makes invalid Mermaid code render but changes math notation from `cos("ωt - kz")` intent into visible single-quoted arguments. The new repair rules should prefer preserving parentheses and plain text semantics, only escaping or rewriting the minimum needed to produce valid Mermaid labels.

Why this over the current blanket rewrite:
- The rendered result must remain faithful to the source meaning, not merely syntactically accepted by Mermaid.
- Math-heavy educational content is common in this project, so visual correctness matters as much as render success.

Alternative considered: do no repair and fail fast on malformed Mermaid.
- Rejected because imported AI-generated content already contains common malformed patterns, and a hard failure would regress user experience substantially.

### Decision: Apply SVG overflow and sizing safeguards in the canonical render pipeline
The safety-margin behavior in `MarkdownRenderer.vue` addresses a real Mermaid limitation for Chinese and mixed-width labels, but it is not applied in `useMermaid.ts`. We will standardize post-processing so every rendered Mermaid SVG gets the same overflow visibility and width adjustments, with tunable constants based on the longest observed clipping cases in the regression fixture.

Why this over pure CSS fixes:
- The clipping is caused by Mermaid's calculated SVG geometry, not only container CSS.
- Updating the generated SVG viewBox and node widths addresses the problem closer to the source.

Alternative considered: add only CSS like `overflow: visible` on wrappers.
- Rejected because wrapper-level overflow does not expand clipped node rectangles or truncated SVG width calculations.

### Decision: Narrow backend Mermaid authoring guidance to avoid nested quoted labels
The backend prompt should continue enforcing valid Mermaid syntax, but it must stop encouraging patterns that almost guarantee frontend repair. Guidance should explicitly discourage nested unescaped double quotes in node labels, recommend plain parentheses for formulas, and prefer simple text labels when the content includes equations.

Why this over fixing everything solely in the frontend:
- Preventing malformed source reduces repair complexity and future edge cases.
- The rendering issue is systemic, and backend prompt quality is one of the causes.

Alternative considered: leave prompts unchanged and rely on tests.
- Rejected because prompt guidance is a cheap, high-leverage fix and belongs in the proposal scope.

### Decision: Add regression tests at both unit and rendered-output levels
Existing tests cover several Mermaid edge cases in `frontend/src/__tests__/utils/markdown.test.ts`, but they currently assert the old single-quote rewrite behavior. We will update expectations to match semantically correct output and add a regression around the imported Markdown fixture or an extracted representative snippet from `tests/test_md_import.md`.

Why this over fixture-only manual verification:
- The bug is easy to reintroduce because it spans Markdown parsing, Mermaid repair, and SVG post-processing.
- Automated assertions are the only reliable way to lock down both syntax repair and clipping prevention.

## Risks / Trade-offs

- [Repair rules still miss uncommon malformed Mermaid patterns] -> Mitigation: scope repairs to known high-frequency patterns, log render failures, and add targeted regression cases before broadening the parser.
- [Shared utility refactor breaks lifecycle assumptions in one rendering path] -> Mitigation: keep public behavior unchanged, migrate both call sites incrementally, and retain queue semantics in `useMermaid.ts`.
- [Larger SVG safety margins introduce extra whitespace around diagrams] -> Mitigation: use conservative constants validated against the known clipping fixture and prefer node-width adjustments over excessive global padding.
- [Prompt changes alter LLM Mermaid output style in unrelated diagrams] -> Mitigation: keep the guidance narrow and syntax-focused, then validate with existing Mermaid rendering tests.
- [Imported Markdown regression is caused partly by upstream Markdown formatting quality] -> Mitigation: explicitly test both well-formed Mermaid blocks and the malformed import cases that the product must tolerate.

## Migration Plan

- Implement the shared Mermaid utility and switch both frontend rendering entry points to it behind existing component/composable boundaries.
- Update backend Mermaid prompt guidance in the same change so newly generated content aligns with the repaired frontend behavior.
- Refresh frontend regression tests to assert the new canonical output.
- Verify the known imported fixture renders correctly in local UI testing before shipping.
- Rollback strategy: revert the shared utility adoption and prompt edits together if rendering failures increase; the prior component-level renderer can still serve as the fallback baseline.

## Open Questions

- Should the imported Markdown regression be covered through a full component fixture test using `tests/test_md_import.md`, or through smaller extracted snippets to keep test runtime stable?
- Should the canonical Mermaid configuration keep the current default theme behavior from `MarkdownRenderer.vue` or the custom theme variables from `useMermaid.ts` once the pipeline is unified?
