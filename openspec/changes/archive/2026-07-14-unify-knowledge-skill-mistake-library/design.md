# 设计：一棵知识树贯穿课程与学习闭环

## 1. 统一数据真源

每个学科只发布一个版本化知识库包：

```text
知识库包
├── 知识节点与知识关系
├── 能力点
│   ├── 易错点
│   └── 提升点
└── 使用策略与来源
```

存储可以继续采用规范化集合，避免在每个知识节点中复制完整子对象，但所有集合必须位于同一个文件、共享同一个 `library_id`、版本和修订。读取视图必须恢复清晰父子关系。

### 1.1 父子约束

- 能力点必须通过 `primary_knowledge_id` 指向一个正式概念或细知识点；`knowledge_ids` 只表示该能力覆盖的更多细知识，不产生第二父级。
- 易错点必须通过 `skill_unit_id` 指向一个能力点。
- 提升点必须通过 `skill_unit_id` 指向一个能力点，不再允许只挂知识节点。
- 易错点和提升点可以继续引用相关知识与易错条目，但这些引用只作语义关联，不改变父子归属。

## 2. 课程生成接入

```text
用户目标与资料
→ 课程局部知识要求
→ 与统一知识库精确匹配
→ 读取命中知识下的能力、易错和提升切片
→ 生成学习目标、课程块、正式题目、诊断与补救资产
→ 质量门检查覆盖与引用合法性
```

正式知识库先于单门课程存在。课程生成只能提出局部表达并建立映射；无法精确命中的表达保留为待归一候选，不能直接写入正式知识库。

## 3. 学习证据接入

正式题目保存目标知识、目标能力和候选易错引用。创建作答时把这些引用复制到作答事实，后续事件、诊断和补救继续携带：

```text
题目引用
→ PracticeAttempt
→ LearningEvent
→ DiagnosticCase / RemediationSession
→ LearnerModel 投影
→ AI 老师最小上下文
```

候选易错只表示题目希望辨析的标准条目，不表示学生已经命中。只有诊断证据确认后，才将对应易错作为已确认或待关注信号；一次答错仍不能形成稳定薄弱结论。

## 4. 学习者模型

模型继续以正式事实为唯一来源，并新增两类可审计投影：

- 能力状态：按能力点聚合同一目标下的正式作答与独立复验，不直接修改知识掌握。
- 易错信号：只从诊断案例的确认状态或多次独立证据形成，保留证据引用和置信度。

知识状态、能力状态和易错信号均为个人覆盖层，不写回统一知识库。

## 5. AI 老师

AI 上下文只装配当前节点相关的最小知识库切片，并保持“先独立判断，再映射标准名称”的原则：

- 命中：引用正式能力或易错 ID。
- 部分命中：只绑定到能够确认的父知识或能力。
- 未命中：保留自然语言判断，不伪造 ID，后续进入候选治理。

## 6. 前端体验

学生仍从现有知识库覆盖层进入，不增加第二网站或第二入口。左侧只展示正式知识树；选择知识节点后，右侧按能力分组：

```text
知识点详情
├── 能力点 A
│   ├── 易错点
│   └── 提升点
├── 能力点 B
│   ├── 易错点
│   └── 提升点
└── 课程正文、题目与掌握证据
```

课程目录继续负责教学顺序，知识库负责跨课程语义，二者只通过课程知识映射关联。

## 7. 迁移与兼容

- 将现有线性代数学科文件与教学标准文件机械合并为统一知识库版本，不改变稳定条目 ID。
- 历史课程中的 `subject_knowledge` 与 `teaching_standards` 只在兼容读取时合成为 `knowledge_library`，新编译结果不再写旧字段。
- 删除独立教学标准加载模块后，生产代码必须统一从知识库模块读取切片。
- 兼容层不得拥有独立版本、缓存、校验、提示词或保存逻辑。
