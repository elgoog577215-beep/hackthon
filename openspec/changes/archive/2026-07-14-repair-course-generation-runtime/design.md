# 设计：按唯一任务状态对齐启动、生成与发布

## 1. 以真实链路划分问题

```text
本地运行环境
  -> GenerationJob 创建
  -> 正常运行 / 检查点恢复
  -> 正文与质量闸门
  -> 正式 CourseDocument 发布
  -> 前端任务状态 + 课程库摘要 + 学习页
```

本次不触碰已经通过真实生成验证的课程生产算法，只修复链路两端和状态投影。后端任务、隔离工作区和正式课程文档仍是唯一真源。

## 2. 恢复语义

`status=pending/running` 只表示任务当前活跃，不能推出它正在恢复。只有任务持有 `last_recovery_reason=service_restart/manual_resume` 时，恢复描述才返回 `auto_resuming`；普通新任务返回 `state=none`，前端按任务状态显示“等待生成”或“正在生成”。

质量阻塞、运行失败、人工暂停和正式完成继续沿用现有独立状态，不改变恢复权限。

## 3. 发布后的前端对账

新建任务首先创建空的正式课程外壳，因此课程库会短暂获得 `node_count=0`。正式发布后，前端必须同时更新：

- 当前课程正文：读取正式 `CourseDocument`；
- 课程库摘要：重新读取 `/api/courses`；
- 任务终态：继续读取同一 `job_id`。

WebSocket 完成事件和 HTTP 轮询完成迁移都执行同一对账动作，以覆盖断线与多课程并发。

## 4. 本地启动契约

`dev.sh` 只负责编排已有运行环境，不在每次启动时静默修改依赖。脚本应：

- 从自身目录解析项目根目录；
- 校验 `.env` 与 `AI_API_KEY`；
- 优先使用 `backend/.venv/bin/python`，并验证必要 Python 包；
- 验证前端依赖目录；
- 通过可配置端口启动后端和前端，并把 Vite 代理指向当前后端端口；
- 等待两个健康端点成功，任一子进程退出则整体失败；
- 收到退出信号时清理两个子进程。

## 5. AI 日志边界

provider 返回的 `reasoning_content` 不属于课程产物，也不应进入普通运行日志。LLM 聚合调用只累计思考字符数，并在 debug 级别记录长度；课程正文仍只读取 `delta.content`。
