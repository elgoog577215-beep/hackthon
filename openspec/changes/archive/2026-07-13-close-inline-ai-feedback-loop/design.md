# 设计：把理解、反馈与正式编辑分成三条清晰路径

## 1. 交互角色

同一个课程块附近允许存在两种能力，但不得混成同一菜单：

- 左侧星光入口属于学生理解，只包含解释、举例、简化和提问，结果进入来源块下方的临时个人补充。
- 右上角课程编辑入口只在 `canImproveBlocks` 为真且目标是正式 `CourseBlock` 时出现，进入既有正文候选工作区。
- 全局 AI 继续承担会话历史、跨块问题和低频兜底，复用同一会话与上下文协议。

这不是增加三个 AI 产品，而是让同一 AI 在不同权限和结果边界下使用不同入口。

## 2. 行内回答结构

行内回答保持正文型补充，而不是聊天卡片：

```text
临时个人内容 · 来源：当前块标题
AI · 解释
回答正文
这次回答解决你的问题了吗？  [已解决] [还不清楚]
继续追问 / 保留为个人内容 / 移除
```

来源信息与动作标题放在同一紧凑标题区。继续追问替换当前行内表达，不堆叠多张卡片；新回答完成后反馈状态重新开始。

## 3. 反馈写入契约

前端通过 `aiTeacher` Store 调用：

```text
POST /api/ai-teacher/conversations/{conversation_id}/messages/{message_id}/feedback
```

请求只携带课程 ID、反馈值、块级动作和稳定内容锚点。后端必须：

1. 校验课程、用户、会话和目标消息归属。
2. 只接受已完成的 assistant 消息。
3. 将 `resolved / unclear` 写入 `assistant_answer_feedback_submitted` LearningEvent。
4. 使用消息 ID 做幂等保护；重复提交返回已有事件，不追加重复事实。

事件只表示用户对当前回答的显式判断。它可以供后续分析使用，但本变更不把它直接投影成掌握、诊断、补救或下一步。

## 4. 状态与失败边界

- 回答生成中或失败时不显示反馈。
- 提交中禁用两个选项，提交成功后保留选中态。
- 提交失败时显示轻量错误并允许重试，不移除回答、不自动改走其他动作。
- 停止生成、取消首次提问、继续追问、保留与移除继续沿用现有状态所有权。

## 5. 响应式与国际化

桌面端反馈问题与两项选择可在同一行，窄屏允许自然换行；按钮仍保持可点击尺寸。正式编辑入口在移动端收成图标按钮，但必须保留本地化 tooltip 和无障碍名称。所有新增用户可见文案同步维护中文与英文。
