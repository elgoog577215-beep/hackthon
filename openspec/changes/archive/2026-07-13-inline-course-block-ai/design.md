# 设计：一个内容块对应一个临时 AI 协作块

## 1. 信息结构

```text
CourseBlock
├── 左侧轻量 AI 入口
│   └── 解释 / 举例 / 简化 / 提问
├── 正式课程内容
└── InlineCourseBlockAI（按需出现）
    ├── 本轮 AI 回答
    ├── 同一位置的继续追问输入
    └── 继续追问 / 保留为个人内容 / 移除
```

正式 `CourseBlock` 始终是主内容。`InlineCourseBlockAI` 是当前学习现场的临时个人覆盖层，不加入 `CourseDocument`，不改变块顺序，也不成为新的课程真源。

## 2. 交互状态

- 默认：只显示低权重星光入口；桌面悬停或聚焦时增强，触屏保持可发现。
- 菜单：同一时间只展开一个块的四项菜单，选择动作后立即收起。
- 解释、举例、简化：打开结果块并开始流式回答。
- 提问：打开结果块中的输入区，用户提交后在原位置流式回答。
- 继续追问：重新打开同一结果块的输入区；新回答替换该块的当前回答表达，会话历史仍由统一 AI Store 保存。
- 保留：通过现有 `create_note` 提案与确认协议写入 `LearningRecord`，随后由 `InlineLearningRecordBlock` 投影。
- 移除：只移除本地临时结果表达；若当前回答仍在生成，则停止该次生成，不删除已经持久化的全局会话历史。

## 3. 上下文与状态所有权

- 请求使用当前 `course_id / course_version_id / node_id / block_id / block_revision_id` 作为稳定上下文引用。
- 块正文作为最小 `selection` 发送，避免默认装配整门课程。
- 流式消息、会话历史、来源和失败状态继续由 `aiTeacher` Store 管理。
- 行内组件只保存菜单、当前显示消息、追问输入和保留状态，不复制会话或课程状态。
- `LearningRecord` 仍是个人内容唯一真源；保存完成后重新读取记录列表。

## 4. 视觉与响应式

- 入口位于内容块左侧语义边缘，不占据标题或正文主阅读宽度。
- 菜单是窄而明确的纵向动作表，不做大卡片墙。
- 结果块使用左侧品牌色和浅紫底区分个人 AI 内容，但宽度、字号和正文节奏继承当前课程流。
- 窄屏入口回收到块标题上方，菜单不得伸出视口；结果操作与输入区允许换行。
- 所有图标按钮具备中文、英文 `title` 和 `aria-label`。

## 5. 边界

- 不显示系统推断的“下一步”。
- 不根据停留、滚动或一次错误主动弹出行内结果。
- 不自动保存回答；只有用户点击“保留为个人内容”才执行写动作。
- 不把“简化”解释为正式改写课程，正式课程修改继续走独立的课程领域命令与确认流程。
