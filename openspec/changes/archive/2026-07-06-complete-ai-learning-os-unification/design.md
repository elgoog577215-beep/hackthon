# AI Learning OS 九阶段归一化设计

## 总方法

这次工程采用“目标架构牵引式合并”：

```text
宏大目标
-> 理想型 AI Learning OS
-> 把旧链路当作养料
-> 归并成统一主链路
-> 让前端体验真实感知智能增强
-> 用测试和评估证明没有倒退
```

主链路固定为：

```text
用户行为
-> LearningEvent
-> LearnerState
-> TeachingDecision
-> LearningOSSnapshot / AI Learning Context
-> AI 助手 / AI 课件生成 / AI 导师 / 复习建议 / 前端画像统计
-> 结果继续回写 LearningEvent
```

任何新增功能都必须经过同一套门禁：

- 证据：它记录了什么真实学习行为？
- 状态：它如何影响学习者状态？
- 决策：它如何影响下一步教学动作？
- 感知：用户在哪里感受到 AI 更懂自己？
- 回写：执行结果如何进入下一轮闭环？
- 合并：它有没有和旧功能重复，是否应该包装、合并或废弃？

## 九阶段不是九个孤岛

九个阶段是同一条主链路的九个落点：

1. 学习证据闭环：让系统真实看见学习发生了什么。
2. 结构化课程 block：让课程从长文本变成可操作知识结构。
3. 画像、状态、痕迹合并：让系统形成统一学习者理解。
4. AI 导师主动引导：让 AI 在学习过程中低打扰地带路。
5. AI 课程生成升级：让课程生成更像教学设计，而不是长文输出。
6. 旧链路深度收敛：让后端和前端只保留一套大脑。
7. 质量评估系统：证明 AI 真的变聪明且没有倒退。
8. 前端体验精修：让智能从架构变成用户可感知体验。
9. 长期智能闭环：从单课工具升级为个人学习操作系统。

执行时允许阶段交叉，但方向不能乱。优先级按“主链路是否打通”决定，而不是按文件或页面决定。

## 旧功能归并矩阵

| 区域 | 旧价值 | 归并方向 |
| --- | --- | --- |
| `learning_events.py` | 事实账本 | 继续扩展事件类型，成为所有学习行为入口 |
| `learner_state.py` | 状态聚合 | 继续消费事件，不直接读取散落旧状态 |
| `teaching_decisions.py` | 教学意图 | 作为导师卡、复习、练习、AI 助手和局部课件生成的主建议源 |
| `learning_os.py` | 统一读模型 | 成为前端学习洞察、旧接口兼容和旧链路迁移的主契约 |
| `ai_learning_context.py` | AI prompt 编排 | 供 AI 助手、CourseService、导师建议复用 |
| `tutor_service.py` | 目标、错题、遗忘曲线 | 保留为 legacy signals，逐步让建议消费快照 |
| `ai_learning_service.py` | 学习路径、掌握度、复习算法 | 逐步消费 `LearningOSSnapshot`，避免重复推断弱点 |
| `learning_record.py` | 旧答题记录 | 保留兼容，新证据优先写入事件账本 |
| `CourseService` | 课程生成与局部内容操作 | 唯一课程 AI 生产入口，接入状态、决策和 block 操作事件 |
| `CourseNode.vue` | 结构化 block 展示雏形 | 保留并打通后端 block 操作、事件回写和 AI 助手动作 |
| `TutorActionCard.vue` | 主动行动入口 | 承载唯一主建议，不让统计/画像/助手各自生成主建议 |
| `LearnerProfile.vue` | 画像解释 | 展示事实证据和系统推断边界 |
| `LearningStats.vue` | 轨迹统计 | 展示过去发生了什么，不做主建议源 |
| `SideAIPanel.vue` | 对话入口 | 展示轻量状态摘要，回答由后端统一上下文驱动 |
| `reviewStore` | 本地错题/测验体验 | 保留快速交互，同时逐步同步后端事件账本 |

## 执行策略

- 战略上一次性推进九阶段，工程上分批提交。
- 每个提交必须是可验证切片，不能糊巨大不可回滚 diff。
- 优先改最能打通闭环的入口：复习提交、block 操作、学习路径/掌握度、导师行动。
- 旧接口保持可用，内部逐步改成新链路薄包装。
- 前端不新增重复中心，优先强化现有四层入口：行动、解释、轨迹、对话。
- 验证失败时先判断是否本轮引入；既有失败要记录边界，不能伪装通过。

## 阶段验收

### 阶段 1：学习证据闭环

- 复习提交写入事件账本。
- block 操作写入事件账本。
- 旧错题/测验历史有后端同步路径。
- `LearnerState` 能从新事件中看到复习和 block 证据。

### 阶段 2：结构化课程 block

- 后端稳定维护 `content_blocks` 与 `node_content`。
- 前端可以折叠、选择并操作 block。
- 简化、扩展、补例子、练习、问 AI、重写进入统一课程/助手链路。
- block 操作不破坏整节点内容。

### 阶段 3：画像、状态、痕迹合并

- 导师卡负责行动。
- 画像负责解释。
- 统计负责轨迹。
- AI 助手负责对话。
- 多入口展示同一套快照洞察，不产生互相冲突建议。

### 阶段 4：AI 导师主动引导

- `TeachingDecision` 动作类型更完整。
- 前端根据动作类型展示低打扰行动。
- 行动执行结果回写事件。

### 阶段 5：AI 课程生成升级

- 通用课程结构生成、个性化节点解释、薄弱点补充内容边界清楚。
- 薄弱点和风险能影响局部内容讲法。
- 课程生成质量有回归测试。

### 阶段 6：旧链路深度收敛

- `ai_learning_service`、`tutor_service`、`learning_record` 的重复判断逐步迁移。
- 旧接口仍可用，但内部不再各自生成主建议。
- 文档与真实链路一致。

### 阶段 7：质量评估

- 有 AI 输出质量、课程结构、状态决策和前端关键流程的回归测试。
- 能定位问题来自事件、状态、决策、上下文还是 AI 执行。

### 阶段 8：前端体验精修

- 入口减少，能力增强。
- 用户能明显感到课程可控、导师主动、画像可信、AI 懂上下文。
- 中英文文案同步，移动端不重叠。

### 阶段 9：长期智能闭环

- 支持长期目标、跨课程薄弱概念、动态复习计划、学习风格和知识迁移的结构预留或轻量实现。
- 系统理解的是“这个人如何成长”，不只是“这节课讲什么”。
