# 设计：反馈语义编译与角色化阅读呈现

## Context

当前唯一正文真源仍是 `CourseDocument / CourseBlock`。生成器以 `##` 教学模块标题切成课程块，前端通过 `CourseBlockStream` 渲染。问题发生在块内部：`feedback` 角色只有标签，没有任务级结构和专用组件；旧课程还包含大量用行内代码表达的数学式。

## Decisions

### 1. 静态参考内容与个性化反馈分层

`feedback` 课程块的正式含义收束为“检查与参考”：它可以保存核对标准、参考结论、推导依据和常见错误，但不声称已经评价某个学生。学生提交后的判断、解释和下一步建议继续由正式练习、学习痕迹与 AI 老师产生。

### 2. Markdown 仍是真源正文，任务级结构成为版本化派生契约

新课程生成继续输出可编辑 Markdown，不引入第二份正文。编译 `feedback` 块时，同时生成：

```text
feedback_structure
  schema_version: course_feedback_v1
  mode: static_reference
  sections[]
    section_id / title / kind / markdown / summary / collapsed_by_default
```

三级标题是新内容的稳定边界。兼容层也识别旧课程行首的“任务 N 答案方向”“评价标准”“常见错误反馈”等加粗标题，但兼容解析只用于呈现，不静默改写正式 Markdown。

### 3. 角色决定默认块类型与前端组件

新编译的 `feedback` 块使用 `review_checkpoint`，`checkpoint` 使用 `mastery_check`；其他普通正文继续使用 `rich_text`。前端根据教学角色选择 `FeedbackReviewBlock`，而不是把所有块都交给同一个无差别 Markdown 容器。

### 4. 旧数学代码只在反馈呈现层安全迁移

新生成内容必须使用 `$...$` 或 `$$...$$`。旧反馈中的反引号片段只有在命中明确数学信号时才在只读呈现中转成行内公式；代码标识、命令和程序片段保持代码。正式源 Markdown及修订 ID不因此改变。

### 5. 质量门验证可观察结构，不以存在符号代替可读性

当反馈块超过长度阈值或包含多个任务时，必须具有任务级三级标题；大量数学型行内代码视为结构缺陷。质量契约升版，旧缓存报告自动失效。短小的单一核对说明仍允许直接正文，避免机械要求空标题。

## Risks and Mitigations

- **旧课程标题不稳定**：兼容解析只匹配任务、评价、参考答案和常见错误等强信号，无法识别时安全回退为单节。
- **错误把程序代码转成数学公式**：只在反馈专用组件中转换具有幂、上下标、复杂度符号或数学比较符号的片段。
- **折叠后信息被隐藏**：短反馈直接展开；多任务反馈显示明确的逐项核对说明和每节摘要。
- **派生结构与正文漂移**：新生成结构与课程块一起编译；旧课程由当前 Markdown即时解析，不持久化第二真源。
