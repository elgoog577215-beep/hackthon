# 设计：难度驱动的课程块配方

## 1. 编译顺序

```text
学科必要与条件模块
→ 课程编排偏好扩展与排序
→ 目标难度选择块配方
→ 节级难度按块角色投影
→ 教学方案确认
→ 正文 prompt 与质量门
→ CourseBlock 落盘
```

难度配方必须发生在正文生成之前。新增或提升为必需的模块会进入同一个 `module_plan`，因此正文 prompt 必须生成对应二级标题，质量门也会检查该模块，不能只停留在展示层。

## 2. 三档难度配方

### 2.1 入门

- 每个正文节点增加“分步示范”。
- 从引导练习阶段起增加“带支架练习”。
- 块级难度契约继续提高示范块支架、降低初始自主性。

### 2.2 进阶

- 使用学科教学结构的标准模块配方。
- 不机械增加难度专属模块。
- 仍按节点角色投影挑战、支架和自主性。

### 2.3 高阶

- 前段节点增加深入推演，避免高阶只表现为术语更多。
- 后半程按主教学模式启用学科高阶模块：
  - 数学：证明与推导、数学建模。
  - 编程：测试与质量、架构设计。
  - 自然科学：实验设计、数据分析。
  - 生命科学：实验依据、正常与异常。
  - 人文社科：材料辨析、边界与反例。
  - 语言学习：文化与语用。
  - 商业职业：数据分析、角色模拟。
  - 通用课程：案例比较。
- 迁移或课程末端节点增加“迁移挑战”。

这些规则按课程相对位置和节点角色逐步启用，不要求每节都堆满全部高阶模块。

## 3. 去重与追溯

每个模块实例保留：

```json
{
  "composition_source": "subject_required | subject_optional | composition_style | difficulty_level",
  "selection_reasons": [
    "subject_optional",
    "difficulty_level"
  ]
}
```

编译器以 `module_id` 去重：

- 学科必要模块再次被偏好或难度选中时保留学科必要来源。
- 学科条件模块被难度选中时提升为 `required=true`，有效来源变为 `difficulty_level`。
- 偏好和难度同时选择同一模块时只保留一个实例，`selection_reasons` 同时记录两者。

全课分布分别统计偏好选择和难度选择的块角色，教学确认据此显示两个来源。

## 4. 验证边界

本变更验证的是确定性编译逻辑，而不是某次模型采样结果：

- 对相同计划分别编译入门、进阶、高阶并比较模块 ID、角色计数和来源。
- 验证高阶学科映射、课程位置渐进、模块去重和条件模块提升。
- 验证已确认模块进入内容块元数据和最终 `CourseBlock.payload`。
- 正文模型质量仍由既有生成与质量门负责，不用昂贵 A/B 证明规则是否存在。
