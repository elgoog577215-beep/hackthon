# 设计：让任务拥有完整且可补偿的生成生命周期

## 1. 生命周期所有权

一次初始课程生成请求只对应一个 `GenerationJob`，并由它拥有本次请求创建的临时对象：

```text
生成请求
  -> GenerationJob
     -> GenerationWorkspace
     -> Blueprint draft / Candidate version
     -> 未发布课程外壳
     -> 后台生成协程
  -> 正式发布后的 CourseDocument
```

`GenerationJob` 可以清理自己拥有的临时对象，但不能删除已经正式发布、可独立学习的 `CourseDocument`。课程本身被用户删除时，方向相反：课程删除先要求 `TaskManager` 终止并清理所有关联任务，再删除正式领域对象。

## 2. 创建过程采用补偿事务

当前文件存储不提供数据库事务，因此创建过程使用显式补偿栈：

1. 校验并规范化请求；
2. 创建工作区；
3. 创建初始课程外壳；
4. 持久化唯一任务并入队；
5. 任一步失败时，按相反顺序删除本次已经创建的对象。

补偿只作用于当前请求新建且带有同一 `job_id` 的对象，不能按 `course_id` 粗暴删除用户已有课程。

同一次前端提交还必须携带稳定的 `request_id`。后端在创建锁内先查找已持久化任务；相同请求号再次到达时返回原 `job_id` 和 `course_id`，不重复创建工作区、课程外壳或任务。弹窗内因网络不确定而重试时沿用请求号；用户关闭后重新发起的新建操作使用新请求号。

## 3. 取消与删除的状态顺序

取消任务必须先阻止新写入，再清理持久化对象：

1. 在任务锁内将任务标记为 `cancelled` 并持久化；
2. 取消节点协程和任务协程，并等待它们结束；
3. 删除工作区、候选版本和蓝图草稿；
4. 如果课程仍是由该任务创建的未发布外壳，则删除外壳；
5. 最后删除任务记录和内存映射。

课程删除必须复用这一套终止逻辑，而不是由路由分别猜测任务内部状态。若课程有关联的多个历史终态任务，全部清理；若存在活动任务，先取消并等待，再删除课程。

## 4. 操作必须符合状态机

- `pause` 只允许作用于 `pending` 或 `running`。
- `resume` 只允许作用于可恢复的 `paused`、`failed` 或受支持的质量阻塞状态。
- 节点跳过、重试、停止、附加指令只允许作用于真正的活动任务。
- 删除任务允许作用于活动或终态任务，但对象不存在时返回 404。
- 状态存在但操作不允许时返回 409，并携带当前状态和可采取的下一步。

路由成功响应必须表示操作已经发生，不能只表示请求被接收。

## 5. 前端只做状态投影

前端继续以 `job_id` 作为唯一任务身份：

- 取消任务成功后，移除本地任务和进度，并重新读取课程列表；
- 删除课程统一走 Course Store，由 Store 先调用后端课程删除，再清理本地课程和生成投影；
- 轮询发现本地未知的后端任务时，对账一次课程列表，补上其他标签页创建的课程外壳；
- 任务状态不允许操作时，保留后端错误语义，不预先伪造“已暂停”等状态；
- 剩余时间至少积累多个有效进度样本后才展示估算值。

## 6. 历史残留与数据边界

新逻辑阻止继续产生孤立对象。课程列表只展示正式发布课程，或仍由真实任务承接的未发布外壳；任务已经丢失的历史外壳继续保留在存储中供审计，但不能伪装成可学习课程。已有孤立工作区、0 节点外壳和损坏 JSON 不在未确认归属时自动批量删除。后续可提供显式维护命令，按“任务不存在、未发布、无正式版本、超过安全时间窗”四个条件执行清理。

## 7. 模型调用的等待与重试边界

模型 SDK 只负责一次网络请求，不拥有业务重试。连接、读取、写入和连接池等待时间通过环境变量给出保守默认值，SDK 的隐藏重试关闭。`AIBase` 只在错误明确表示“当前模型不可用”时切换候选模型；普通网络超时交给调用方的有限重试，不能遍历全部模型重复等待。

流式调用出现认证失败时继续作为不可重试的 provider 故障；超时、连接失败和空响应作为可重试异常抛出。`CourseService` 不再接收形如 `[Error: ...]` 的伪正文，因此节点级 `MAX_RETRIES`、草稿保存和最终失败状态可以真实接管。

## 8. 质量闸门分级

质量报告同时给出两个正交结论：

- `final_status`：严格质量是否全部通过，取值为 `passed` 或 `completed_with_warnings`；
- `publication_allowed`：课程是否具备安全、完整、可学习的最低发布条件。

以下问题阻断发布：正文为空或关键结构损坏、存在 critical 节点问题、无效证据引用、用户声明的必用资料未覆盖、学习资产存在 blocking issue、任何学习节点生成失败。少量难度契约、迁移任务、表达方式或知识归一建议只形成 warning，不阻断可用课程。

带警告发布后，任务状态保持 `completed_with_warnings`，但阶段为 `completed`，课程文档拥有正式发布回执并可以进入学习页。真正被阻断的任务才使用 `quality_failed`，保留工作区等待定点修复。服务启动时可以重新评估旧的 `quality_failed` 工作区；若新分级确认可发布，只执行质量对账和发布，不重新调用模型生成正文。

## 9. 验证矩阵

| 场景 | 任务 | 工作区 | 课程 | 前端 |
|---|---|---|---|---|
| 正常完成 | 终态可追溯 | 已发布 | 正式文档可读 | 课程卡片与学习页一致 |
| 创建中途失败 | 不残留 | 不残留 | 不残留空外壳 | 显示真实错误 |
| 同一请求重复到达 | 返回同一 job | 只有一份 | 只有一门 | 保持同一生成现场 |
| 暂停再恢复 | 同一 job | 保留检查点 | 保留外壳 | 同一进度继续 |
| 取消未发布课程 | 删除 | 删除 | 删除空外壳 | 任务与卡片同时消失 |
| 清理已发布任务 | 删除 | 删除 | 保留正式文档 | 课程仍可学习 |
| 删除生成中课程 | 删除并等待协程 | 删除 | 删除 | 不再重新出现 |
| 无效状态操作 | 不改变 | 不改变 | 不改变 | 显示 404/409 |
| provider 超时 | 有限节点重试或失败 | 保留检查点 | 不发布错误正文 | 显示真实失败与恢复入口 |
