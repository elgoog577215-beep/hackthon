# 变更：组装前端学习主链路

## 为什么

第 8.4 步已经证明后端标准课程主学习链可以通过公开 API 闭环，但前端仍有三处断裂：掌握检查动作只打开通用练习页而不定位具体任务；连续性条、画像和统计各自翻译同一个动作；浏览器验收默认落入共享 `default_user`。这些问题会让正确的后端运行态在页面上表现为错误任务、文案漂移或数据串扰。

## 改什么

- 让 `NextLearningAction.task_ref` 成为前端任务跳转的唯一依据，掌握检查必须指向具体 `mastery_check` 题目修订。
- 统一连续性动作的标签和原因展示，所有消费者读取同一套中英文翻译。
- 保持 `LearningRuntime` 为正式状态聚合真源，写操作后确定性刷新，不增加前端业务状态副本。
- 为本地和自动化验收提供显式、可选的学习者身份头，避免污染共享默认用户。
- 在标准课程 `cv2` 上完成桌面、移动端、刷新恢复和中英文主链验收。

## 不做什么

- 不新增课程、练习、诊断或掌握状态真源。
- 不在前端推导第二个“推荐下一步”。
- 不把验收身份覆盖当作正式登录系统。
- 不在本变更处理课程版本迁移和旧课程降级，这些留在 8.5B/8.5C。

## 影响

- 后端：学习动作任务引用投影。
- 前端：课程主动作路由、练习恢复、动作展示、HTTP 验收身份。
- 验收：标准课程浏览器主链与运行态一致性。
