# 设计：前端学习主链组装

## 1. 真源边界

`LearningRuntime.continuation.primary_action` 决定下一步，`primary_action.task_ref` 决定跳到哪个正式对象。前端只保存界面模式、当前选中题目和未同步草稿；阅读、记录、Attempt、诊断、补救和掌握状态仍由后端领域对象决定。

## 2. 精确任务路由

后端在生成 `start_mastery_check` 时，从当前课程正式题目中寻找与目标和节点一致的 `mastery_check` 任务，并返回 `kind=practice`、空 `object_id`、具体 `task_revision_id` 的任务引用。空 `object_id` 表示尚未创建 Attempt，前端只定位题目，不伪造或自动创建 Attempt；已有 Attempt 的恢复仍要求对象 ID 与任务修订同时匹配。

前端课程页按 `task_ref.kind` 路由任务型动作，不再用 `scope + action_type` 维护第二套任务分类。

## 3. 运行态刷新

正式写操作仍由对应 Store 执行。写成功后调用已有运行态刷新路径；组件不监听内部数组猜测变化。本地草稿只用于断网恢复，修订冲突时必须阻止提交并保留冲突提示。

## 4. 展示统一

新增纯展示工具，将动作类型和原因码映射到已有 `courseWorkspace.continuity` 中英文词条。连续性条、学习者画像和学习统计共同使用该工具，不改变动作本身。

## 5. 验收身份

HTTP 客户端仅在 `VITE_LEARNER_USER_ID` 显式配置时发送 `X-User-Id`。未配置时保持现有行为，正式身份系统不在本变更范围。浏览器验收服务器必须配置一次性隔离用户，并通过后端运行态确认写入归属。

## 6. 浏览器验收矩阵

1. 首次进入显示唯一主动作并进入准确阅读目标。
2. 完成阅读后，主动作切换为掌握检查并定位准确题目。
3. 刷新后保留服务器任务位置，不依赖旧组件内存。
4. 正式练习失败后进入诊断、补救和独立复验，同一主动作持续更新。
5. 画像、统计和连续性条展示同一动作含义。
6. 390px 移动端无主动作遮挡和页面横向溢出。
7. 中文和英文模式不出现动作原始代码或中文残留。
