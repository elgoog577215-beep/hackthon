# 设计：一份课程语义向外编译多种教学表达

## 1. 真源边界

`CourseDocument / CourseBlock` 继续负责课程结构、顺序和正文语义；`CourseKnowledgeBase / CourseKnowledgeMap` 负责当前课程知识语义和映射；`PracticeTask / MasteryCriterion` 负责正式题目与评分。大纲是课程结构投影，教案、PPT、讲义、图解和动画是 `TeachingRepresentation`，不得复制课程语义所有权。

## 2. 共享修订向量

课程仓库为每次正式发布和命令提交持久化 `CourseRevisionEvent`：

```text
previous_revision_vector
current_revision_vector
changed_source_keys
added_source_keys
removed_source_keys
affected_block_ids
```

修订键第一阶段包括：

```text
course_document
section:<section_id>
block:<block_id>
objective:<objective_id>
```

事件写入现有课程操作日志，派生服务即使暂时不可用也能稍后重放，不阻断课程正式提交。

## 3. 来源绑定

每个表达保存 `SourceBinding`，绑定课程、章节、块、选区、知识、目标、题目、资料及其源修订。课程级表达依赖 `course_document`；小节表达依赖对应 `section:*`；局部表达优先依赖精确 `block:*`，避免一次局部修改使全课程产物全部过期。

## 4. 派生依赖与陈旧传播

`AssetDerivationGraph` 使用有向边连接源对象、表达规格、教学表达和最终产物。课程修订事件到达后：

1. 比较新旧修订向量。
2. 更新源节点修订。
3. 从变化源节点沿依赖边查找下游对象。
4. 将受影响表达标记为 `stale` 并记录原因。
5. 后续构建器只重建受影响单元。

陈旧标记是确定性动作，不依赖大模型。表达构建失败不得回滚已经提交的课程语义。

## 5. 表现与语义编辑

表示编辑分为：

```text
presentation
equivalent_semantic
semantic
ambiguous
```

前两类只改变表示修订；`semantic` 转换为 `CourseAuthoringChange`；`ambiguous` 要求用户决定。只有教师或课程维护者确认后，基础课程领域命令才能覆盖正式语义。个人学习证据和 `PersonalAdaptationPlan` 不进入这条写入链。

教师在 PPT 目标页完成输入后，工作台对明确的教学维度变化自动运行影响预览，并用独立全屏影响图展示“修改前教学证据 → 系统语义解释 → 实际依赖单元 → 明确保留单元”。影响图中的教案重点、讲义引导、概念/例题页和理解检查必须来自注册表中的真实单元绑定，不能用静态 Demo 卡片代替。教师确认前课程真源保持不变；同步回执逐单元比较前后规格，区分 `content_changed` 与 `source_verified`，因此“无需修改”同样有可核验依据。

## 6. 任务与恢复

表达构建复用现有可恢复任务能力和同一服务端活动状态原则。WebSocket 只提供增量，服务端工作区与检查点是真源。课程提交先完成，表达重建异步进行；最后一个可用表达在新版本质量通过前继续保留。

课件构建对用户开放页级进度，但“看见正在生成的页面”不等于提前发布正式版本：编译器通过 SSE 发送临时 `slide_upsert`，前端使用同一 `SlideDeckSpec` 预览；只有完整规格通过质量门后，注册表才原子切换到新版本。中途失败时，临时页面丢弃，旧版本保持可用。

## 7. 结构化演示文稿

`SlideDeckSpec` 不是课程正文切片，而是对课程教学意图的页面级编译结果。每页必须声明：

```text
layout / slide_purpose
title / key_message / structured blocks / speaker_notes
section / block / revision source bindings
knowledge / ability / misconception / mastery references
practice task references
```

第一阶段注册封面、路线、章节、目标、概念、对比、过程、代码、易错、练习和小结版式。浏览器预览和 `.pptx` 导出读取同一份结构化规格；语义质量门检查课程来源、正式题目和知识绑定，视觉质量门检查版式容量、长段正文复制、Markdown 泄漏、原始 LaTeX、替换字符、常见错误解码片段和未知版式。进入可见页面前，公式表达统一转译为 PowerPoint 可直接显示的数学符号、Unicode 上下标、矩阵/方程组括号和运算箭头；任何关键问题都阻止新版本发布与导出。

编译缓存同时绑定共享表达编译器版本与课件编译器版本。公式转译、版式或质量规则升级后，即使课程修订没有变化，也必须重建旧课件，不能把旧生成器产物当成可复用单元继续导出。

页面标题、版式和讲者备注等表现修改保存在 `presentation_overrides`。基础课程变化后，编译器只在页面仍存在且基础字段兼容时重放表现修改；发生语义冲突时记录 `override_conflicts`，不得用旧表现修改覆盖新课程含义。

课件构建借鉴启智 PPT 模块的“大纲规划 → 分页填充 → HTML / PPTX 共用结构 → 讲者备注”思路，但不复制其独立资源真源：灵知先从 `CourseDocument`、知识绑定和正式题目规划整套页序，再逐教学单元填充页面，并经过质量门原子发布。每个可学习单元至少形成“学习目标 → 核心讲解 → 理解检查”三类页面；教案已有正式题目时引用原题，没有现成题目时只能基于本节目标和原课程块生成解释任务与自检标准，不得编造课程事实。

学习目标不仅绑定目标页，还作为教案重点、讲义引导、核心讲解、例题和理解检查的显式依赖。目标从计算技能转向概念理解时，派生规格必须实际改变教学重点、阅读追问、页面目标对齐提示与解释性检查；无共同来源的章节、资料和历史事实保持原样。课堂演示继续消费同一 `SlideDeckSpec`，并提供前后翻页、讲者备注、进度、临时黑屏和退出快捷键。

PPT 在界面上独立、在数据上同源。课程学习页顶栏右上角提供一级入口，进入 `/course/:courseId/ppt` 后使用独立全屏工作台，不再占用通用教学资源覆盖层。工作台由左侧完整页序、中部 16:9 页面画布、右侧质量/来源/编辑面板和顶部操作区组成；顶部最右侧始终是 `.pptx` 导出，旁边提供同步课程、交给 AI 老师和课堂全屏演示。课堂演示继续读取当前 `SlideDeckSpec`，支持键盘翻页、讲者备注和退出，不维护另一份页面内容。

中文与公式可用性不是字体名称存在即可。真实导出验收必须同时检查：PPTX 可解析、页面数量一致、中文文本存在、East Asian 字体声明存在、讲者备注存在、可见文本不含原始 LaTeX 或替换字符，并在原生 PowerPoint 中实际打开确认中文、上下标、数学符号、矩阵与运算箭头无乱码。缺字库的第三方转换器只作为兼容性信号，不得替代原生课堂软件验收。

## 8. 实施顺序

1. P0：修订事件、来源绑定、表示注册表、派生依赖图和陈旧传播。
2. R1：大纲、教案、讲义、题目投影和 `SlideDeckSpec` 最小编译闭环。
3. R2：增量重建、状态恢复、降级和跨产物一致性。
4. R3：结构化 PPT 反向语义编辑。
5. M：图解与结构化动画。
6. I：独立完成同源闭环，再验证与个人覆盖层的受控边界协作。
