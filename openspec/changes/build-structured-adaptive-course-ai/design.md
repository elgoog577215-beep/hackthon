# 设计：知识坐标驱动、证据可解释的课程生长内核

## Context

当前主链已经完成以下收束：

- `backend/course_document.py` 与 `backend/course_commands.py` 承担正式 `CourseDocument / CourseBlock` 真源与命令。
- `backend/subject_knowledge.py` 仍保留旧跨课程知识设施，`backend/course_knowledge_map.py` 保存课程覆盖；两者当前边界需要按“一门课程一套知识库”重构。
- `backend/learning_events.py`、正式领域对象与 `backend/learner_model_service.py` 提供可追溯学习事实和确定性模型。
- `backend/learning_runtime.py` 装配当前学习现场，`backend/ai_teacher_context.py` 与 `backend/ai_teacher_actions.py` 承担统一 AI 上下文和动作协议。
- 前端以 `CourseBlockStream.vue`、`InlineCourseBlockAI.vue`、`SideAIPanel.vue` 和 `KnowledgeLibrary.vue` 组成唯一学习现场。
- 现有个人适配块已经证明“先持久化方案、再确认、冲突时拒绝覆盖”的基本模式可行；下一步需要让确认后的变化进入同一课程真源，而不是长期停留在平行覆盖层。

新的产品目标跨越课程、知识、证据、AI、前端和恢复机制，不能以扩大一个 prompt 或继续增加临时 `adaptive_blocks` 完成。

主要约束：

1. 当前课程结构只有一个正式 `CourseDocument` 真源；经确认的学习证据驱动变化必须写入该真源并形成新修订，不得建立第二份长期课程正文。
2. `CourseKnowledgeBase` 必须独立完成当前课程的生成、发布和运行；正式链路不得读取跨课程知识设施，也不为历史课程保留迁移分支。
3. 每条用户输入可以被记录为原始证据，但不能直接成为稳定画像或掌握事实。
4. AI 可在方案层规划高权限课程变化，但用户确认、课程领域命令、质量门和修订门不可绕过。
5. 现有课程、笔记、作答、锚点和历史证据必须在课程变化中继续可追溯。
6. 当前存储仍以文件仓库为主，设计必须先通过仓库接口和事务日志成立，不能假定已经迁入数据库。
7. 本次只修改灵知当前 `course_id` 的课程，不跨课程传播学习结果，不修改启智、共享模板、PPT 或视频生产。

## Goals / Non-Goals

**Goals:**

- 建立当前课程/知识引用、证据、适应判断、课程方案、领域命令与修订之间的正式对象链。
- 支持单条强证据即时局部行动与多源证据广域行动，同时避免机械阈值。
- 支持当前课程结构操作、未来章节预调和具体难度变化，并基于知识关系决定范围。
- 建立可跨会话恢复、可部分确认、可冲突检测、可撤销的课程方案层。
- 复用现有课程/知识引用、学习事实、AI 老师、任务和审计体系，删除被接管的旧个人覆盖状态机。
- 为未来启智公共课程模板与灵知当前课程连接保留显式导入契约，但导入后必须复制、重标识并成为当前课程自己的知识，不共享跨课程运行时身份。

**Non-Goals:**

- 不建立、读取或更新跨课程产品知识目录，也不要求不同课程共用知识 ID。
- 不把 `AdaptationHypothesis` 合并进 `LearnerModel`。
- 不建立第二个课程正文仓库或学生拖拽编排工作室。
- 不使用浏览停留、滚动或一次错误独立触发广域修改。
- 不实现跨课程个性化、班级聚合、教师审核、PPT 反向编辑或视频生成。

## Decisions

### 1. 一门课程只有一套产品知识库，跨课程设施退出新链

采用：

```text
CourseDocument
  章节、小节与教学顺序真源
            ↓ path refs
CourseKnowledgeBase
  当前课程唯一知识库：概念组、原子知识点、能力、易错、掌握标准、六类关系
            ↓ precise bindings
CourseKnowledgeMap / KnowledgeBinding
  正文、课件、目标、题目和记录的知识引用与教学作用
```

原因：

- 用户产品中的知识库只针对当前课程；它的颗粒度、前置边界和掌握标准必须服从这门课程，不能由外部标准库决定是否存在。
- 章节路径和知识语义承担不同职责：路径说明怎样学，知识库说明学什么、会什么、容易错什么以及知识为什么相连。
- 同名术语在不同课程中可以具有不同颗粒度、边界和掌握标准，因此必须拥有各自课程内身份。
- 新课程生成、知识维护、AI 解释和掌握投影只读取当前 `course_id`；不存在外部参考缺失或冲突导致的隐式分支。

备选方案及放弃理由：

- **直接修改 `SubjectKnowledgeLibrary`**：个人证据会污染公共标准，权限和发布风险不可接受。
- **把课程目录当知识树**：课程顺序与学科关系混在一起，已被现有知识库决策淘汰。
- **让课程知识点必须映射跨课程正式 ID**：会把参考目录变成发布门禁，并迫使不同目标、不同颗粒度的课程共享错误身份。
- **把跨课程参考保留为“只读可选能力”**：仍会产生隐性依赖、来源混乱和错误复用；因此正式链路完全不读它。

### 2. 当前课程直接生长，不建立长期覆盖层

正式状态始终只有一个版本化 `CourseDocument`。经确认的课程生长块直接写入该文档，
正文、练习、版本和教学表达都从当前课程文档读取，不再建立按学习者区分的正式课程
投影或可见版本。

页面渲染时合成：

```text
CourseDocument
+ PendingAdaptationOverlay
+ InlineLearningRecordProjection
+ EphemeralAIProjection
= RenderedCourse
```

`PendingAdaptationOverlay` 只保存未确认方案差异；接受后通过课程领域命令生成新的
`CourseDocument` 修订，方案退出待确认投影。笔记和锚定 AI 解答自动保存为
`LearningRecord` 并投影进正文，
可编辑、删除和重新生成。教学内容变化不自动改写 `CourseKnowledgeBase`；涉及知识
语义时另走当前课程知识维护候选。

原因：

- 用户需要长期看到未确认变化，同时确认后的教学调整必须真的成为当前课程。
- 长期覆盖层会形成“看起来变了，课程真源没变”的双状态；直接课程修订更符合真实教学。
- 待确认投影可以精确表现差异，课程命令与修订日志负责正式应用和撤销。

备选方案：

- **候选只放 AI 聊天中**：无法原位阅读、跨章节定位和长期保留。
- **每个候选复制整门课程**：存储和冲突成本高，局部原因与差异丢失。
- **长期写入 PersonalCourseOverlay**：正式课程与实际学习流分裂，派生课件和后续生成无法消费变化。
- **跨课程传播学习结果**：当前课程证据越权改变其他课程，违反单课程边界。

### 3. 原始证据、适应假设和正式模型三层分离

采用：

```text
EvidenceItem：发生过什么，引用正式来源
AdaptationHypothesis：这些证据可能意味着什么
LearnerModel：正式规则可重算的长期学习投影
```

`EvidenceItem` 只做轻量索引，完整用户文字仍由 `AIConversation` 或 `LearningRecord` 保存。`AdaptationHypothesis` 可由 AI 辅助产生，但必须保留证据、反证、范围、置信度和有效期。它不写入 `LearnerModel`；只有正式领域事实按现有规则影响模型。

原因：

- 用户明确要求每一行输入都成为证据，但“一句话被记录”和“一句话成为稳定结论”必须分开。
- 假设需要可反证、可缩小范围和可过期，正式模型需要确定性和审计稳定性。
- 该分层允许 AI 灵活判断而不破坏已经建立的正式证据原则。

备选方案：

- **所有聊天直接进入 LearnerModel**：模型会漂移、不可重算，并把临时表达变成人格标签。
- **只使用错题等静态事实**：丢失学生明确表达的思维证据和讲法需求。
- **复制全部原始文本到新账本**：形成隐蔽第二真源并增加删除、导出和隐私成本。

### 4. 动态门槛采用混合决策器，不由固定次数或单次模型调用决定

决策器分两层：

1. 确定性特征与门禁：身份、课程范围、来源可信度、直接请求、目标位置、独立性、时间、新旧修订、正式诊断、反证、拒绝冷却。
2. AI 语义判断：证据表达的教学问题、范围是否相同、应调整的难度维度和候选行动。

决策结果包括：

```text
observe
respond_ephemerally
create_local_change_set
create_dependency_change_set
recommend_broad_scope
invalidate_or_shrink_existing_hypothesis
```

单条明确且范围唯一的自我报告可以通过直接性门立即产生低风险局部候选。普通提问
与笔记即使相互印证，也只形成观察中的假设；扩大到当前与后续位置必须至少包含正式
失败或重复独立正式信号。广域候选需要显式广域指令，或跨位置、多源、结果一致的
证据。具体分数可迭代校准，但每次决定必须能解释使用了哪些维度。

原因：

- 固定“三次”无法覆盖一句话就足够明确和十次模糊信号仍不够的差异。
- 完全交给模型会导致相同证据不稳定、拒绝冷却失效和范围漂移。
- 混合模式可测试硬边界，又保留语义理解能力。

### 5. 使用课程生长操作代数表达学习现场变化

核心操作分三组：

```text
Course evolution operations
  PATCH_COURSE_SPAN / INSERT_COURSE_BLOCK / REPLACE_COURSE_BLOCK
  RETIRE_COURSE_BLOCK / RESEQUENCE_COURSE_PATH
  ADD_CHECKPOINT / ADD_REMEDIATION_PATH / ADJUST_COURSE_DIFFICULTY
```

每个操作都带目标引用、预期课程修订、before 引用、after 载荷、差异、依赖和质量状态。`CourseEvolutionPlan` 是有向无环操作图，而不是一段模型建议。普通教学支架可写当前课程；需要修改知识语义时，另行生成当前课程知识维护候选。

原因：

- 课程操作是影响分析、部分接受、幂等、撤销和效果评价的共同语言。
- 对行级补写使用整块替换会制造无关 diff 并增加冲突。
- 对拆分、合并和重排必须显式保存身份映射，不能让模型自由覆盖 JSON。

备选方案：

- **继续只支持附加说明块**：无法表达行级补充、支架变化和未来路径预调。
- **保存 JSON Patch 作为唯一协议**：通用但没有教学语义，难以做领域质量门和锚点迁移。
- **每种 UI 单独建接口**：确认、冲突和历史会再次分裂。

### 6. 知识库提供坐标与影响，知识语义变化走当前课程维护链

课程生长从 `CourseKnowledgeBase / KnowledgeBinding` 读取知识点、能力点、易错点、掌握标准和依赖范围。普通讲解、支架、节奏和检查变化不修改知识库；若分析发现知识语义本身存在缺陷，生成当前课程知识维护建议：

```text
source_evolution_plan_id
evidence_summary
suggested_course_or_knowledge_scope
impact_reason
analysis_revision
```

用户接受建议后进入当前课程 `CourseAuthoringChange`，使用课程知识质量门和发布流程。课程或知识修订改变时，待确认方案重新定位、标记冲突或重算。

原因：

- 教学表达缺陷与知识语义缺陷可能相关，但质量门和影响范围不同。
- 区分两类操作可让学习证据真正改进课程，同时避免把一次困惑直接写成正式知识结论。

### 6.1 本节课程生长统一用户请求与证据建议

本节课程调整不再建立“用户手动重生成”和“证据适配”两套链。两种触发都创建或更新
同一 `CourseEvolutionPlan`：

```text
manual_section_request / learning_evidence
→ requested_roles + difficulty_delta
→ 读取本节真实 CourseBlock.role
→ 已有 role: REPLACE_COURSE_BLOCK
→ 缺失 role: INSERT_COURSE_BLOCK
→ 每个候选生成后保存到方案工作区
→ 本节 CourseKnowledgeBase 引用、范围、重复和候选质量检查
→ 用户整体确认
→ replacements + insertions 单次 CourseDocument 提交
→ 受影响 TeachingRepresentation 对账与重建
→ 新难度独立任务复验
```

重复正式通过具有双重但不冲突的含义：它仍是“存在知识缺口”的反证，同时可作为
“当前难度已掌握、可以提高挑战”的正证据。挑战方案必须保留旧难度掌握和历史作答，
将新状态记录为“等待更高挑战复验”，不得先清空掌握再让学生重新证明。
挑战效果直接由后续更高难度独立任务判断，不要求学习者先点击或评价某个候选块；
一道不同任务通过只形成初步支持，至少两道不同任务通过才形成持续确认。更高挑战
连续失败只说明挑战跨度或支架需要调整，不能反向生成“旧难度未掌握”的结论。

候选工作区与正式课程提交分离：每个块候选生成后立即保存，防止长链中断丢失；只有
全部候选和整组同源检查通过后才允许确认。确认使用一个稳定命令 ID 和一次课程文档
提交，使块内升级与缺失块新增同时生效；撤销使用同一操作组恢复被替换块的旧载荷并
退役本次新增块。

### 7. 课程生长应用使用预检加可恢复操作组

当前文件仓库不具备数据库事务，因此采用：

1. 读取并固定完整 `base_revision_vector`。
2. 对当前课程/知识引用、用户范围和课程操作执行预检与质量门。
3. 写入状态为 `prepared` 的命令组日志和各域预期结果摘要。
4. 通过单次课程文档提交原子写入所选操作，并记录新课程修订。
5. 所有步骤完成后写入 `committed` 回执。
6. 进程重启时对 `prepared/applying` 命令组对账：若可幂等完成则继续，否则执行补偿并恢复原修订引用。

命令组使用稳定 `idempotency_key`。应用端只能消费已接受操作 ID，不能在执行时让模型改变范围或内容。

原因：

- 简单顺序写文件会产生课程操作只应用一半的状态。
- 复制整份仓库快照回滚会覆盖并发合法变化。
- 可恢复命令组与现有生成任务、块候选恢复思想一致，可在当前存储架构上实施。

未来迁移到事务数据库时，仓库接口和领域语义保持不变，可将课程操作组提交替换为数据库事务。

### 8. 难度使用差异向量，不保存新的单一“当前等级”

课程生长方案中的 `difficulty_delta` 只描述相对当前课程契约的局部变化：

```text
knowledge_granularity
abstraction
reasoning_depth
derivation_completeness
example_count_and_distance
scaffold_intensity
task_complexity
integration_scope
pacing_granularity
feedback_frequency
mastery_independence
```

规划器必须把 delta 编译为课程操作。接受后只更新受影响位置的支架、例子、任务或节奏，不把整门课程粗暴切换为 `beginner/intermediate/advanced`。

原因：

- 用户的“这一章推导更细、后面题目更难”是局部多维变化，单等级无法表达。
- 沿用既有难度契约可以复用质量门并避免伪难度。

### 9. 候选存储归属于当前课程，界面共享同一 Store

后端新增仓库接口，建议初始模块：

```text
backend/course_knowledge_base.py
backend/learning_evidence_index.py
backend/adaptation_hypotheses.py
backend/personal_adaptation_plans.py
backend/personal_adaptation_planner.py
backend/personal_adaptation_impact.py
backend/personal_adaptation_quality.py
backend/personal_course_overlay.py
backend/routers/course_evolution.py
```

实际文件可在实现时根据现有模块边界合并，但必须保持领域职责。课程候选、回执和操作历史按当前 `course_id` 保存；当前本地用户的学习事实继续作为证据来源，但不参与课程可见性分支。

前端建议由唯一 `courseEvolution` Store 管理：

```text
pending adaptation plans
operation decisions
scope preview
generation/recovery state
receipts and history
```

`CourseBlockStream`、目录、`KnowledgeLibrary` 和 `SideAIPanel` 只消费投影，不各自保存接受状态。

原因：

- 同一个人方案会在多个入口出现，分散 Store 会形成重复确认和状态漂移。
- 按课程组织便于删除、导出、性能分页和撤销对账。

### 10. 候选交互以原位差异加统一摘要为主

桌面：

- 行级变化原位高亮。
- 新块保持正文宽度并带候选边缘标记。
- 左侧目录显示未来章节待处理数量。
- AI 老师显示本次变更理由、范围和对话解释。
- 多操作个人方案使用统一摘要抽屉或覆盖层，不新增课程工作室。

移动：

- 正文仍显示轻量标记。
- 详细差异、范围树和批量操作在全宽覆盖层中完成。
- 底部固定操作必须避开现有工具坞和安全区。

接受、拒绝、重新生成、补充要求和范围调整全部调用同一 Store 与服务端状态机。

原因：

- 用户必须在课程语境中看懂改了什么，同时又需要跨章节总览。
- 只用行内按钮无法处理跨章节影响；只用独立管理页又会割裂学习现场。

### 11. 候选内容交互与正式掌握严格分离

候选块和候选检查使用 `candidate_ref` 记录交互。它们可以进入修改效果分析，但不能直接形成正式掌握。只有候选被接受，且其中的正式任务通过资产质量门产生稳定题目修订后，未来 Attempt 才能成为正式证据。

原因：

- 用户需要能先阅读候选再决定，但候选可能被拒绝或重生。
- 将候选任务当正式掌握会让历史证据引用不存在的正式内容。

### 12. 适应效果采用前后同类证据比较

接受变更时冻结 `effect_baseline`：问题类型、诊断、用户选择范围、目标块、目标知识、
能力、易错、关键证据、原失败 Attempt 与题目修订。后续评价使用同类证据：

- 相同或相关知识的求助频率。
- 回答“已解决/还不清楚”。
- 正式题目与独立复验。
- 用户明确纠正或回退。
- 候选内容是否产生新困惑。

不使用内容长度、接受按钮点击、动画播放、关键帧切换或参与时长独立证明效果。
结构化动画中的概念判断必须记录明确答案和正确性，但该判断仍只证明课程支持被实际
使用；最终效果必须由方案绑定的全新正式题验证。一个不同正式任务通过只标记
`initial_support`，至少两个不同正式任务持续通过才标记 `confirmed`。效果不足可以
生成新假设或回退候选，但仍需要用户确认持久修改。

### 13. 路径树、知识关系网和叶子能力包必须分工

课程知识结构固定分为三个互补视角：

```text
路径树：CourseDocument 章节 → 小节 → ConceptGroup → KnowledgePoint
关系网：KnowledgePoint 之间的 prerequisite / derives / equivalent_to /
        contrasts_with / applies_to / generalizes
能力包：KnowledgePoint → SkillUnit / Misconception / MasteryCriterion
```

路径树中的 `contains`、上一节和下一节不是知识语义关系；正文、课件和题目到知识点的连接属于 `KnowledgeBinding`；能力、易错和掌握标准是知识点关联实体。正式关系网不得提供无明确教学作用的 `related` 兜底。

每个关系必须保存方向、理由、条件、来源和修订。`prerequisite` 表达移除前置后目标无法或明显难以独立学习；`derives` 表达在条件和步骤下推出；`applies_to` 表达用已有规则处理目标；`equivalent_to` 和 `contrasts_with` 为规范化对称关系；`generalizes` 从一般规律指向特殊情形。多前提关系使用 `relation_group_id + all_of / any_of`，避免把联合条件误读为单边充分条件。

原因：

- 只有路径树会让知识库退化成章节目录的复制品。
- 只有关系图会失去课程学习顺序、内容分类和可读入口。
- 只有知识名称没有能力、易错和掌握标准，无法驱动题目、诊断、AI 教学和个人状态。
- 三者分工后，同一课程知识坐标才能被正文、课件、练习、AI 和学习事实稳定复用。

## Data Flow

### 证据到候选

```text

→ 原对象持久化
→ EvidenceItem 幂等登记
→ 相关 Hypothesis 增量更新

```

### 方案到当前课程修订

```text

→ 重新计算操作依赖和影响
→ 最终预检

→ CourseDocument 领域命令
→ committed ActionReceipt

→ 候选退出活动投影
→ 建立 effect_baseline
```

## Quality Gates

每个课程生长方案至少经过：

1. **结构门**：目标存在、操作合法、依赖无环、位置合法、ID 与修订完整。
2. **知识门**：读取的知识、能力、易错、掌握标准和依赖关系存在且引用完整；知识语义写操作必须单独通过当前课程知识质量门。
3. **教学门**：修改与证据问题匹配，避免重复、断层、错误前置和无意义扩写。
4. **难度门**：delta 与真实内容一致，拒绝只增长度、术语或题量。
5. **资产门**：正式题目、掌握标准、媒体和目标引用未被静默覆盖；新增检查明确标记来源和效力。
6. **锚点门**：学习记录与历史事实能够保留、迁移或明确进入待确认。
7. **范围门**：当前课程和作用范围合法，只读写当前 `course_id`，未查询或修改其他课程及跨课程知识设施。
8. **差异门**：候选有实质变化，且没有覆盖范围外内容。

硬错误不得被综合分数掩盖。模型质量检查失败时最多按明确修订指令重试，不能无限自循环。

### 首次课程生成改为六步确认生产线

首次生成继续复用唯一 `GenerationJob`、隔离工作区、`CourseDocument`、
`CourseKnowledgeBase` 与发布质量门，但用户只面对一条明确的六步生产线：

1. **需求**：用户提交主题、目标、难度、规模和资料，提交动作冻结需求修订。
2. **目录**：生成并审阅课程定位、学习目标、章节、小节、每节责任、前置和范围。
3. **知识蓝图**：按小节生成概念组、原子知识、能力、易错、掌握标准和关系摘要。
4. **教学方案**：根据已确认知识蓝图规划每节讲法、例子、练习和教学表达。
5. **课程内容**：根据已确认教学方案生成正文和资产，并建立精确知识绑定。
6. **质量与发布**：检查结构、知识、内容、来源和版本链，用户确认后原子发布。

提交需求视为确认第 1 步；第 2 至第 6 步都必须进入 `waiting_for_review`，同一个
`job_id` 只有收到与当前待确认步骤匹配的确认命令后才能继续。产品不再提供快速
模式或直接生成入口，服务重启也不得自动跨过确认门。

内部仍然用三类可恢复中间产物控制返工：

- `CourseOutlineRevision` 保存目录与每节责任。
- `SectionKnowledgePackage / SectionRelationBatch` 保存逐节知识能力包与关系批次。
- 正文块、学习资产、质量报告分别保存自己的检查点和当前修订。

每步确认都保存 `artifact_revision` 与 `input_revisions`。后续产物只能消费已确认
上游修订；上游修订变化时，依赖旧修订的下游步骤必须进入 `needs_regeneration`。
最终同源报告逐项验证需求、目录、知识蓝图、教学方案、内容和知识绑定仍属于同一
确认链，任何漂移都阻止发布。

目录确认后不得由后续模型静默增加、删除或重排小节。所有长时间模型调用都必须
周期更新心跳和当前产物；任务事件同时给出 `item_index / item_total / item_id`，
前端以步骤、阶段和完成数量表达进度，不再用一个固定百分比代替真实状态。

这条生产线的最小恢复单元依次是目录、单个知识包、单个关系批次和单个正文块。
任何中间产物都只能投影为生成现场草稿，不能进入正式课程。

## Risks / Trade-offs

- **[风险] 每条输入都登记导致数据量快速增长** → 只保存源对象引用、内容指纹和必要索引；按课程分片，支持增量摘要和分页，不复制全文。
- **[风险] AI 过度解释局部证据并频繁改课** → 动态范围门槛、反证、拒绝冷却、同义候选去重和广域更强证据门。
- **[风险] 候选长期堆积使正文难读** → 同一目标与问题合并候选，目录显示数量，正文只显示当前位置差异，提供统一待处理摘要和批量处理。
- **[风险] 局部问题被过度外推** → 动态范围门、最小充分范围、明确差异和用户确认。
- **[风险] 文件仓库课程操作半应用** → 预检、单次课程文档提交、幂等回执、启动对账和补偿，不使用整库快照覆盖。
- **[风险] 拆分合并破坏笔记和历史作答** → 稳定 ID、墓碑、旧新映射、文本锚点重解析和歧义待确认。
- **[风险] 候选中的练习污染掌握** → 所有候选交互绑定 candidate ID，只进入效果分析；正式任务需接受并通过资产质量门。
- **[风险] 大课程全局影响分析过慢** → 依赖索引、按受影响子图计算、后台生成、流式状态和可取消任务；阅读主链不等待分析。
- **[风险] 跨课程设施重新成为隐藏真源** → 新生成与运行链禁止调用并移除活动入口；契约测试以调用监控证明正式流程零访问。
- **[风险] 知识库再次退化成章节标题复制** → 轻量目录确认后按小节生成知识包，原子性与模板内容作为局部门；未通过质量门的新课程不得发布。
- **[风险] 分阶段调用增加模型请求次数** → 用小节检查点、有限并发和局部重试换取可恢复性；总输出量受单节 schema 约束，不重复发送整份原始蓝图。
- **[风险] 六类关系被模型泛化滥用** → 关系类型白名单、方向规则、移除测试、条件与理由审查、环检测和候选确认共同约束。
- **[权衡] 用户确认降低完全自动化程度** → 候选可以自动生成并持续可见，用户只在持久改变时确认；这是高权限 AI 可被信任的必要条件。
- **[边界] 学习证据只作用于当前课程** → 这是当前产品规则，不预留跨课程课程生长分支。

## Implementation Plan

### 1. 建立新课程对象与质量评估

- 增加 `CourseKnowledgeBase` 只读消费、证据索引、假设、课程生长方案和课程操作组接口。
- 新课程生成时从 `CourseDocument`、正文块、目标、学习资产和资料建立知识库；不能确认原子知识时阻止发布，禁止用章节标题确定性伪造知识点。
- 新增读取接口和内部质量评估，验证新课程学习主链。

### 2. 收束正式写入口

- 所有持久课程变化统一进入课程维护链。
- `adaptive_blocks` 只承担即时临时支持，不能保存正式课程变化。
- 删除学习者可见课程投影和其他平行正文写入口，不迁移历史课程数据。

### 3. 接入证据与局部候选

- 先登记 AI 用户消息、明确回答反馈和现有正式对象引用。
- 只开启明确局部请求的动态候选，验证误触发、冷却和冲突。
- 再接入多源假设、后续章节预调和广域范围推荐。

### 4. 接入完整课程生长操作

- 按 `PATCH_COURSE_SPAN / INSERT_COURSE_BLOCK / REPLACE_COURSE_BLOCK / RETIRE_COURSE_BLOCK / RESEQUENCE_COURSE_PATH` 依次实现课程操作和锚点迁移。
- 每接管一种操作就删除或拒绝对应旁路写入口。
- 全部操作通过单一课程方案协议后，再启用部分接受和批量课程操作组。

### 5. 接入知识坐标与维护者建议

- 先让课程方案读取知识、能力、易错、掌握标准和依赖范围。
- 再实现课程与知识修订后的候选重定位和冲突提示。
- 最后实现“重复学习问题 → 当前课程改进建议”，知识语义变化进入独立 `CourseAuthoringChange` 维护流程。

### 6. 前端收束与平行状态退役

- 建立唯一 `courseEvolution` Store。
- 让正文、目录、知识库和 AI 老师消费同一候选状态。
- 旧 `adaptive_blocks` 只保留即时临时解释语义；任何持久课程变化改走 `CourseEvolutionPlan / CourseDocument`。
- 完成中英文、多视口、断网、刷新、重启和并发验收后删除旧平行状态。

### 回滚策略

- 新对象和接口以功能开关分阶段启用；关闭时现有正式课程读取与学习主链继续工作。
- 未确认方案可以停用投影而不影响当前课程；已确认变化通过补偿课程命令撤销。
- 已接受变化使用操作日志和补偿命令撤销，禁止恢复整门课程旧快照。

## Open Questions

产品方向与权限边界已经确定，没有需要阻塞实施的产品问题。实现阶段需要通过真实样本校准但不改变契约的参数包括：

- 不同证据类型的初始权重、衰减窗口和同义候选合并窗口。
- 大课程中全局影响分析的异步阈值与分页大小。
- 候选高亮的最终视觉 token 和统一摘要在各视口的具体尺寸。
- 文件仓库阶段的命令组日志保留周期与未来数据库迁移时机。

这些参数必须通过样本、性能和端到端测试确定，不得反向改变“动态而非固定次数、当前课程隔离、用户确认、一门课程一套知识库、外部参考非阻塞”的产品契约。
