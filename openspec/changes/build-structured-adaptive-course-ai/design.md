# 设计：知识坐标驱动、证据可解释的课程生长内核

## Context

当前主链已经完成以下收束：

- `backend/course_document.py` 与 `backend/course_commands.py` 承担正式 `CourseDocument / CourseBlock` 真源与命令。
- `backend/subject_knowledge.py` 仍保留旧跨课程知识设施，`backend/course_knowledge_map.py` 保存课程覆盖；两者当前边界需要按“一门课程一套知识库”重构。
- `backend/learning_events.py`、正式领域对象与 `backend/learner_model_service.py` 提供可追溯学习事实和确定性模型。
- `backend/learning_runtime.py` 装配当前学习现场，`backend/ai_teacher_context.py` 与 `backend/ai_teacher_actions.py` 承担统一 AI 上下文和动作协议。
- 前端以 `CourseBlockStream.vue`、`InlineCourseBlockAI.vue`、`SideAIPanel.vue` 和 `KnowledgeLibrary.vue` 组成唯一学习现场。
- 现有个人适配块已经证明“先持久化方案、再确认、冲突时拒绝覆盖”的基本模式可行；下一步需要让确认后的变化进入同一课程真源，而不是长期停留在平行覆盖层。

新的产品目标跨越课程、知识、证据、AI、前端和恢复机制，不能以扩大一个 prompt 或继续增加临时 `adaptive_blocks` 完成。

主要约束：

1. 当前课程结构只有一个正式 `CourseDocument` 真源；经确认的学习证据驱动变化必须写入该真源并形成新修订，不得建立第二份长期课程正文。
2. `CourseKnowledgeBase` 必须独立完成当前课程的生成、发布和运行；正式链路不得读取跨课程知识设施，也不为历史课程保留迁移分支。
3. 每条用户输入可以被记录为原始证据，但不能直接成为稳定画像或掌握事实。
4. AI 可在方案层规划高权限课程变化，但用户确认、课程领域命令、质量门和修订门不可绕过。
5. 现有课程、笔记、作答、锚点和历史证据必须在课程变化中继续可追溯。
6. 当前存储仍以文件仓库为主，设计必须先通过仓库接口和事务日志成立，不能假定已经迁入数据库。
7. 本次只修改灵知当前 `course_id` 的课程，不跨课程传播学习结果，不修改启智、共享模板、PPT 或视频生产。

## Goals / Non-Goals

**Goals:**

- 建立当前课程/知识引用、证据、适应判断、课程方案、领域命令与修订之间的正式对象链。
- 支持单条强证据即时局部行动与多源证据广域行动，同时避免机械阈值。
- 支持当前课程结构操作、未来章节预调和具体难度变化，并基于知识关系决定范围。
- 建立可跨会话恢复、可部分确认、可冲突检测、可撤销的课程方案层。
- 复用现有课程/知识引用、学习事实、AI 老师、任务和审计体系，删除被接管的旧个人覆盖状态机。
- 为未来启智公共课程模板与灵知当前课程连接保留显式导入契约，但导入后必须复制、重标识并成为当前课程自己的知识，不共享跨课程运行时身份。

**Non-Goals:**

- 不建立、读取或更新跨课程产品知识目录，也不要求不同课程共用知识 ID。
- 不把 `AdaptationHypothesis` 合并进 `LearnerModel`。
- 不建立第二个课程正文仓库或学生拖拽编排工作室。
- 不使用浏览停留、滚动或一次错误独立触发广域修改。
- 不实现跨课程个性化、班级聚合、教师审核、PPT 反向编辑或视频生成。

## Decisions

### 1. 一门课程只有一套产品知识库，跨课程设施退出新链

采用：

```text
CourseDocument
  章节、小节与教学顺序真源
            ↓ path refs
CourseKnowledgeBase
  当前课程唯一知识库：概念组、原子知识点、能力、易错、掌握标准、六类关系
            ↓ precise bindings
CourseKnowledgeMap / KnowledgeBinding
  正文、课件、目标、题目和记录的知识引用与教学作用
```

原因：

- 用户产品中的知识库只针对当前课程；它的颗粒度、前置边界和掌握标准必须服从这门课程，不能由外部标准库决定是否存在。
- 章节路径和知识语义承担不同职责：路径说明怎样学，知识库说明学什么、会什么、容易错什么以及知识为什么相连。
- 同名术语在不同课程中可以具有不同颗粒度、边界和掌握标准，因此必须拥有各自课程内身份。
- 新课程生成、知识维护、AI 解释和掌握投影只读取当前 `course_id`；不存在外部参考缺失或冲突导致的隐式分支。

备选方案及放弃理由：

- **直接修改 `SubjectKnowledgeLibrary`**：个人证据会污染公共标准，权限和发布风险不可接受。
- **把课程目录当知识树**：课程顺序与学科关系混在一起，已被现有知识库决策淘汰。
- **让课程知识点必须映射跨课程正式 ID**：会把参考目录变成发布门禁，并迫使不同目标、不同颗粒度的课程共享错误身份。
- **把跨课程参考保留为“只读可选能力”**：仍会产生隐性依赖、来源混乱和错误复用；因此正式链路完全不读它。

### 2. 当前课程直接生长，不建立长期覆盖层

正式状态始终只有一个版本化 `CourseDocument`。经确认的课程生长块直接写入该文档，
正文、练习、版本和教学表达都从当前课程文档读取，不再建立按学习者区分的正式课程
投影或可见版本。

页面渲染时合成：

```text
CourseDocument
+ PendingAdaptationOverlay
+ InlineLearningRecordProjection
+ EphemeralAIProjection
= RenderedCourse
```

`PendingAdaptationOverlay` 只保存未确认方案差异；接受后通过课程领域命令生成新的
`CourseDocument` 修订，方案退出待确认投影。笔记和锚定 AI 解答自动保存为
`LearningRecord` 并投影进正文，
可编辑、删除和重新生成。教学内容变化不自动改写 `CourseKnowledgeBase`；涉及知识
语义时另走当前课程知识维护候选。

原因：

- 用户需要长期看到未确认变化，同时确认后的教学调整必须真的成为当前课程。
- 长期覆盖层会形成“看起来变了，课程真源没变”的双状态；直接课程修订更符合真实教学。
- 待确认投影可以精确表现差异，课程命令与修订日志负责正式应用和撤销。

备选方案：

- **候选只放 AI 聊天中**：无法原位阅读、跨章节定位和长期保留。
- **每个候选复制整门课程**：存储和冲突成本高，局部原因与差异丢失。
- **长期写入 PersonalCourseOverlay**：正式课程与实际学习流分裂，派生课件和后续生成无法消费变化。
- **跨课程传播学习结果**：当前课程证据越权改变其他课程，违反单课程边界。

### 3. 原始证据、适应假设和正式模型三层分离

采用：

```text
EvidenceItem：发生过什么，引用正式来源
AdaptationHypothesis：这些证据可能意味着什么
LearnerModel：正式规则可重算的长期学习投影
```

`EvidenceItem` 只做轻量索引，完整用户文字仍由 `AIConversation` 或 `LearningRecord` 保存。`AdaptationHypothesis` 可由 AI 辅助产生，但必须保留证据、反证、范围、置信度和有效期。它不写入 `LearnerModel`；只有正式领域事实按现有规则影响模型。

原因：

- 用户明确要求每一行输入都成为证据，但“一句话被记录”和“一句话成为稳定结论”必须分开。
- 假设需要可反证、可缩小范围和可过期，正式模型需要确定性和审计稳定性。
- 该分层允许 AI 灵活判断而不破坏已经建立的正式证据原则。

备选方案：

- **所有聊天直接进入 LearnerModel**：模型会漂移、不可重算，并把临时表达变成人格标签。
- **只使用错题等静态事实**：丢失学生明确表达的思维证据和讲法需求。
- **复制全部原始文本到新账本**：形成隐蔽第二真源并增加删除、导出和隐私成本。

### 4. 动态门槛采用混合决策器，不由固定次数或单次模型调用决定

决策器分两层：

1. 确定性特征与门禁：身份、课程范围、来源可信度、直接请求、目标位置、独立性、时间、新旧修订、正式诊断、反证、拒绝冷却。
2. AI 语义判断：证据表达的教学问题、范围是否相同、应调整的难度维度和候选行动。

决策结果包括：

```text
observe
respond_ephemerally
create_local_change_set
create_dependency_change_set
recommend_broad_scope
invalidate_or_shrink_existing_hypothesis
```

单条明确且范围唯一的自我报告可以通过直接性门立即产生低风险局部候选。普通提问
与笔记即使相互印证，也只形成观察中的假设；扩大到当前与后续位置必须至少包含正式
失败或重复独立正式信号。广域候选需要显式广域指令，或跨位置、多源、结果一致的
证据。具体分数可迭代校准，但每次决定必须能解释使用了哪些维度。

原因：

- 固定“三次”无法覆盖一句话就足够明确和十次模糊信号仍不够的差异。
- 完全交给模型会导致相同证据不稳定、拒绝冷却失效和范围漂移。
- 混合模式可测试硬边界，又保留语义理解能力。

### 5. 使用课程生长操作代数表达学习现场变化

核心操作分三组：

```text
Course evolution operations
  PATCH_COURSE_SPAN / INSERT_COURSE_BLOCK / REPLACE_COURSE_BLOCK
  RETIRE_COURSE_BLOCK / RESEQUENCE_COURSE_PATH
  ADD_CHECKPOINT / ADD_REMEDIATION_PATH / ADJUST_COURSE_DIFFICULTY
```

每个操作都带目标引用、预期课程修订、before 引用、after 载荷、差异、依赖和质量状态。`CourseEvolutionPlan` 是有向无环操作图，而不是一段模型建议。普通教学支架可写当前课程；需要修改知识语义时，另行生成当前课程知识维护候选。

原因：

- 课程操作是影响分析、部分接受、幂等、撤销和效果评价的共同语言。
- 对行级补写使用整块替换会制造无关 diff 并增加冲突。
- 对拆分、合并和重排必须显式保存身份映射，不能让模型自由覆盖 JSON。

备选方案：

- **继续只支持附加说明块**：无法表达行级补充、支架变化和未来路径预调。
- **保存 JSON Patch 作为唯一协议**：通用但没有教学语义，难以做领域质量门和锚点迁移。
- **每种 UI 单独建接口**：确认、冲突和历史会再次分裂。

### 6. 知识库提供坐标与影响，知识语义变化走当前课程维护链

课程生长从 `CourseKnowledgeBase / KnowledgeBinding` 读取知识点、能力点、易错点、掌握标准和依赖范围。普通讲解、支架、节奏和检查变化不修改知识库；若分析发现知识语义本身存在缺陷，生成当前课程知识维护建议：

```text
source_evolution_plan_id
evidence_summary
suggested_course_or_knowledge_scope
impact_reason
analysis_revision
```

用户接受建议后进入当前课程 `CourseAuthoringChange`，使用课程知识质量门和发布流程。课程或知识修订改变时，待确认方案重新定位、标记冲突或重算。

原因：

- 教学表达缺陷与知识语义缺陷可能相关，但质量门和影响范围不同。
- 区分两类操作可让学习证据真正改进课程，同时避免把一次困惑直接写成正式知识结论。

### 6.1 本节课程生长统一用户请求与证据建议

学生侧不再建立“正文重生成”“用户手动调课”和“证据适配”三套链。正文“调整这段”
只是“调整课程”的就近快捷入口，AI 老师侧栏是主入口；用户主动请求与系统问题建议都
创建或更新同一 `CourseEvolutionPlan`：

```text
manual_request / learning_evidence
→ current_block / current_section / whole_course 用户硬范围
→ requested_roles + difficulty_delta
→ 读取范围内真实 CourseBlock.role
→ 已有 role: REPLACE_COURSE_BLOCK
→ 缺失 role: INSERT_COURSE_BLOCK
→ 每个候选生成后保存到方案工作区
→ 当前 CourseKnowledgeBase 引用、范围、重复和候选质量检查
→ 单项留在侧栏；两项及以上、跨小节或全课程进入居中工作台
→ 用户整体确认
→ replacements + insertions 单次 CourseDocument 提交
→ 受影响 TeachingRepresentation 对账与重建
→ 新难度独立任务复验
```

重复正式通过具有双重但不冲突的含义：它仍是“存在知识缺口”的反证，同时可作为
“当前难度已掌握、可以提高挑战”的正证据。挑战方案必须保留旧难度掌握和历史作答，
将新状态记录为“等待更高挑战复验”，不得先清空掌握再让学生重新证明。
挑战效果直接由后续更高难度独立任务判断，不要求学习者先点击或评价某个候选块；
一道不同任务通过只形成初步支持，至少两道不同任务通过才形成持续确认。更高挑战
连续失败只说明挑战跨度或支架需要调整，不能反向生成“旧难度未掌握”的结论。

候选工作区与正式课程提交分离：每个块候选生成后立即保存，防止长链中断丢失；只有
全部候选和整组同源检查通过后才允许确认。确认使用一个稳定命令 ID 和一次课程文档
提交，使块内升级与缺失块新增同时生效；撤销使用同一操作组恢复被替换块的旧载荷并
退役本次新增块。当前内容可以复用既有块级生成器，但必须在服务端转换为同一方案操作；
前端不得再维护第二套候选、恢复或确认状态。同一请求 ID 只能原子认领一次，关闭、刷新
或换入口后继续恢复同一未完成方案。

### 6.2 Workflow 中加入轻量 AI 场景判断

本节课程生长不建立独立 Agent 层，也不把所有学习场景写成固定分支。现有 Workflow
在“收到需求 / 汇聚证据”和“盘点课程结构”之间加入一次受约束的 AI 场景判断：

```text
用户要求 / 当前课程证据
→ AI 读取本节标题、目标、现有教学 role、课程知识上下文和本节证据摘要
→ 输出场景摘要、目标 role、难度方向、difficulty_delta、资料要求和判断理由
→ 系统过滤非法 role、范围和难度值
→ AI 不可用或输出非法时使用确定性规则结果
→ 系统根据真实 CourseBlock.role 决定 REPLACE 或 INSERT
→ 继续原有候选、同源、确认、提交、撤销和复验链
```

AI 只负责难以穷举的语义理解，不拥有课程写权限，也不能指定任意块 ID、跨小节范围、
正式课程修订或确认结果。它只能在系统声明的教学作用和难度维度中提出结构化判断；
模型输出始终作为计划输入，不是课程命令。

确定性规则不是另一套产品逻辑，而是同一判断契约的保底实现。规则必须覆盖明确的理论、
实战、案例、练习、概念、提高挑战与降低支架表达；AI 可在复杂、间接或组合表达中修正
规则结果。模型失败、超时或返回未知字段时，课程生长仍可按规则继续，且方案记录本轮
使用“AI 理解”还是“规则保底”。

资料要求属于判断结果的一部分。若用户要求最新、前沿、近期行业事实，AI 必须标记
`verified_current_sources`，候选生成器只能使用已提供的可信资料，或生成不冒充最新事实
的任务框架；界面必须明确提示仍需可信来源。AI 不得用普通模型记忆伪装成当前行业证据。

### 7. 课程生长应用使用预检加可恢复操作组

当前文件仓库不具备数据库事务，因此采用：

1. 读取并固定完整 `base_revision_vector`。
2. 对当前课程/知识引用、用户范围和课程操作执行预检与质量门。
3. 写入状态为 `prepared` 的命令组日志和各域预期结果摘要。
4. 通过单次课程文档提交原子写入所选操作，并记录新课程修订。
5. 所有步骤完成后写入 `committed` 回执。
6. 进程重启时对 `prepared/applying` 命令组对账：若可幂等完成则继续，否则执行补偿并恢复原修订引用。

命令组使用稳定 `idempotency_key`。应用端只能消费已接受操作 ID，不能在执行时让模型改变范围或内容。

原因：

- 简单顺序写文件会产生课程操作只应用一半的状态。
- 复制整份仓库快照回滚会覆盖并发合法变化。
- 可恢复命令组与现有生成任务、块候选恢复思想一致，可在当前存储架构上实施。

未来迁移到事务数据库时，仓库接口和领域语义保持不变，可将课程操作组提交替换为数据库事务。

### 8. 难度使用差异向量，不保存新的单一“当前等级”

课程生长方案中的 `difficulty_delta` 只描述相对当前课程契约的局部变化：

```text
knowledge_granularity
abstraction
reasoning_depth
derivation_completeness
example_count_and_distance
scaffold_intensity
task_complexity
integration_scope
pacing_granularity
feedback_frequency
mastery_independence
```

规划器必须把 delta 编译为课程操作。接受后只更新受影响位置的支架、例子、任务或节奏，不把整门课程粗暴切换为 `beginner/intermediate/advanced`。

原因：

- 用户的“这一章推导更细、后面题目更难”是局部多维变化，单等级无法表达。
- 沿用既有难度契约可以复用质量门并避免伪难度。

### 9. 候选存储归属于当前课程，界面共享同一 Store

后端新增仓库接口，建议初始模块：

```text
backend/course_knowledge_base.py
backend/learning_evidence_index.py
backend/adaptation_hypotheses.py
backend/personal_adaptation_plans.py
backend/personal_adaptation_planner.py
backend/personal_adaptation_impact.py
backend/personal_adaptation_quality.py
backend/personal_course_overlay.py
backend/routers/course_evolution.py
```

实际文件可在实现时根据现有模块边界合并，但必须保持领域职责。课程候选、回执和操作历史按当前 `course_id` 保存；当前本地用户的学习事实继续作为证据来源，但不参与课程可见性分支。

前端建议由唯一 `courseEvolution` Store 管理：

```text
pending adaptation plans
operation decisions
scope preview
generation/recovery state
receipts and history
```

`CourseBlockStream`、目录、`KnowledgeLibrary` 和 `SideAIPanel` 只消费投影，不各自保存接受状态。

原因：

- 同一个人方案会在多个入口出现，分散 Store 会形成重复确认和状态漂移。
- 按课程组织便于删除、导出、性能分页和撤销对账。

### 10. 候选交互以原位差异加统一摘要为主

桌面：

- 行级变化原位高亮。
- 新块保持正文宽度并带候选边缘标记。
- 左侧目录显示未来章节待处理数量。
- AI 老师显示本次变更理由、范围和对话解释。
- 多操作个人方案使用统一摘要抽屉或覆盖层，不新增课程工作室。

移动：

- 正文仍显示轻量标记。
- 详细差异、范围树和批量操作在全宽覆盖层中完成。
- 底部固定操作必须避开现有工具坞和安全区。

接受、拒绝、重新生成、补充要求和范围调整全部调用同一 Store 与服务端状态机。

原因：

- 用户必须在课程语境中看懂改了什么，同时又需要跨章节总览。
- 只用行内按钮无法处理跨章节影响；只用独立管理页又会割裂学习现场。

### 11. 候选内容交互与正式掌握严格分离

候选块和候选检查使用 `candidate_ref` 记录交互。它们可以进入修改效果分析，但不能直接形成正式掌握。只有候选被接受，且其中的正式任务通过资产质量门产生稳定题目修订后，未来 Attempt 才能成为正式证据。

原因：

- 用户需要能先阅读候选再决定，但候选可能被拒绝或重生。
- 将候选任务当正式掌握会让历史证据引用不存在的正式内容。

### 12. 适应效果采用前后同类证据比较

接受变更时冻结 `effect_baseline`：问题类型、诊断、用户选择范围、目标块、目标知识、
能力、易错、关键证据、原失败 Attempt 与题目修订。后续评价使用同类证据：

- 相同或相关知识的求助频率。
- 回答“已解决/还不清楚”。
- 正式题目与独立复验。
- 用户明确纠正或回退。
- 候选内容是否产生新困惑。

不使用内容长度、接受按钮点击、动画播放、关键帧切换或参与时长独立证明效果。
结构化动画中的概念判断必须记录明确答案和正确性，但该判断仍只证明课程支持被实际
使用；最终效果必须由方案绑定的全新正式题验证。一个不同正式任务通过只标记
`initial_support`，至少两个不同正式任务持续通过才标记 `confirmed`。效果不足可以
生成新假设或回退候选，但仍需要用户确认持久修改。

### 13. 路径树、知识关系网和叶子能力包必须分工

课程知识结构固定分为三个互补视角：

```text
路径树：CourseDocument 章节 → 小节 → ConceptGroup → KnowledgePoint
关系网：KnowledgePoint 之间的 prerequisite / derives / equivalent_to /
        contrasts_with / applies_to / generalizes
能力包：KnowledgePoint → SkillUnit / Misconception / MasteryCriterion
```

路径树中的 `contains`、上一节和下一节不是知识语义关系；正文、课件和题目到知识点的连接属于 `KnowledgeBinding`；能力、易错和掌握标准是知识点关联实体。正式关系网不得提供无明确教学作用的 `related` 兜底。

每个关系必须保存方向、理由、条件、来源和修订。`prerequisite` 表达移除前置后目标无法或明显难以独立学习；`derives` 表达在条件和步骤下推出；`applies_to` 表达用已有规则处理目标；`equivalent_to` 和 `contrasts_with` 为规范化对称关系；`generalizes` 从一般规律指向特殊情形。多前提关系使用 `relation_group_id + all_of / any_of`，避免把联合条件误读为单边充分条件。

首次生成不能让每个小节一边创建知识点、一边只看前序名称猜关系。系统先完成全部
小节知识节点并编译课程内稳定 ID 注册表；关系阶段再按“当前小节目标点 + 显式
前置小节 + 本章前序小节”形成可恢复邻域。模型只能复制允许 ID，并为每个目标点
提交 `connected / course_entry` 决定。所有邻域最终合并为一张全课关系图；未知
ID、自环、重复、非法类型、遗漏决定和前置或一般化环路属于结构错误，失败只重做
当前关系邻域。关系数量和覆盖率只用于观察，不作为综合评分驱动的整课重写条件。

原因：

- 只有路径树会让知识库退化成章节目录的复制品。
- 只有关系图会失去课程学习顺序、内容分类和可读入口。
- 只有知识名称没有能力、易错和掌握标准，无法驱动题目、诊断、AI 教学和个人状态。
- 三者分工后，同一课程知识坐标才能被正文、课件、练习、AI 和学习事实稳定复用。

### 14. 教案是唯一教学设计真源，课程蓝图是单向派生的机器执行稿

采用：

```text
已确认需求 + 目录
        ↓
同一份全课教案
├── 总体教案：定位、对象、目标、前置、策略、评价与章节安排
└── 分小节教案：知识、能力、易错、掌握标准与课程块职责
        ↓ 本地编译
当前课程知识库 + 教案知识绑定
        ↓ 确定性提取
课程蓝图：当前小节生成所需的机器执行稿
        ↓
正文、例子与练习
```

总体教案和分小节教案是同一教案的两种用户安全投影，不建立两份可独立修改的计划。
总体教案优先复用已经存在的课程定位、学习目标、前置要求、教学画像、章节职责和
掌握标准，本轮不得为宏观文案增加独立模型调用。分小节教案继续由现有全课身份骨架、
有界详细批次和本地唯一汇编产生。

课程知识库与教案不是两个互不相干的串行产物。模型在教案阶段共同规划知识身份、
首次负责、复用、前置、能力、易错、掌握标准和课程块职责；教案汇编后由确定性代码
分配课程内稳定 ID、编译关系与绑定。前端教案只能显示指向正式课程知识 ID 的知识
标签；无法绑定的条目必须明确显示待编译或待复核，不能伪装成普通标签。

课程蓝图保留现有下游兼容价值，但边界改为：

- 只能从已冻结教案、当前课程知识切片和资料、难度等确定性生成约束编译。
- 不提供独立产品入口和独立编辑真源。
- 教案修订变化后，旧蓝图必须按现有修订失效机制过期。
- 大模型对机器执行稿的异议必须回到教案候选，不能静默产生平行教学设计。
- 生成时只装配当前小节与直接依赖的有界切片，不把总体教案或整库重复广播。

原因：

- 教师需要正统、可阅读的教案，模型需要紧凑、可执行的输入，两者可以有不同表达，
  但不能拥有不同教学语义。
- 现有课程生成已经先汇编教案、再编译知识库和课程蓝图；保留这一顺序能复用成熟的
  检查点、恢复与正文生成协议。
- 删除课程蓝图外壳会扩大一次性改动，继续允许其独立编辑又会制造第二真源。单向派生
  同时保留稳定性与概念收束。

### 15. 教师可读总体教案必须对详细教案与正文产生同源引领

采用：

```text
课程计划 + 教学画像 + 分节目标与评价
        ↓ 确定性提取，不调用模型
总体教案：定位、成果、对象、起点、主线、章节推进、评价
        ├── 教师视图
        ├── 有界详细教案批次：落实为课程块职责与讲法
        └── 当前小节正文：落实为内容选择、教学路径与评价证据
```

此前总体目标、教学画像、课程定位已经零散进入生成模板，但教师视图中的章节推进、
学习起点、评价策略和全课主线没有作为同一结构进入下游，因此“总体教案好看”并不
自动等于“总体教案引领课程”。本次建立一个纯确定性的同源提取器：教师视图与生成
Prompt 都从它读取，不另存第二份总体教案。

详细教案批次只接收压缩后的全课引领，并且必须把课程成果、教学主线和评价策略落实到
本节课程块的具体职责与讲法；正文同时接收当前章节责任、本节责任、前后衔接与评价
证据。Prompt 明确禁止复述教案条目，也禁止借宏观目标修改已经冻结的目录、知识身份、
模块集合、资料边界或难度合同。

该变化只升级现有 Prompt 契约，不改变 `1 → N → 1`、模型调用数、调用顺序、共享容量、
检查点、恢复或发布门。同输入对照必须证明：改变教学主线会改变详细教案与正文 Prompt，
而课程块集合和生成调用数量保持不变。

## Quality Gates

每个课程生长方案至少经过：

1. **结构门**：目标存在、操作合法、依赖无环、位置合法、ID 与修订完整。
2. **知识门**：读取的知识、能力、易错、掌握标准和依赖关系存在且引用完整；知识语义写操作必须单独通过当前课程知识质量门。
3. **教学门**：修改与证据问题匹配，避免重复、断层、错误前置和无意义扩写。
4. **难度门**：delta 与真实内容一致，拒绝只增长度、术语或题量。
5. **资产门**：正式题目、掌握标准、媒体和目标引用未被静默覆盖；新增检查明确标记来源和效力。
6. **锚点门**：学习记录与历史事实能够保留、迁移或明确进入待确认。
7. **范围门**：当前课程和作用范围合法，只读写当前 `course_id`，未查询或修改其他课程及跨课程知识设施。
8. **差异门**：候选有实质变化，且没有覆盖范围外内容。

以上规则只用于用户确认后的课程生长候选，不得重新接回首次生成主链。课程生长候选
出现结构化模型输出损坏时最多按明确结构错误纠正一次，不能启动综合评分或无限自循环。

### 首次课程生成改为按预算并行的全课教案规划四步生产线

首次生成继续复用唯一 `GenerationJob`、隔离工作区、`CourseDocument`、
`CourseKnowledgeBase` 与确定性发布预检，但用户只面对一条明确的四步生产线：

1. **需求**：用户提交主题、目标、难度、规模和资料，提交动作冻结需求修订。
2. **目录**：每门课程都先冻结轻量章节骨架，再按章并行、章内有界顺序分片，最后
   本地汇编并审阅课程定位、学习目标、章节、小节、每节责任、前置和范围。
3. **课程生成**：用连续的有界骨架分片冻结全课知识身份和职责，再以有界批次并行展开
   详细教案，最后本地汇编唯一正式计划并编译知识库、稳定 ID、六类关系和逐节绑定，
   随后在全局并发上限内独立生成正文、题目和学习资产。
4. **确认发布**：只检查结构、ID、引用、绑定、上游修订和正式产物可用性，用户
   确认后原子发布。

提交需求视为确认第 1 步；只有目录和发布进入 `waiting_for_review`。正文完成后
系统自动确认同一任务内的内容检查点并继续编译学习资产和发布材料，不再建立额外的
“课程内容确认”人工停顿。教案、知识库和关系图是课程生成内部的同源结构，不单独
等待用户确认。同一个 `job_id` 只有收到与当前待确认步骤匹配的确认命令后才能继续；
服务重启不得跨过目录或发布确认门。

内部使用以下可恢复中间产物控制返工：

- `CourseOutlineSkeletonV2` 只保存课程级定位、目标、章节顺序、各章责任和小节数。
  它不展开全部小节，因此课程规模增加不会线性扩大这次请求。
- `CourseOutlineBatchV2` 每批默认最多展开 6 节。不同章节在有界并发内独立生成；
  同章多个批次保持顺序，只读取相邻章节衔接、当前章已完成小节和最多 4 条相关证据。
  每批完成即保存，`CourseOutlineAssembler` 依据稳定节点 ID 本地汇编唯一目录修订。
- `CourseOutlineRevision` 保存目录与每节责任。
- `CourseTeachingPlanSkeletonV3` 是一份连续增长的全课身份产物。每次模型调用最多读取
  两个待处理小节、去重后的难度与模块合同，以及这些小节直接依赖的前序知识边界；
  本地合并器重排分片临时键并冻结全课知识键、规范陈述、唯一负责、复用、前置和模块边界。
- `CourseTeachingPlanBatchV3` 每批最多处理配置允许的小节数与知识点数；同一课程默认
  并发 2，完成一批立即持久化，失败只纠正或重试当前批次。
- `CourseTeachingPlanAssembler` 按目录顺序合并全部合法批次，生成唯一正式
  `CourseTeachingPlanV3`；语义修订不依赖批次完成顺序。
- `CourseKnowledgeBase / CourseKnowledgeRegistry / CourseKnowledgeGraph` 由系统从
  同一教案确定性编译，分别保存知识身份、稳定课程内 ID、六类关系与精确绑定。
- 正文块、学习资产和发布预检分别保存自己的检查点和当前修订。

每步确认都保存 `artifact_revision` 与 `input_revisions`。后续产物只能消费已确认
上游修订；上游修订变化时，依赖旧修订的下游步骤必须进入 `needs_regeneration`。
最终同源报告逐项验证需求、目录、全课教案、知识库、内容和知识绑定仍属于同一
版本链，任何漂移都阻止发布。

目录确认后不得由后续模型静默增加、删除或重排小节。所有长时间模型调用都必须
周期更新心跳和当前产物；任务事件同时给出 `item_index / item_total / item_id`，
前端以步骤、阶段和完成数量表达进度，不再用一个固定百分比代替真实状态。

目录确认后，系统执行“全局身份有界串行、局部展开有界并行、最终汇编串行”的路径。
所有课程的骨架默认按最多 2 节顺序分片，包括用户明确要求只有 1 节的课程；详细教案
按单个小节拆成独立、可并行、可续跑的工作单元，
并发受课程级与服务级信号量共同约束。每次调用先选择可容纳的完整、紧凑或最小语义
合同；详细批次仍超预算时递归拆分。无法再拆或提供方失败的最小单元由本地编译器保底，
并在正式阶段产物中标记降级与人工复核。教案通过结构检查后，知识库、关系图、稳定 ID
和绑定都在本地同步编译，不占用模型槽位。

完整、紧凑或最小只表示单个有界请求携带的语义载荷等级，不是课程模式。每门课程都由
骨架与详细批次汇编同一 `CourseTeachingPlanV3`；新任务和旧检查点恢复后的正式写入
都只能保存 V3，V2 只作为历史读取输入。用户未指定规模时，brief 写入至少 6 章、
18 节的完整课程基线；显式规模精确遵守，但不改变骨架—批次链。教学画像也属于
确定性编译：用户显式选择优先，否则从主题、
目标和资料元数据按稳定规则解析，不允许为了“再判断一次”增加可阻断目录生成的模型
调用。每次真正的模型请求必须先把任务阶段切换到对应产物，立即失败时也不能沿用上
一阶段的名称。

这条生产线的最小恢复单元是目录章节骨架、目录批次、教案骨架分片、详细教案批次和
单个正文块。模型输出损坏只纠正当前最小单元一次；已经通过的批次不得因其他批次失败
而丢弃。目录全局骨架没有形成有效课程语义时不得伪造整门占位课程；骨架有效后，
单个目录批次失败才允许按该章责任局部保底并标记人工复核。教案骨架冲突时产生受控
的新骨架修订，只使引用变化知识键的批次失效。无论两节还是四十八节，最终都只有
一份正式目录和一份正式教案，知识库和关系图模型调用数都为零。

生成现场使用与正式学习相同的课程页面和 URL。新建课程成功后前端直接导航到该
`course_id`，不先打开任务中心。页面把四步用户路径投影为五个可见生产阶段：
“需求 / 目录 / 教案与知识库 / 正文生成 / 确认发布”；插入“教案与知识库”只为解释
内部真实产物，不增加确认门。顶部一级视图只保留“教案 / 课程 / 练习”，目录由左侧
导航承载，练习在发布前可见但禁用。

教案视图直接读取 `CourseTeachingPlan` 的安全投影。教案尚未返回时按已确认目录显示
小节占位，返回后原地替换为知识点与教学模块，不得再次触发 AI 生成。生成阶段默认
跟随当前内部阶段和章节：教案阶段进入教案，正文阶段进入课程；用户手动选择视图或
节点后停止自动切换，并可显式恢复跟随。发布完成时同一页面从生成投影切换到正式
`CourseDocument`，不得刷新或创建新的课程身份。

首次生成不再设置独立 AI 质量检测链。目录只检查 JSON、目录形状和用户明确规模；
学科模板、难度和编排画像先确定课程块硬骨架，模型只返回知识语义、可选块和局部
强调。本地教案编译器丢弃越权块、补齐必需块及默认职责，并把未显式绑定的新知识
绑定到核心教学块，因此“模型漏写模板复述”不再触发纠正调用。编译后的全课教案再
检查可解析、目录覆盖、稳定归属、具名知识陈述、掌握标准、知识块绑定和合法关系
引用；本地关系编译检查未知名称、自环、重复、非法类型和结构环路；正文只检查非空、
引用合法和必要结构。一般质量偏好不阻断、不追加模型评分或修复调用。

正文生成后不得再调用 AI 执行整课复审、节点打分修复或跨章节自动重写。题目质量
来自 `AssessmentIntent + QuestionSpec + 私有解答合同 + 确定性领域校验`，课程生成
阶段不再对全部题目执行两段式 AI 预解析和失败重写；作答时的个性化诊断仍按需调用
AI。最终发布预检继续守版本同源、稳定引用、基础绑定和正式产物可用性。

### 正式题目改为“生成时编译合同、作答时按需诊断”的同源链

正式题目不能只凭生成时挂上的知识 ID 证明自己考对了内容，也不能只用一个总分
代替对学生作答过程的理解。课程内容阶段在每道主练习、综合检测、诊断题、补救题
和独立复验题上编译 `AssessmentIntent`，明确：

- 为什么需要这道题；
- 声明考查的当前课程知识、能力、易错与掌握标准；
- 学生必须表现出的可观察动作；
- 正确答案必须满足的成立条件；
- 当前难度契约。

题目生成后不再启动独立 AI 解析、映射和失败重写。系统直接从同一份
`AssessmentIntent`、公开 `QuestionSpec`、私有解答合同和学科
`validation_plugin` 编译 `question_analysis_v1`：验证目标 ID 是否都在当前课程
范围内、题面是否给出必要条件与响应格式、答案是否可判定、能力是否可观察、难度和
题型是否仍符合合同。任一结构性条件不成立就把具体题目放入 `waiting_review`；
不能通过再次调用 AI、从头重生课程或静默改写题面掩盖问题。第 3 步向用户展示
“为什么出这道题、它考什么、命中了哪些能力和易错、哪条合同不成立”，而不是展示
一个模型自评分数。

题库内部使用一条明确的公开与私有合同边界。`question_spec_v2` 只保存学生可见题面、
任务、限制、响应格式、来源和风险，并通过 `solution_revision_id` 引用私有
`solution_envelope_v1`；标准答案、量规、隐藏测试、验证器参数和解题图不得复制回
公开题目、持久化的 `formal_task` 或教师题面修订。现有学科适配器继续负责提供可解的
确定性题型、推理路径和领域校验，但只作为 `validation_plugin` 接入 V2 题目，不再形成
第二套公开题目合同。V2 答案修订必须经过私有解答合同，公共题面修订接口不得接受
`answer_spec`。

标准主练习采用“同一语义合同、默认单选投影”的交付方式。每个二级学习目标原有的
`concept_check / objective_practice / mastery_check` 三个任务继续承担不同证据责任，
但在题库编译时统一投影为四选一题面：原标准答案仍是私有语义真源，系统确定性生成
三个可解释干扰项，再把唯一正确选项 ID 写入私有解答合同。练习接口只返回一个固定
三题批次；显式请求或未完成作答优先进入批次并替换普通候选，不允许无限追加。教师
导入题、历史题目修订和显式其他题型的综合检测不被默认策略覆盖。这样可以降低练习
启动成本，同时保留题目语义、评分真源和历史证据边界；代价是单选答案本身不能证明
完整推理，因此诊断只能解释所选命题，不能推断学生未展示的解题过程。史料论证、语言
生产等没有唯一标准成品时，不合成虚假范文，而是把任务改写为识别“唯一完整覆盖全部
材料、限制和可观察量规的作答方案”；三个干扰项分别显式遗漏必要要求，私有合同保存
完整量规方案作为选择依据。

题库覆盖率只计算合同通过且已经批准的正式题目。来源不足、高风险、私有解答合同与
题面不一致或校验插件不可用的候选进入 `waiting_review`，不能因字段完整就算作覆盖或进入学习任务。
题库重建使用带身份的持久异步任务；局部安全通过时只发布明确批准的子集，尚无批准题目
时保留原有可用课程资产并等待审核。题库修订、学习快照和作答等运行时 JSON 属于数据
仓库状态，不是代码或课程蓝图真源，不得作为普通源码提交。

学生提交答案时，量规评分与独立作答诊断并发执行：

```text
正式题目 + AssessmentIntent
          ├─ 量规评分：分数、是否达标、各标准证据
          └─ 作答诊断：学生实际思路、正确部分、关键差距、不确定性
                                ↓
                    映射当前课程知识 / 能力 / 易错
                                ↓
                 学生可见反馈 + 唯一下一步动作
                                ↓
             既有诊断 / 补救 / 独立复验 / 针对再练
```

评分决定本题是否达标，作答诊断解释为什么；二者不得互相伪装。作答诊断只使用
学生答案中的可见证据，不补写学生没有表达的推理，也不泄露完整答案。真实诊断优先
驱动后续诊断假设和针对再练；分析暂时不可用时，已保存答案和原有评分链继续成立，
不得制造一个假的个性化结论。

### 首次生成使用共享容量、活跃超时和教案语义门

目录、教案和正文虽然属于不同业务阶段，但最终竞争的是同一个提供方和模型容量。
因此固定阶段信号量只保留为本地工作单元上限，真正的模型请求统一进入共享容量调度器：

```text
全部待生成单元
  → 按提供方平滑启动
  → 按模型取得在途租约
  → 成功样本连续出现后逐级扩容
  → 429 / 额度错误时立即收缩、冷却并让等待单元改走可用模型
  → 请求结束释放租约，队列继续
```

共享调度的目的不是拒绝课程规模，而是让更多小节成为排队中的独立工作单元，避免四个
协程同时撞向同一个模型后一起失败。初始容量、最大容量和启动间隔是运行参数；系统以
等待换取有效吞吐，不把等待中的单元判为失败。容量状态只记录在途数、当前窗口、成功与
限流次数等非敏感指标，不记录 prompt、密钥或隐藏推理。

结构化目录与教案调用本身使用流式传输。超时语义改为“连续没有任何内容或推理分片的
时间”，而不是从请求开始计算固定墙钟时间；只要模型持续产生数据，调用可以超过原有
60/180/360 秒。整课阶段以全部最小单元结算，不再设置课程规模越大越容易触发的总时限。
底层 HTTP 连接与单次读写仍保留异常保护，但不能代替业务活跃度判断。

教案阶段同时建立语义门。模型批次全部通过时状态为 `ai_complete`，才允许正文消费；
本地编译器只负责稳定 ID、模块硬骨架和失败单元预览。任一知识骨架或详细教案批次使用
本地语义兜底时，状态为 `retry_required`，保存已成功批次和失败范围并停止在正文之前；
恢复时只重跑非模型批次，不重跑已经通过的模型单元。系统不得再把“结构上可解析的本地
模板”写成“全课 AI 教案已完成”。

正文和资产采用逐节确定性收口：流结束后先标准化 LaTeX，再重新执行当前小节质量检查，
未闭合的块级公式只能修复或把当前小节标为失败，不能等到整课结束才暴露；练习合同把空
对象视为缺失并编译真实输入方式，目标练习的量规必须明确检查题面依据、可观察过程和
结果复核。发布审阅合并正文与资产阻断项，用户看到的阻断数必须与发布门一致。

同输入对照以真实供应商完成，不以 mock 代替。固定同一主题、6 章 12 节、资料与模型池，
对比总耗时、每阶段耗时、最大单次载荷、实际提供方尝试、429/额度错误、AI 教案覆盖、
正文模型覆盖、局部降级、最终阻断数和是否可发布。目标不是单纯减少调用数，而是在不把
大文本塞进单次请求的前提下，提高成功吞吐并把失败限制在最小单元。

## Risks / Trade-offs

- **[风险] 每条输入都登记导致数据量快速增长** → 只保存源对象引用、内容指纹和必要索引；按课程分片，支持增量摘要和分页，不复制全文。
- **[风险] AI 过度解释局部证据并频繁改课** → 动态范围门槛、反证、拒绝冷却、同义候选去重和广域更强证据门。
- **[风险] 候选长期堆积使正文难读** → 同一目标与问题合并候选，目录显示数量，正文只显示当前位置差异，提供统一待处理摘要和批量处理。
- **[风险] 局部问题被过度外推** → 动态范围门、最小充分范围、明确差异和用户确认。
- **[风险] 文件仓库课程操作半应用** → 预检、单次课程文档提交、幂等回执、启动对账和补偿，不使用整库快照覆盖。
- **[风险] 拆分合并破坏笔记和历史作答** → 稳定 ID、墓碑、旧新映射、文本锚点重解析和歧义待确认。
- **[风险] 候选中的练习污染掌握** → 所有候选交互绑定 candidate ID，只进入效果分析；正式任务需接受并通过资产质量门。
- **[风险] 大课程全局影响分析过慢** → 依赖索引、按受影响子图计算、后台生成、流式状态和可取消任务；阅读主链不等待分析。
- **[风险] 跨课程设施重新成为隐藏真源** → 新生成与运行链禁止调用并移除活动入口；契约测试以调用监控证明正式流程零访问。
- **[风险] 知识库再次退化成章节标题复制** → 全课教案必须为每个原子知识输出具名且有实际陈述的内容、唯一首次归属、合法复用与课程块责任；标题复制由确定性结构门阻止。
- **[风险] 全课教案 Prompt 随大课膨胀** → 骨架按最多两节分片，每片只读取直接依赖的前序知识边界；详细教案以单节为最小单元，只读取当前知识、复用项和直接前置闭包。最终 Prompt 在发送前从完整、紧凑和最小语义合同中选择，硬门只作为不可绕过的最终熔断。
- **[风险] 取消课程总小节数门禁后目录输出再次膨胀** → 所有课程目录先生成只含章节责任和节数的轻量骨架，再将不同章节并行展开；单章继续按最多六节顺序分片。目录批次不接收全课小节明细，只接收相邻章衔接、当前章已完成摘要和少量相关证据；检查点和本地汇编保证课程规模只增加工作单元，不增加单次请求体积。
- **[风险] 教案、知识库与正文身份漂移** → 知识库、关系图与稳定 ID 只从同一冻结教案编译；正文只读该修订，任何目录或计划变化都使下游失效。
- **[风险] 教案与正文并发放大为提供方限流** → 教案批次和正文都使用全局有界信号量；默认并发只在硬上限内配置。429 打开模型冷却，教案只降级当前最小单元，正文保存草稿或生成明确的本地脚手架并等待人工复核；不得靠无限重试、丢失既有成功产物或绕过结构门换取表面完成。
- **[风险] 学习前置关系把正文生成退化为长串行链** → `prerequisite_node_ids` 继续表达学习顺序，但正文 Prompt 读取冻结的前序责任、知识身份和一致性契约，不等待前序正文完成。只有未来显式建模且确实需要读取前序生成产物的 `generation_dependency_ids` 才能改变生成顺序。
- **[风险] 候选模型、请求重试与节点重试相乘形成小时级等待** → 单个逻辑模型请求限制提供方总尝试次数，而不是对每个候选分别重试；目录与教案的非流式最小调用保留截止时间，正文按连续无新内容窗口识别卡死。持续产出的流不受单节或整课固定总时长限制；卡死时只取消当前小节并保留节点草稿，全部正文以所有小节单元结算为结束条件。
- **[风险] 并行流式草稿反复重写整门课程拖慢事件循环** → 每个正文小节把累计草稿写入独立原子 sidecar，读取工作区时再叠加恢复；定稿、上游失效、发布或删除时清理。正文热路径不得加载、深拷贝、格式化和 `fsync` 整门课程文件。
- **[风险] 只记录百分比无法判断慢在输入、模型还是调度** → 任务和阶段产物只保存 Prompt 字符数、估算输入 token、输出上限、调用数、最大尝试数、并发与耗时，不保存 Prompt 正文、密钥或隐藏推理。
- **[风险] 六类关系被模型泛化滥用** → 关系类型白名单、方向规则、移除测试、条件与理由审查、环检测和候选确认共同约束。
- **[权衡] 用户确认降低完全自动化程度** → 候选可以自动生成并持续可见，用户只在持久改变时确认；这是高权限 AI 可被信任的必要条件。
- **[边界] 学习证据只作用于当前课程** → 这是当前产品规则，不预留跨课程课程生长分支。

## Implementation Plan

### 1. 建立新课程对象与质量评估

- 增加 `CourseKnowledgeBase` 只读消费、证据索引、假设、课程生长方案和课程操作组接口。
- 新课程生成时从 `CourseDocument`、正文块、目标、学习资产和资料建立知识库；不能确认原子知识时阻止发布，禁止用章节标题确定性伪造知识点。
- 新增读取接口和内部质量评估，验证新课程学习主链。

### 2. 收束正式写入口

- 所有持久课程变化统一进入课程维护链。
- `adaptive_blocks` 只承担即时临时支持，不能保存正式课程变化。
- 删除学习者可见课程投影和其他平行正文写入口，不迁移历史课程数据。

### 3. 接入证据与局部候选

- 先登记 AI 用户消息、明确回答反馈和现有正式对象引用。
- 只开启明确局部请求的动态候选，验证误触发、冷却和冲突。
- 再接入多源假设、后续章节预调和广域范围推荐。

### 4. 接入完整课程生长操作

- 按 `PATCH_COURSE_SPAN / INSERT_COURSE_BLOCK / REPLACE_COURSE_BLOCK / RETIRE_COURSE_BLOCK / RESEQUENCE_COURSE_PATH` 依次实现课程操作和锚点迁移。
- 每接管一种操作就删除或拒绝对应旁路写入口。
- 全部操作通过单一课程方案协议后，再启用部分接受和批量课程操作组。

### 5. 接入知识坐标与维护者建议

- 先让课程方案读取知识、能力、易错、掌握标准和依赖范围。
- 再实现课程与知识修订后的候选重定位和冲突提示。
- 最后实现“重复学习问题 → 当前课程改进建议”，知识语义变化进入独立 `CourseAuthoringChange` 维护流程。

### 6. 前端收束与平行状态退役

- 建立唯一 `courseEvolution` Store。
- 让正文、目录、知识库和 AI 老师消费同一候选状态。
- 旧 `adaptive_blocks` 只保留即时临时解释语义；任何持久课程变化改走 `CourseEvolutionPlan / CourseDocument`。
- 完成中英文、多视口、断网、刷新、重启和并发验收后删除旧平行状态。

### 回滚策略

- 新对象和接口以功能开关分阶段启用；关闭时现有正式课程读取与学习主链继续工作。
- 未确认方案可以停用投影而不影响当前课程；已确认变化通过补偿课程命令撤销。
- 已接受变化使用操作日志和补偿命令撤销，禁止恢复整门课程旧快照。

## Open Questions

产品方向与权限边界已经确定，没有需要阻塞实施的产品问题。实现阶段需要通过真实样本校准但不改变契约的参数包括：

- 不同证据类型的初始权重、衰减窗口和同义候选合并窗口。
- 大课程中全局影响分析的异步阈值与分页大小。
- 候选高亮的最终视觉 token 和统一摘要在各视口的具体尺寸。
- 文件仓库阶段的命令组日志保留周期与未来数据库迁移时机。

这些参数必须通过样本、性能和端到端测试确定，不得反向改变“动态而非固定次数、当前课程隔离、用户确认、一门课程一套知识库、外部参考非阻塞”的产品契约。
