## ADDED Requirements

### Requirement: 课程目录与知识蓝图必须分阶段生成

系统 MUST 先生成只包含课程定位、学习目标、章节、小节、每节唯一责任、前置小节
和范围边界的轻量目录。系统 MUST 在目录修订冻结后，先形成覆盖全部小节的职责
与后续保留边界契约，再用目录责任与逐节限量证据提示执行一次紧凑全课规划，形成
`CourseKnowledgeSkeleton`，为每个小节冻结首次负责的规范知识名称和允许复用的
前序名称。每个小节 MUST 只读取
当前职责切片、对应身份合同和本节证据包，并分别生成概念组、原子知识点、能力、
易错与掌握标准。全部知识节点冻结后，系统 MUST 分配课程内稳定知识 ID 并形成
全课注册表；独立关系阶段 MUST 按小节邻域读取该注册表，模型只能复制允许的 ID，
逐一提交 `connected / course_entry` 决定与六类正式关系。系统 MUST 合并所有
关系邻域为一张全课关系图。系统 MUST NOT 在每个逐节 Prompt 中重复完整的全课
职责对象，也 MUST NOT 要求一次模型输出同时承载整课目录、全部知识能力包和全课
关系网，更不能让任何模型创建内部知识 ID。

#### Scenario: 用户要求一章两个递进小节

- **WHEN** 用户原始要求明确指定一章和两个小节
- **THEN** 轻量目录 MUST 恰好包含一章和两个小节
- **AND** 数量验收结果 MUST 写入目录修订与最终质量报告
- **AND** 后续知识包生成 MUST NOT 增加、删除或重排小节

#### Scenario: 一个小节知识包不合法

- **WHEN** 某个小节的知识包不可解析或没有通过原子性检查
- **THEN** 系统 MUST 只纠正该小节知识包
- **AND** 已通过检查的小节目录和知识包 MUST 保持不变
- **AND** 局部纠正仍失败时 MUST 保留检查点并指出具体小节与问题

#### Scenario: 前面小节试图提前完成后面小节任务

- **WHEN** 当前小节生成知识包时包含已在全课职责契约中保留给后续小节的主要任务
- **THEN** 当前小节 MUST 根据全局边界重新生成或进入局部失败
- **AND** 系统 MUST NOT 通过事后删字段伪装为职责没有冲突

#### Scenario: 并行小节生成使用同一个知识规范名称

- **WHEN** 两个小节都需要使用同一个规范知识名称
- **THEN** 知识身份骨架 MUST 指定唯一首次负责小节
- **AND** 其他小节 MUST 通过 `reused_knowledge_names` 引用该名称
- **AND** 逐节详情输出与身份合同不一致时 MUST 只纠正当前小节
- **AND** 系统 MUST NOT 因并发完成顺序创建两个不同知识身份

### Requirement: 每个生成阶段必须拥有独立产物与最小返工边界

同一 `GenerationJob` MUST 持久化课程 brief、目录修订、逐节知识节点包、全课
知识 ID 注册表、关系邻域批次、逐节正文草稿和质量报告。每个阶段 MUST 定义输入、结构化输出、确定性检查、
检查点和最小重试单元。模型或服务失败后，恢复 MUST 从最小未完成单元继续，
MUST NOT 丢弃已经通过检查的阶段产物。

#### Scenario: 服务在第五个小节知识包生成时重启

- **WHEN** 前四个小节知识包已经通过检查并保存检查点
- **THEN** 服务恢复后 MUST 从第五个小节继续
- **AND** 前四个知识包 MUST 保持原修订和稳定知识身份

#### Scenario: 关系批次引用未知知识 ID

- **WHEN** 一个关系批次引用了允许上下文中不存在的知识 ID
- **THEN** 系统 MUST 拒绝该关系候选并只纠正目标关系批次
- **AND** 已冻结的知识结构和系统分配的稳定知识 ID MUST 保持不变
- **AND** MUST NOT 重新生成知识点、目录或正文

#### Scenario: 全课知识点必须逐一进入关系规划

- **WHEN** 所有小节知识节点已经冻结
- **THEN** 每个活动原子知识点 MUST 恰好拥有一个 `connected / course_entry` 决定
- **AND** `connected` 知识点 MUST 存在对应正式关系
- **AND** `course_entry` MUST 保存明确入口理由并继续出现在关系视图中
- **AND** 系统 MUST NOT 只把已有关系边的端点当作全课关系网节点集合

#### Scenario: 关系邻域生成失败后恢复

- **WHEN** 前四个关系邻域已完成而第五个邻域解析失败或服务重启
- **THEN** 恢复 MUST 从第五个关系邻域继续
- **AND** 已冻结知识 ID 与前四个关系邻域 MUST 保持原修订
- **AND** 系统 MUST NOT 重新调用前四个小节的知识节点生成

### Requirement: 知识详情与关系邻域必须受控并行且确定性收敛

系统 MUST 使用服务级、可配置且有硬上限的并发预算执行逐节知识详情和关系邻域。
默认生产上限 MUST 为 4，允许通过环境配置下调或在压测后上调，但 MUST NOT 超过
8。身份骨架、逐节 schema、模型等级、局部纠正、检查点、稳定 ID、关系结构检查和
全课发布门不得因并发而降级。

系统 MUST 先预计算每个并发单元的不可变输入，再按有限公平波次启动模型调用，
避免一门大课预先占满服务级等待队列。完成结果 MUST 按课程小节顺序确定性合并，
而不是按网络返回顺序合并。任一单元失败时，同波次其他已经通过检查的结果 MUST
保存为可恢复检查点，尚未开始的后续波次 MUST 留给恢复流程。

#### Scenario: 二十一节标准课程执行知识与关系规划

- **WHEN** 课程包含二十一个需要生成的知识包和二十一个关系邻域
- **THEN** 默认并发预算下模型等待关键路径 SHOULD 不超过一次身份规划加六轮知识
  详情和六轮关系邻域
- **AND** 相比原先四十二轮串行等待，等待轮次 SHOULD 至少下降百分之六十
- **AND** 每个知识包和关系邻域仍 MUST 独立通过与串行版本相同的确定性检查

#### Scenario: 并发批次中一个单元失败

- **WHEN** 同一并发批次中一个小节失败而其他小节通过
- **THEN** 通过的小节 MUST 按各自修订保存
- **AND** 失败小节 MUST 保存具体问题并成为恢复起点
- **AND** 服务恢复时 MUST NOT 重做已经通过的小节

### Requirement: 首次生成必须按六步依次确认

首次课程生成 MUST 只提供“需求、目录、知识蓝图、教学方案、课程内容、质量与发布”
六步正常路径。提交需求 MUST 冻结第 1 步修订；其余每一步完成检查后 MUST 在同一
`GenerationJob` 中进入 `waiting_for_review`。只有与当前步骤匹配的用户确认命令
才能继续，系统 MUST NOT 提供绕过确认的直接生成模式。

用户只审阅当前步骤的产品内容；内部知识 ID、关系端点、质量矩阵和格式修复 MUST
由系统负责。刷新、断网或服务重启 MUST 保持原待确认步骤。

#### Scenario: 目录等待确认

- **WHEN** 轻量目录已经通过形状和责任检查
- **THEN** 任务 MUST 在第 2 步进入 `waiting_for_review`
- **AND** 逐节知识包、知识关系与正文生成 MUST 尚未开始
- **AND** 用户确认后 MUST 在同一 job_id 内从目录修订继续

#### Scenario: 用户试图越过当前步骤

- **WHEN** 当前等待确认的是知识蓝图而客户端提交课程内容确认
- **THEN** 系统 MUST 拒绝该命令并返回当前待确认步骤
- **AND** 任务、检查点和正式课程 MUST 保持不变

#### Scenario: 用户从后续确认点返回修改目录

- **WHEN** 用户在知识、教学、内容或发布审阅时返回已确认目录并形成新目录修订
- **THEN** 系统 MUST 在同一任务中将知识、教学、内容和发布步骤全部标记为 `needs_regeneration`
- **AND** 重新确认目录后 MUST 清除旧知识、教学、正文、学习资产和质量报告
- **AND** 新知识蓝图 MUST 从新目录修订重新生成

#### Scenario: 用户确认课程内容

- **WHEN** 第 5 步进入 `waiting_for_review`
- **THEN** 完整正文、内容块、练习题、掌握标准、易错与知识映射 MUST 已经生成并完成质量检查
- **AND** 用户确认后发布阶段 MUST 只执行同源与发布判断，不得再静默改写已审阅内容

### Requirement: 六步产物必须形成可验证的同源版本链

每一步 MUST 保存当前产物修订和它消费的已确认上游修订。教学方案只能消费已确认
知识蓝图，课程内容只能消费已确认教学方案，质量与发布必须检查当前目录、知识库、
教学方案、正文、目标、题目、教学表达与知识绑定仍引用同一条已确认版本链。

#### Scenario: 上游修订变化

- **WHEN** 已确认目录形成新修订
- **THEN** 消费旧目录修订的知识、教学、内容和发布步骤 MUST 标记为 `needs_regeneration`
- **AND** 旧下游产物 MUST NOT 进入新的发布链

#### Scenario: 发布前发现内容引用旧知识修订

- **WHEN** 内容或教学资产仍绑定上一次知识蓝图修订
- **THEN** 同源报告 MUST 指出失配对象和应返回的步骤
- **AND** 第 6 步 MUST NOT 允许用户确认发布

### Requirement: 首次生成必须先编译全课一致性契约

课程目录、逐节知识包和 `CourseKnowledgeBase` 确定后，系统 MUST 编译版本化的全课一致性契约。契约 MUST 至少记录有序小节、每节唯一学习责任、硬前置、上下文承接、后续保留边界、知识与能力产出和规范术语。

#### Scenario: 蓝图包含可并行与有依赖的小节

- **WHEN** 课程蓝图完成学科、难度、资料和知识结构校验
- **THEN** 系统 MUST 生成一份与当前蓝图及课程知识库修订绑定的一致性契约
- **AND** 只有真实硬前置才能影响调度顺序
- **AND** 只需上下文衔接的小节 MUST NOT 被伪造为硬依赖

### Requirement: 节点生成必须消费最小必要的全课上下文

每个小节的正文与修复 prompt MUST 获得当前位置、前置产出、本节唯一推进、后续保留内容和规范术语。系统 MUST 允许必要的短回顾，MUST NOT 要求当前节复制前节实质讲解，也 MUST NOT 在当前节提前完成后续小节的主要任务。

#### Scenario: 多个小节并行生成

- **WHEN** 调度器同时生成没有硬依赖的多个小节
- **THEN** 每个小节 MUST 从同一份稳定契约读取自己的责任与边界
- **AND** 一致性 MUST NOT 依赖于其他并行小节先完成正文

### Requirement: 最终质量门必须检查跨小节一致性

首次发布前，系统 MUST 确定性检查小节顺序、依赖有效性、学习责任完整性、显式承接和跨节实质段落重复。短回顾和必要术语重复 MUST NOT 被当作实质正文复制。

#### Scenario: 两节包含近乎相同的长段讲解

- **WHEN** 两个小节出现超过实质段落阈值且高度相似的讲解
- **THEN** 质量门 MUST 记录定位到目标节点的阻断问题
- **AND** 课程 MUST NOT 直接发布

#### Scenario: 后一节用一两句回顾前置

- **WHEN** 当前节仅使用简短术语或承接句建立与前置的关系
- **THEN** 质量门 MUST 允许该内容
- **AND** MUST NOT 为了消除重复而删除必要的学习承接

### Requirement: 自动修复必须定点且可验收

一致性质量门 MAY 对可修复的阻断问题调用一次定点修复，但 MUST 保留未命中小节不变。修复结果只能在目标节点质量通过、原问题消失且全课阻断数下降时接受。

#### Scenario: 模型返回了新正文但没有改善一致性

- **WHEN** 修复候选仍包含原阻断问题、引入新节点问题或未通过节点质量检查
- **THEN** 系统 MUST 拒绝该候选
- **AND** MUST 保留修复前的课程工作区

### Requirement: 契约必须兼容正式课程文档与版本变化

一致性检查 MUST 同时支持任务工作区草稿和正式 `CourseDocument` 投影。当小节顺序或课程知识库修订变化时，旧契约 MUST 重新编译，MUST NOT 继续作为当前生成上下文。

#### Scenario: 学习资产编译更新课程知识库

- **WHEN** 最终学习资产编译产生新的课程知识库修订
- **THEN** 系统 MUST 在最终质量检查前重新签发一致性契约
- **AND** 新契约 MUST 引用最终课程知识库修订

### Requirement: 所有正式题目必须先通过独立解析与同源评判门

课程内容阶段 MUST 为主练习、综合检测、诊断题、补救题和独立复验题编译同源
`AssessmentIntent`，明确题目存在理由、当前课程知识 / 能力 / 易错 / 掌握标准、
可观察动作、答案成立条件和难度契约。系统 MUST 在不暴露课程内部 ID 的情况下先
独立解析题目实际考查目标，再映射回当前课程允许的 ID。独立解析与声明目标不一致、
引用范围外 ID、答案不可判定、存在隐藏评分条件或能力不可观察时，系统 MUST 阻止
内容确认和发布。

#### Scenario: 一道题第一次独立解析未通过

- **WHEN** 题目存在可通过澄清题干修复的歧义、隐藏条件或不可观察动作
- **THEN** 系统 MUST 只重写该题题干并签发新的题目修订
- **AND** MUST 保持原 `AssessmentIntent`、题型、难度和答案量规不变
- **AND** MUST 再次执行独立解析与同源映射
- **AND** 第二次仍未通过时 MUST 显式阻止确认，不得从头重生整门课程或静默放行

#### Scenario: 题目挂了知识 ID 但实际没有考到

- **WHEN** 独立解析得到的实际任务无法命中声明的当前课程知识或能力
- **THEN** 题目 MUST 标记为阻断
- **AND** 第 5 步 MUST 展示实际考查目标、声明目标与阻断原因
- **AND** 用户 MUST NOT 确认该内容进入发布步骤

#### Scenario: 学生提交一份正式答案

- **WHEN** 学生提交已通过题目解析门的正式题目答案
- **THEN** 系统 MUST 并发执行量规评分与独立作答诊断
- **AND** 作答诊断 MUST 说明学生实际思路、正确部分、关键差距和证据不足
- **AND** 诊断映射 MUST 只能使用该题 `AssessmentIntent` 允许的当前课程 ID
- **AND** 学生可见结果 MUST 给出一个具体下一步动作，不得只返回总分或抽象标签

#### Scenario: 作答诊断暂时不可用

- **WHEN** 评分已完成但独立作答诊断模型失败或未配置
- **THEN** 系统 MUST 保存并返回原有正式评分结果
- **AND** MUST 将诊断明确标记为不可用
- **AND** MUST NOT 伪造知识、能力、易错映射或个性化反馈
