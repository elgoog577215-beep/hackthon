# AI 教育产品与开源项目调研：灵知的移植、借鉴与差异化路线

> 调研日期：2026-07-14｜来源：38｜研究模式：Standard｜复杂度：High｜类型：Comparative｜版本：扩展版｜Evaluator：PASS（独立终审 37.2/40）
>
> 对照基线：灵知当前唯一产品真源 `docs/product-blueprint.md`。本报告负责提供外部证据和融合建议，不自动改变产品蓝图与实施进度。

## 执行摘要

本轮共批准 38 个来源，覆盖 8 个开源仓库、10 个闭源产品实体、7 个效果或架构来源和 9 个灵知本地真源。核心结论是：**“一句话生成课程”已经成为商品化入口，真正仍然稀缺的是生成之后的持续学习闭环。** Coursebox、Sana 已把目标、文档和网页转成可审阅课程；A-Learn、Parlina、Oboe、StudyFlow 已把地图、课时、测验、多媒体、断点与复习组合为个人学习体验；Coursebox、Sana 公开了课程内 Tutor，Mindsmith 则把 analytics 连接到课程修改。[9][10][11][12][15][17][18][21][22]

市场能力可以分成三层：第一层“一句话/资料→大纲、正文、测验、多媒体”已高度普及；第二层“课程地图、连续学习、上下文 Tutor、进度与复习”正在成为标准体验；第三层“正式学习事实→可重算模型→诊断补救→独立复验→受控改课”仍缺少公开完整样本。在本轮 18 个开源与闭源样本中，没有一项在七环能力矩阵中全部获得明确公开证据；这只是基于公开资料的有限判断，不代表整个市场绝对不存在。[1][5][11][15][17][23][24][27][29]

对灵知的判断不是“我们功能更全”，而是：**已经拥有较强的系统骨架，但正在从能力存在走向闭环成立。** 当前 `CourseDocument / CourseBlock`、资料证据、正式作答与诊断、可重算 `LearnerModel`、无存储 `LearningRuntime`、最小 `AIContextPackage`、学科知识映射和质量发布门已经明确课程、事实、画像、AI 与正式修改的所有权；最明显的断点是学习反馈还不能稳定聚合为 `CourseQualityIssue`，经确认修改后也缺少正式效果观察窗。[30][31][32][33][34][35][36][37][38]

开源侧不建议 fork 任一完整平台。最佳策略是“能力切片 + 接入契约 + 领域命令”：

- **选择性移植** OpenTutor 的并行派生能力、独立容错和实验状态标记，以及 AI Course Builder 的稳定目标注册表、跨产物一致性检查和构建器。[1][6]
- **隔离试验** OATutor 的无状态 BKT 公式、技能映射和提示阶梯；输入只来自正式作答，输出只是可重算投影，不引入其题库、Firebase/LTI、旧前端或持久状态。[5][24]
- **独立实现设计思想**：Bloom 的“学习—反馈—约束下一课”，Lunaris 的 typed draft、先修 DAG、逐 claim 校验和 coverage gate，ChatTutor 的稳定 page/object 白板。[2][3][7]
- **不引入整仓** LearnHouse 与 Tutor-GPT：前者会带来第二套 LMS、编辑器、权限和课程实体，后者会带来第二会话/画像与外部 SaaS 依赖；两者还有 AGPL/GPL 边界。[4][8]

闭源侧应借鉴可观察的成熟机制，而不猜测代码和内部状态：Coursebox 的多来源生产工作流；Sana 的候选审阅、位置/进度感知 Tutor 和“定位—过程—复习”三段自适应；Mindsmith 的页面/题目级 analytics；A-Learn/Parlina 的马上可学、断点和课内 Tutor；Oboe 的自动媒介编排；StudyFlow 的资料到复习计划。[9][11][12][13][15][17][18][21][22] 灵知应在这些机制前后补上它们未公开证明的控制层：稳定锚点、正式证据、修改确认、修订冲突、撤销、历史冻结和修改前后效果复验。

效果证据要求保持克制。Khanmigo 的平台实验支持“结构化近期作答与先修缺口比原始聊天日志更有用”，但报告的是近迁移；OATutor 显示经过质量筛选的 AI 提示可能带来学习增益，同时原始单次生成有 32% 未通过质量检查；ALEKS 则说明精细模型存在并不保证真实课堂发生个性化。[23][24][27][28] 因此，灵知每个 AI 教学动作都必须同时经过内容正确性/泄题门、独立近迁移、延迟保持和实施忠实度四层验证。

本报告给出的近期优先级是：

1. 先补 claim-level 证据语义和统一 objective/knowledge ID，让正文、题目、错因、补救和多媒体共享同一坐标；
2. 建立最小 `CourseQualityIssue → ActionProposal → 确认 → CourseCommand → Receipt → 效果观察窗`；
3. 再做 OATutor BKT 离线回放与只读投影；
4. 最后扩展稳定对象白板、跨学习者质量聚合和更丰富的多模态教学。

战略上，灵知不应把自己定义为“更完整的 AI 课程生成器”，而应把课程理解为可验证的编译产物：基础课程稳定，个人运行时叠加临时解释与补救，重复学习证据触发受控局部重新编译。最匹配的首个市场切入口，是“带具体目标与资料、希望在数周内形成可验证掌握的深度自学”，而不是泛化任何主题或立即复制企业 LMS。正式定位建议为：

> **灵知是一套以可追溯课程块为学习现场、以正式学习事实驱动可解释辅导，并把重复学习问题转成可确认课程改进的个人 AI 学习系统。**

面向用户压缩为：

> **一句话开启一门课，真实学习证据持续帮助你学会。**

## 调研方法与样本边界

本轮以 2026-07-14 可公开核验的信息为截点，先用灵知的完整学习主链定义“相似”，再检索三类对象：主题相似的 AI 课程产品、能力相似的开源组件，以及能验证类似效果的实验或论文。只有来源可定位、能力与本报告决策直接相关、且能说明边界的样本才进入注册表。

最终批准 38 个来源：8 个开源仓库、14 个闭源产品官方页面、7 个效果或架构来源、9 个灵知本地真源。其中 14 个闭源页面合并为 10 个产品实体，因此后文能力矩阵共有 18 个样本，而不是 22 个来源条目。另剔除 4 个样本：两个与当前缺口相关度低或缺少可用许可证的泛化 Tutor 项目，以及两项不能为产品效果提供独立证据的材料。开源仓库核验许可证、关键实现、依赖与成熟度；闭源产品只记录官方公开机制，不推断内部数据结构；效果判断优先论文与实际实施研究，并显式保留厂商实验和研究设计的限制。

本报告没有登录全部闭源产品，也没有逐一部署开源项目或用同一道题做盲测。因此，“✓”只能表示公开来源明确支持，“△”表示存在相邻能力但未达到灵知的正式链路口径，“—”表示本轮来源未公开，不能解释为产品内部一定不存在。

## 1. 调研基线：市场已经做到什么

本报告以灵知完整主链作为相似度基线，而不是只比较首页是否有一个 prompt 输入框：

```text
目标与资料
→ 可追溯的课程与学习资产
→ 阅读 / 提问 / 作答 / 复习事实
→ 可解释的学习者模型
→ AI 老师与诊断补救
→ 独立复验
→ 可确认的课程改进
```

公开产品大致形成三层市场。

| 市场层 | 代表产品 | 已公开能力 | 对灵知的意义 |
| --- | --- | --- | --- |
| AI 课程生产与交付 | Coursebox、Sana、Mindsmith、Mini Course Generator | 多来源生成、大纲审阅、富媒体和测验、课程发布、学习分析或 Tutor。[9][10][11][12][13][14][15][16] | 说明“资料/一句话→完整课程”已成为成熟卖点，灵知应学习生产流程而非重复定位。 |
| 个人 AI 学习产品 | A-Learn、Parlina、Crescin、UCHU、Oboe、StudyFlow | 课程地图、课时、测验、课内聊天、笔记/断点、游戏化、备考计划和复习。[17][18][19][20][21][22] | 说明用户已经期待生成后可以直接学；课程地图、恢复和课内 AI 属于基础体验。 |

按这个比普通“功能清单”更严格的口径，18 个样本中没有一项在七列都得到明确公开证据。最接近的 Sana 仍未公开正式状态字段、独立复验语义和受控课程改进；Mindsmith 虽已把 analytics 连接到改内容，却未公开确认、撤销、历史冻结和修改后效果窗口。[11][12][13][15] 这是对本轮公开证据的有限结论，不是对整个市场的存在性证明。

这三层目前大多彼此分离：课程生产工具主要服务作者与企业培训，个人学习产品重视低门槛和可见进度，智能辅导系统则集中在数学、技能题或单一学科。BMC 2026 的知识图谱 AI 导师研究报告了延迟保持的积极信号，但只有单校 6 个班，且学生级分析未校正整群随机带来的相关性；它只能作为整套教学处理的待复现信号，不能证明任意 AI 课程平台都具备同样效果。[29]

因此，更准确的判断不是“市场上没有与灵知相同的产品”，而是：灵知的每一段能力都已有强参照物，但本轮公开证据尚未显示某个样本把它们放在同一正式事实链上并完成验证，仍有产品空间。

### 1.2 哪些能力已经商品化，哪些仍然稀缺

如果把市场按“用户能看到的功能”来比较，产品会显得彼此差异很大；如果按“学习系统真正承担了什么责任”来比较，市场分化反而很清楚。

| 能力层 | 市场成熟度 | 已有代表 | 对灵知的判断 |
| --- | --- | --- | --- |
| prompt/文件/URL 生成大纲与正文 | 高，已经商品化 | Coursebox、Sana、Mini Course Generator、A-Learn、Parlina、Oboe、StudyFlow [9][11][16][17][18][21][22] | 必须做好，但只能作为获客入口和起课效率，不能再承担核心差异化。 |
| 课程地图、课时、测验和多媒体 | 中高，已成为用户预期 | A-Learn、Parlina、UCHU、Oboe、Mindsmith [14][17][18][20][21] | 应成为默认产品体验；重点不是“有没有”，而是是否与同一课程目标、证据和学习状态对齐。 |
| 课程内上下文 Tutor | 中，快速普及 | Coursebox、Sana、A-Learn、Parlina [10][12][17][18] | 需要进入正文现场并理解当前位置，但不能退化为把课程全文塞进聊天窗口。 |
| 来源追溯与生成质量门 | 中低，少数产品或项目明确披露 | Lunaris、Coursebox、Sana、灵知现有资料与质量链 [3][9][11][33][38] | 这是可信课程的基础设施，也是灵知现有积累可以放大的第一道壁垒。 |
| 正式学习事实与可重算学习状态 | 低，公开披露有限 | OATutor、ALEKS；Sana 只公开到 progress-aware Tutor [5][12][13][27] | 不能用聊天记忆、完成率或用户标签替代。灵知应把“证据从哪里来、怎样重算”做成内部核心协议。 |
| 诊断—补救—独立复验 | 低，多数只到测验或复习 | OATutor、ALEKS、Sana [5][13][27] | 真正的个性化不是多讲一次，而是识别错因、最小补救，再用没有泄题的新任务验证。 |
| 学习证据驱动的受控改课 | 极低，公开样本多停在 analytics 或 Agent 建议 | Mindsmith 最接近，Sana 提供候选审阅边界 [11][15] | 这是灵知最有机会建立系统性差异的位置：聚合问题、提出修改、确认执行、可撤销并复验效果。 |
| 长期保持、跨情境迁移与实施忠实度 | 证据稀缺 | Khanmigo、OATutor、KG-RAG、ALEKS 实施研究、医学知识图谱试验 [23][24][26][28][29] | 这是所有产品都不能轻率宣称的最终结果层。灵知应把它变成实验体系，而不是宣传口号。 |

这张表带来一个关键转向：**市场竞争的前半段正在从“能不能生成”转向“生成得多快、多丰富、多好看”，而真正尚未被充分解决的后半段是“系统凭什么知道学习发生了，以及它依据什么采取下一步动作”**。前半段主要是模型、模板和媒体供给竞争，容易被基础模型升级迅速抹平；后半段依赖稳定对象、正式事实、知识坐标、领域命令、效果门和长期数据治理，更接近系统能力而不是单次生成能力。

### 1.3 竞争对象不是一个产品，而是三种优势的组合

灵知不应寻找一个“完全一样的竞品”然后逐项复制。它实际面对的是三组最强参照的组合：

1. **Coursebox / Sana / Mindsmith 的生产与商业完成度**：它们更擅长让作者快速导入资料、审阅大纲、编排互动内容、发布和查看分析。[9][11][14][15]
2. **A-Learn / Parlina / Oboe / StudyFlow 的个人学习外壳**：它们更擅长把一句目标迅速变成地图、课时、多媒体、进度和复习，让用户马上感到“有东西可学”。[17][18][21][22]
3. **OATutor / ALEKS / Khanmigo 的教学状态与效果方法**：它们提供技能映射、结构化作答、知识追踪、先修补救和线上实验的可借鉴骨架。[5][23][27][28]

因此，灵知真正要构建的是“生产完成度 × 学习体验 × 教学证据”的乘积。只补其中一项都不够：只有严谨内核而没有快速起课和连续体验，用户感知不到价值；只有漂亮课程和聊天，难以形成持续教学差异；只有测评模型而没有课程现场，又会退化为狭窄题库系统。

**Confidence：High**

**Basis：** 14 个闭源官方产品来源、8 个开源仓库和多项效果研究共同支持市场分层；对实际生成质量与稳定性的判断仍保留未实测边界。

## 2. 闭源产品：七类机制如何借鉴

闭源产品的价值在于观察成熟交互和商业化取舍，不在于猜测其内部代码。借鉴时应同时回答四个问题：借什么、落到灵知哪里、保留什么边界、明确不照搬什么。

| 机制 | 主要参照 | 灵知应借鉴 | 落入现有模块 | 不照搬 |
| --- | --- | --- | --- | --- |
| 生成入口 | Coursebox、Sana、StudyFlow | 同时接受目标、文档、图片、URL；AI 先生成可审查大纲或候选卡片，再确认进入正式课程。[9][11][22] | `INPUT → MATERIAL → DESIGN → PLAN`、GenerationJob 工作区、资料覆盖报告 | 一句话直接发布；让用户修内部块结构；把整份来源放进长期导师记忆 |
| 课程地图 | A-Learn、Parlina、UCHU、Oboe | 创建后立即看到模块、顺序、当前位置和目标；地图是课程与知识坐标的可视投影。[17][18][20][21] | `CourseDocument / Section / Group`、知识树、左侧目录 | 游戏地图或 AI 自造节点成为第二套知识真源；向学生暴露版本和块拖拽 |
| 多模态块 | Coursebox、Mindsmith、Oboe | 由学习目标和教学角色决定何时使用文本、图、代码、音视频与题目。[9][14][21] | `CourseBlock.kind + role + asset_refs/evidence_refs`、MediaAsset、质量门 | 为了“丰富”而堆媒体；复制二进制资产；把生成媒体默认当作有效教学设计 |
| 课程内导师 | Sana、Coursebox | Tutor 读取当前课程、目标、位置、必要进度与限定来源，支持选区解释、苏格拉底追问和局部测验。[10][12][13] | `LearningRuntime → LearnerModel → AIContextPackage` | 通用聊天机器人；倾倒全部聊天；未经授权的长期记忆；一次错误自动写画像 |
| 测评与补救 | Sana、A-Learn | 把定位、过程检查、个性复习分开；保留题目来源、提示使用和尝试历史。[13][17] | `PracticeTask → PracticeAttempt → DiagnosticCase → RemediationSession → 独立复验` | AI 评分或一次答对等同掌握；临时 AI 题直接写入正式事实 |
| 复习 | Sana、StudyFlow | 把知识缺口、到期时间、考试目标和新旧内容混合变成可执行复习任务。[13][22] | `review_task`、`review_checkpoint`、LearningRuntime 到期投影 | 固定比例当普适算法；逾期等同不会；复习结果绕过独立验证 |
| 课程反馈迭代 | Mindsmith、Sana | 学习分析先形成问题和修改候选，再由作者或授权流程接受；修改不回写历史学习状态。[11][15] | `CourseQualityIssue → ActionProposal → CourseCommand → Receipt → 前后效果对照` | 单次困惑或掉点直接改课；Agent 静默改正文；更新课程后重置已完成状态 |

其中三项最值得优先吸收。

第一，Sana 将资料转换和正式插入分开，候选可以接受、丢弃或继续对话；这与灵知现有生成工作区和候选块方向一致。[11] 第二，Sana Tutor 明确把课程内容、课程目标、learning journey 和 progress 纳入回答，而不是只把“个性化”写成营销词。[12][13] 第三，Mindsmith 把停留、退出、逐题结果和尝试历史用于发现具体课程问题，但灵知应在它之前增加证据独立性，在它之后增加确认、撤销和效果复验。[15]

闭源产品普遍没有公开 Tutor 的长期状态字段、置信度、遗忘规则和跨课程隔离方式。Sana 虽公开“consider learner progress”，也没有披露这些底层契约；因此，课程上下文聊天、完成率、streak 或聚合分析都不能直接等同于正式学习者模型。[10][12][17][18][19][20]

### 2.1 Coursebox：学习它的“生产工作流”，不要误读为学习闭环

Coursebox 已经把 PDF、Word、PowerPoint、纯文本、YouTube 和网页 URL 等多来源入口，与可编辑大纲、课程正文、视频和测验组合成相对成熟的课程生产体验；课程内 Tutor 可以被限制在课程与上传资料范围内。[9][10] 它对灵知最大的启发不是再增加一种文件格式，而是把“来源进入—结构提取—大纲预览—内容生产—学习者预览—正式发布”做成连续、低摩擦的作者体验。

灵知当前资料管线和生成工作区已经比普通生成器拥有更清晰的证据、状态与发布边界，[31][33][38] 但在用户感知上仍需要吸收 Coursebox 的产品化能力：上传后立即显示解析范围、可用证据和预计生成物；蓝图审阅时让用户改“目标、深度、顺序和资料用途”，而不是修内部 `CourseBlock`；发布前提供真正的学习者视角预览。不能照搬的是把创作会话记忆、资料检索和长期学习者状态混为一体，也不能因为 Tutor 被来源限制就推断它已拥有正式掌握模型。

### 2.2 Sana：本轮最完整的闭源机制参照

Sana 值得拆成三段看。第一段是生成：文件可以被转换为候选大纲和学习页，候选内容在进入正式课程前可接受、丢弃或继续对话迭代。[11] 第二段是 Tutor：它明确读取具体课程、学习目标、当前位置和进度，并支持问题建议、选区解释、苏格拉底追问、实例和测验。[12] 第三段是 adaptive learning：placement test、过程 challenge 和 personalized review 被拆成三个不同环节，而不是笼统写成“AI 个性化”。[13]

这三段几乎对应灵知的“课程候选—LearningRuntime—AIContextPackage—正式测评/补救”链，但仍有一个决定性差异：公开资料没有披露 Sana 的状态字段、置信度、遗忘规则、独立复验或改课控制协议。因此灵知应借用它的**交互分层**，而不是假设并复制一个看不见的内部模型。

### 2.3 Mindsmith：最接近课程质量回流，但仍缺治理层

Mindsmith 的优势在 authoring 与 analytics：它从文档、演示、视频、SOP 等来源生成 storyboard 和 30 多种互动单元，并追踪跨会话进度、逐页停留、退出点、逐题结果和多次尝试；Analytics Agent 可以定位具体内容问题并修改课程。[14][15] 这比只看课程评分更接近“真实使用反向改进内容”。

但灵知不能照搬“发现问题后直接修改”。停留时间长可能意味着困难，也可能意味着内容重要；退出可能来自外部打断；一道题反复错误可能是题目歧义、前置缺口、界面问题或真正不会。灵知应在 Mindsmith 之前增加证据解释层：同一课程修订、同一稳定锚点、多个独立学习者或同一学习者跨时间的重复证据，才能形成 `CourseQualityIssue`；在它之后增加治理层：影响范围、修改候选、确认权限、预期修订、幂等回执、撤销和前后观察窗。[15][30][38]

### 2.4 A-Learn、Parlina、Oboe、StudyFlow：学习“马上进入学习”的感觉

A-Learn 的强项是将目标、水平、深度和语言迅速变成模块、课时、测验、考试和节点图，同时保存断点、笔记、高亮与尝试历史；课内 Chat 理解当前内容。[17] Parlina 把主题、文档或链接变成课程地图，加入旁白演示、开放题评价、答案修订和受当前课程约束的语音 Tutor。[18] Oboe 从学习目标反向规划章节，并自动选择文本、Podcast、测验、Flashcard、STEM 图表和代码教学等形态。[21] StudyFlow 则把 PDF、照片、URL 或主题收束为分章摘要、Bloom 分层题、考试计划和新旧内容混合复习。[22]

四者共同的产品精华是：**用户完成一句输入后，不需要理解“课程生成系统”，马上就能看见路线、开始第一小节、完成一次动作并知道下次从哪里继续。** 灵知应把深层架构藏在这种连续体验后面。尤其是多模态选择，应像 Oboe 一样由教学目标和内容类型自动编排，而不是让用户在创建课程时勾选“要不要播客、图表、卡片”。但 A-Learn 的节点图、UCHU 的游戏地图或 StudyFlow 的固定 60/40 比例都只能作为交互参照，不能升级为新的正式知识图或通用教学算法。[17][20][21][22]

### 2.5 Crescin、UCHU、Mini Course Generator：有价值，但证据要降权

Crescin 的公开页面覆盖分级解释、课时 Tutor、自动评分和难度复习题，表面上与灵知重合很高；但本轮只有营销页或客户端表面证据，无法确认这些功能是否共享同一学习状态，因此矩阵中大量记为“△”。[19] UCHU 证明角色伙伴、积分、联赛、成就和离线同步可以把生成课程包装成移动端日常习惯，但这些机制不能替代学习证据和保持验证。[20] Mini Course Generator 的成熟点是微课生产、互动块和导出，公开重点仍是创作与分发，而不是长期个体模型。[16]

它们对灵知的意义是提醒产品不能只有正确架构，还需要节奏、反馈和完成感；但游戏化、证书、排行榜、模板数量和“更上瘾”不应在核心闭环之前成为研发主线。

**Confidence：Medium-High**

**Basis：** 结论来自产品官方页面和帮助中心，能证明功能宣称与交互机制；未登录产品，因此内部状态模型、默认授权范围和真实稳定性仍未确认。

## 3. 开源项目：移植决策矩阵

灵知不需要另一套教育平台底座，而需要补齐现有链路中的特定机制。移植决策首先看数据真源是否兼容，其次才看功能多少。

| 项目 | 决策 | 可吸收部分 | 灵知接入位置 | 许可证与风险 |
| --- | --- | --- | --- | --- |
| OpenTutor | **选择性移植** | 资料摄取后的 notes/flashcards/quiz 并行编排、独立容错、错误汇总和实验状态标记 | 现有 MaterialAsset/ParsedDocument/EvidenceUnit 与 CourseBlock 编译链 | MIT；保留版权与许可。项目仍是单用户 public beta，LOOM/LECTOR 等能力标为 experimental，不引入其课程工作区和独立画像。[1] |
| Bloom | **只借鉴设计** | “完成一课—批注/反馈—生成下一课”的渐进体验 | 正式反馈先进入 LearningEvent/Record，再由 Runtime 和课程候选命令处理 | MIT；不复制其 prompt-memory 状态机。一次措辞不能成为长期掌握或自动改课依据。[2] |
| Lunaris | **只借鉴并独立实现** | 先修 DAG、逐 claim 证据校验、coverage gate、Draft/降级契约 | 加强现有 GenerationWorkspace、material evidence、学科知识映射和 final quality report | AGPL-3.0；复制进网络服务可能产生源代码提供义务，需法务确认。当前实现耦合 Claude/Tavily/Voyage/Supabase，且本轮未运行验证。[3] |
| LearnHouse | **不建议引入** | 仅参考 `plan → iterate → finalize → activity` 的规划会话 | 如采用，映射为现有 GenerationJob 检查点和正式发布命令 | AGPL-3.0，Enterprise 另有商业许可证。整仓会带入第二 LMS、组织权限、作业和编辑器，双写成本高。[4] |
| OATutor | **仅隔离移植算法边界** | 标准 BKT 更新、技能—题目映射、最低掌握选择和分层提示思想 | 以 PracticeAttempt/Event 为输入，在 LearnerModel 中形成只读知识点投影，由 Runtime 决定候选动作 | 代码 MIT、题目内容 CC BY 4.0，必须分开治理。参数与技能映射若未经校准会制造虚假精确度。[5][24] |
| AI Course Builder | **选择性移植** | 稳定 LO registry、slides/notes/audio/quiz 一致性检查、静态导出器 | LO 映射为 objective/knowledge ID，多产物成为 CourseBlock/MediaAsset 派生物 | MIT；保留许可。避免引入其 Agent 目录和 HTML player 作为课程真源，LLM coherence judge 需确定性校验兜底。[6] |
| ChatTutor | **只借鉴设计** | 有稳定 page/object ID 的多页白板，AI 更新对象而非输出截图 | `diagram/graph_embed` CourseBlock 或 MediaAsset，正式写入仍经课程命令 | AGPL-3.0；不复制代码和 prompt。必须加入 schema、沙箱、资源限额、对象版本与失败回退。[7] |
| Tutor-GPT | **不建议引入** | 仅把“检索和用户上下文先于回答”作为反例参照 | 当前 AIContextPackage 已覆盖更严格的最小证据装配 | GPL-3.0，且依赖 Honcho、Supabase、OpenRouter、Stripe。会形成第二会话、第二画像和外部 SaaS 锁定。[8] |

这里的“直接复用”并不等于复制整个 OATutor 模块。可考虑复用的只是输入输出明确、没有持久状态的 BKT 计算边界；题库、前端、Firebase/LTI 和内容资产不在范围内。[5] 同理，OpenTutor 中与灵知课程或学习状态边界重叠的能力不应直接进入主链，因为灵知的正式事实、诊断和知识坐标已经存在，外部模块只能作为无状态算法、编排器或设计参照。[1][30][31][32][33][34][35][36][37]

许可证处理建议固定为四档：MIT 代码在依赖审计、保留原版权与许可声明、建立来源清单并通过测试后可以隔离移植；CC BY 内容必须单独署名和治理；GPL/AGPL 项目默认不复制到主产品，只做独立实现参考；任何 AGPL 网络服务边界、模型权重、字体、素材和第三方 API 条款仍需专项法律审查。以上只用于技术筛选，不构成法律意见。[1][2][3][4][5][6][7][8]

### 3.1 OpenTutor：可迁移的是编排与降级，不是另一套学习工作区

OpenTutor 的真实主链不是“一句话生成完整课程”，而是先创建名称与描述，再上传资料，由 `auto_prepare` 并行生成 notes、flashcards、quiz 等派生产物。[1] README 同时把产品限定为本地单用户 public beta，并把部分能力标为 experimental，因此不能把路线图或实验入口视为已经验证的稳定闭环。[1]

最值得移植的是三件小而明确的东西：派生能力并行执行、单项失败不拖垮全部结果、把稳定/实验/不可用状态公开给上层。落到灵知时，输入仍是现有 `MaterialAsset / ParsedDocument / EvidenceUnit`，输出只允许成为课程块或学习资产候选；错误汇总进入 GenerationJob/资料状态，不新增 OpenTutor 课程、画像、图谱或复习状态。换言之，移植的是 orchestration pattern，不是 product domain。

### 3.2 Bloom：最好的地方是渐进体验，最危险的地方也是渐进体验

Bloom 从 topic 生成 syllabus 与第一课，随后把上一课反馈、思考题答案和划线问答放入下一课 prompt。[2] 这形成非常顺滑的“学完一课，下一课更懂我”感受，适合灵知借鉴为：完成当前单元后，系统用正式反馈和作答事实解释下一单元的侧重点，并允许生成临时补充块。

风险在于 prompt 上下文会把一次措辞、一次情绪或一次错误放大为长期判断。灵知不能复制这种隐式状态机：用户明确提交的笔记、问题和反馈应进入各自正式对象，作答进入 `PracticeAttempt`，模型从事实重算；下一课若需要改变正式课程，只能先形成候选并经过 CourseCommand。基础 `CourseDocument` 应保持稳定，个人变化优先进入可跳过、可收起的 `AdaptiveBlock`。[2][30][35]

### 3.3 Lunaris：本轮最值得重写式吸收的质量架构

Lunaris 从一个 topic 依次执行需求解释、研究、learner frontier、先修 DAG、课时编写、claim verification、资源整理和 coverage gate；它把确定性先修关系与逐 claim 支持状态放在 Agent 不能绕过的工具边界中。[3] 对灵知而言，精华不是其具体 prompt，而是三条契约：课程草稿是 typed artifact；重要主张以 `SUPPORTED / CUT` 形成明确发布边界；质量不足时进入 Draft 或阻断，而不是用自然语言“尽量修好”后继续发布。灵知可以在自身资料与质量模型中进一步表达 `CONFLICT / GAP`，但这属于拟议扩展，不是 Lunaris 已核验的原生状态。[33][38]

不建议复制源码：项目为 AGPL-3.0，真实管线耦合 Claude、Tavily、Voyage 和 Supabase，缺 key 时还会进入 stub/Draft；本轮也没有完成实际部署验证。[3] 灵知应在现有工作区、资料证据、学科知识关系和质量报告中独立实现同类契约。尤其不能再建第二张先修图：claim verifier 只能读取当前正式知识坐标并返回证据判定。

### 3.4 LearnHouse：参考“规划—正式化”，不要吞下一整套 LMS

按本轮公开功能广度，LearnHouse 是最接近完整 LMS 的开源样本：它包含 AI courseplanning 的 start、iterate、finalize、generate activity、save activity content，以及课程、块编辑器、作业、讨论、白板、代码执行、证书、组织和权限。[4] 对灵知最有用的是把规划会话与正式课程分开：先迭代 CoursePlan，确认后正式化结构，再逐活动生成和保存。

但整仓引入会把灵知带回最危险的状态：两个课程模型、两个编辑器、两个作业系统、两个权限与进度体系。AGPL 与 Enterprise 双许可证又增加商业边界复杂度。[4] 因此只把其阶段门映射到现有 GenerationJob 检查点、蓝图审阅和发布命令；不引入任何 LearnHouse 领域实体。

### 3.5 OATutor：可以成为算法试验台，不能成为第二个状态系统

OATutor 用稳定技能 ID 将题目、BKT 参数、提示和日志绑定：每次回答更新技能掌握概率，再按最低平均掌握选择下一题，并提供分步题与 hint/scaffold。[5] 在本轮样本中，它是机制与实现证据较清楚的“事实—知识追踪—教学动作”骨架，也有对应学习实验。[24]

移植应严格收缩为无状态函数和映射协议：`update(prior, correct, learn, guess, slip) -> posterior`；所有输入来自灵知正式作答与技能映射，所有输出只是 `LearnerModel` 的可重算投影，不能建立可写 BKT 表。题目内容单独为 CC BY 4.0，不能因为代码 MIT 就一并复制；React 16、Firebase、LTI、题库和旧前端均不进入灵知。[5] 在参数未用真实回放校准前，界面也不应显示看似精确的“掌握 73%”。

### 3.6 AI Course Builder：用一个目标注册表约束所有产物

AI Course Builder 从 topic 生成带 Bloom 标签的 objectives、Marp slides、notes、TTS script/audio、section quizzes 和静态 player；核心约束是所有产物引用不可随意重编号的 LO registry，再做跨产物 coherence review。[6] 这对灵知非常实用：同一个 objective/knowledge ID 应贯穿正文解释、例子、图示、音频、正式题目、复习检查点和导出包，避免“视频讲 A、测验考 B、正文又用另一套命名”。

可移植的是注册表校验器、引用完整性检查和导出 builder 的思路。LLM coherence judge 只能作为候选信号，必须由确定性 schema、引用存在性、目标覆盖与正式质量门兜底；静态 HTML player 不成为新课程真源，所有媒体只是 `CourseBlock / MediaAsset` 派生物。[6][32][37][38]

### 3.7 ChatTutor：把 AI 可视化变成稳定对象，而不是一次性图片

ChatTutor 通过特定代码块驱动 GeoGebra、Mermaid 多页白板与页内 notes，Agent 操作的是具有 page/object ID 的教学对象，而不是只返回截图。[7] 这个设计适合灵知的公式推导、几何、流程、图谱和代码执行场景：AI 老师可以在同一对象上增量添加辅助线、突出节点或回到前一步，学生也能引用具体对象提问。

但 ChatTutor 是 AGPL-3.0，且没有课程生成、正式学习事实与学习者模型；灵知应独立定义安全 schema、对象版本、沙箱、资源限额、静态降级和 CourseBlock 引用协议。会话结束后需要保存的对象必须经正式课程命令或学习记录命令，不允许藏在聊天会话里成为不可追溯副本。

### 3.8 Tutor-GPT：把它当成“画像越多未必越好”的反例

Tutor-GPT 将 Honcho 用户表征、PDF 检索和 Tutor 响应串成个性化对话，说明“先装配用户与资料上下文，再生成回答”是常见方向。[8] 但其用户表征不是正式掌握事实，并依赖 Honcho、Supabase、OpenRouter、Stripe 等外部服务；GPL-3.0 与第二会话/画像风险使其不适合移植。

灵知现有 `AIContextPackage` 的方向更严格：按意图选择当前块、正式任务、必要学习证据和会话摘要；AI 回答、用户接受某种表述或 provider 失败都不能直接改写模型。[30][36] 因此 Tutor-GPT 的价值主要是反向确认：个性化不是收集更多心理描述，而是为当前教学动作提供更少、更可靠、可解释的证据。

**Confidence：High**

**Basis：** 8 个仓库均核验了许可证、目录或关键实现；但未实际部署和调用真实模型，运行质量、安装成功率和生产性能尚未验证。

## 4. 移植到灵知的目标架构

灵知当前已经拥有足够完整的领域骨架：`CourseDocument` 保存有序异构块和稳定修订；资料管线保存资产、解析文档和证据；正式作答与诊断补救形成学习事实；LearnerModel 从事实确定性重算；LearningRuntime 组合当前学习现场；AIContextPackage 只装配最小必要上下文；版本化学科知识库、课程知识映射和课程质量门控制正式发布。[30][31][32][33][34][35][36][37][38]

因此，所有外部能力必须经过下列映射，而不是保留外部实体。

| 外部能力 | 允许进入的灵知真源 | 允许输出 | 禁止形成的新状态 |
| --- | --- | --- | --- |
| OpenTutor 式资料摄取 | MaterialAsset、ParsedDocument、EvidenceUnit [33] | 资料状态、证据目录、CourseBlock 候选 | 第二资料库、第二课程、独立 RAG 用户画像 |
| Lunaris claim verifier | 现有 evidence refs、knowledge relations、course quality [33][37][38] | claim 绑定结果、缺口、冲突、阻断原因 | 第二先修图、第二发布管线 |
| AI Course Builder 多产物 | objective/knowledge ID、CourseBlock refs、MediaAsset [32][37] | slides、讲稿、音频、测验等派生产物 | 静态 player 的独立课程进度 |
| OATutor BKT | PracticeAttempt、LearningEvent、DiagnosticCase [34] | 带模型版本和来源修订的只读知识点掌握概率 | 可写 BKT 状态表、替代正式掌握结论 |
| Bloom 反馈推进 | LearningRecord/Event、LearnerModel、Runtime [35] | 下一步候选或课程修改提案 | prompt memory 直接改课或改画像 |
| ChatTutor 白板 | CourseBlock、MediaAsset、CourseCommand [32] | 版本化 diagram/graph 对象 | 会话内私藏的白板副本 |
| Tutor 个性化 | AIContextPackage、Conversation、Proposal/Receipt [36] | 临时回答、解释、受限动作提案 | 第二 AI 会话、第二用户画像、导师自有下一步 |

接入时应采用“适配器向内、领域命令落地”的形态：外部算法只接受灵知已版本化的数据，返回候选结果；候选先经过 schema、许可、证据和质量门；正式变化只能调用现有领域服务，并留下 expected revision、受影响对象、幂等回执和撤销路径。[30][31][32][33][34][35][36][37][38]

明确禁止三种做法：

1. 为了快速获得功能，直接 fork LearnHouse、OpenTutor 或 Lunaris 并与灵知双写课程。
2. 把 BKT、Honcho 或聊天摘要保存成第二份可写学习者画像。
3. 让 Agent 绕过课程命令、诊断服务或正式评分器直接修改正文、掌握和知识图。

### 4.1 外部能力接入必须满足的六个契约

“选择性移植”如果没有可执行标准，很容易在开发中重新变成复制模块。建议任何外部能力进入灵知前都必须同时满足六个契约：

| 契约 | 必须回答的问题 | 最低验收 |
| --- | --- | --- |
| 身份契约 | 输入输出引用哪个正式对象？ | 使用 `course_id / block_id / knowledge_id / attempt_id / evidence_id` 等稳定标识，不用自然语言标题充当身份。 |
| 修订契约 | 结果基于哪一批课程、证据与模型状态？ | 携带 `source_revision_vector` 或等价修订；过期结果不能静默覆盖当前对象。 |
| 事实契约 | 哪些输入是正式事实，哪些只是模型推断？ | 正式作答、记录、诊断与资料证据分层；LLM 评价、聊天摘要和偏好推断不得冒充事实。 |
| 输出契约 | 外部模块返回正式修改还是候选？ | 默认只返回 typed candidate；正式写入只能调用现有领域命令。 |
| 失败契约 | 服务不可用、输出不合法或许可不满足时怎样继续？ | 当前课程仍可学习；失败有明确状态、日志与降级，不留下半写入对象。 |
| 治理契约 | 如何追踪来源、许可、确认、撤销和效果？ | 记录代码/设计来源、许可证、配置版本、确认人、回执和观察指标。 |

这六个契约把“外部项目”降为能力提供者。即使未来更换模型、检索器、BKT 实现或白板渲染器，课程、学习事实、画像、AI 动作和课程修改的所有权仍留在灵知领域层。

### 4.2 建议先做四个隔离 PoC，而不是立刻进入正式主链

1. **Lunaris 式 claim verifier PoC**：选线性代数一门标准课，只对 30—50 个重要 claim 做证据支持、冲突、缺口和 `CUT` 判定；与现有质量门对比漏检、误报、成本和延迟。PoC 只输出报告，不修改课程。[3][33][37][38]
2. **OATutor BKT 离线回放 PoC**：从已有正式 `PracticeAttempt` 选择技能映射可靠的封闭题，比较不同先验、guess/slip/learn 参数下的排序稳定性，并检查提示后作答、重复题和独立新题如何影响状态。[5][24][34][35]
3. **ChatTutor 稳定对象沙箱 PoC**：先支持 Mermaid/二维图一种对象类型，验证 schema、对象 ID、增量更新、资源限额、静态回退和正文锚点；不直接引入 Agent 自由执行脚本。[7][32]
4. **Mindsmith 式质量问题聚合 PoC**：只读取去标识化、同修订、同锚点的重复求助与独立失败，输出 `CourseQualityIssue` 候选；人工判断是否为内容问题、前置缺口、题目问题或界面问题，不自动改课。[15][30][34][38]

四个 PoC 的共同停止条件是：无法绑定稳定对象、需要建立第二真源、不能在服务失败时保持当前课程可用、或效果信号无法在独立任务中复验。停止不是失败，而是证明该外部机制不适合进入主链。

**Confidence：High**

**Basis：** 映射基于灵知当前产品蓝图、架构文档和八个主链代码文件，数据归属明确；具体接口改动仍需在 OpenSpec 阶段逐项设计。

## 5. 灵知当前项目：已经站在哪里，还缺哪一段

外部产品调研只有落回灵知当前真源才有意义。根据 `product-blueprint.md` 与当前主链实现，灵知不是一张待填的竞品功能表，而是已经形成明确领域边界、但尚未完成最后一段学习反馈闭环的系统。[30][31][32][33][34][35][36][37][38]

### 5.1 用五层边界看灵知：真正的主线不是页面，而是状态如何流动

灵知的长期产品边界可以压缩为：

```text
课程内容
→ 学习痕迹 / 正式学习事实
→ 可重算的学习者模型
→ AI 老师与学习运行时
→ 课程质量反馈与持续迭代
```

当前实现中，这五层已经不是抽象概念：

| 层 | 当前正式对象与职责 | 已形成的优势 | 当前缺口 |
| --- | --- | --- | --- |
| 课程内容 | `CourseDocument + ordered CourseBlock[]`；资料通过 `MaterialAsset / ParsedDocument / EvidenceUnit` 进入生成，质量通过后发布 [30][31][32][33][38] | 课程有稳定块、修订、证据引用和发布门，不再以整节 Markdown 或用户可见版本包为本体。 | 正式学科包主要覆盖线性代数；claim-level 支持/冲突/coverage 与跨产物目标一致性仍可加强。[30][37][38] |
| 学习事实 | `LearningEvent / LearningRecord / PracticeAttempt / DiagnosticCase` 记录行为、作答、记录和诊断 [30][34] | 正式题、提示、评分、补救和历史事实有独立对象，不让临时聊天承担证据责任。 | 需要进一步统一知识 ID 绑定、区分 scaffolded 与 independent evidence，并建立跨修订聚合规则。 |
| 学习者模型 | `learner_model_v1` 从同批正式证据确定性重算 [30][35] | 模型是事实的解释，不是第二份不可审计画像；AI 回答和 provider 失败不能直接改写。 | 尚未形成经校准的知识点级概率模型、遗忘/保持参数和跨学科通用映射；不能提前显示伪精确掌握值。 |
| AI 老师与运行时 | `LearningRuntime` 装配当前现场，`AIContextPackage` 按意图最小化选择上下文，AI 只回答或提出受限动作 [30][35][36] | 课程、正式任务、学习证据和会话不再由前端整包拼接；AI 不拥有第二学习状态和竞争“下一步”。 | 需要让更多前台交互真正消费同一知识坐标，验证建议曝光、采用、完成与效果，而不只验证 API 返回正确。 |
| 课程迭代 | 当前生成质量门存在，目标链为 `CourseQualityIssue → Proposal → CourseCommand → Receipt → 效果复验` [30][38] | 已有领域命令、修订保护和质量发布基础，可承接受控改课。 | 学习反馈聚合层尚未正式形成，是当前最明确的系统断点；同一位置反复困惑还不能稳定转成可确认改进并比较修改前后效果。[30] |

这说明灵知的最大资产并不是“已经有多少按钮”，而是五层之间的所有权基本清楚：课程只认一个当前 `CourseDocument`，事实由各领域对象保存，画像从事实派生，AI 临时装配上下文，正式修改由领域命令完成。外部能力必须进入这条链，而不能在旁边再长出一套状态。

### 5.2 用四层分析法看项目：实体层强，逻辑层正在闭环，AI 层必须继续受约束

| 分析层 | 灵知当前状态 | 与市场相比 |
| --- | --- | --- |
| 实体层 | 课程、块、资料、证据、题目、尝试、事件、记录、模型、运行时、AI 提案和学科知识均有明确对象。[30][32][33][34][35][36][37] | 比多数闭源公开描述更强调真源和修订；但对象丰富不等于前台体验已成熟，也不等于学习效果已验证。 |
| 功能层 | 产品蓝图与当前实现记录已覆盖目标/资料成课、正文学习、块级 AI、正式练习、诊断补救、学习记录、知识视图与恢复。[30][32][33][34][35][36][37][38] | 与 A-Learn/Parlina 等相比，内核边界更深；与 Coursebox/Sana/Mindsmith 相比，商业化生产体验、多模态完成度和分析工作台仍有差距。[9][11][14][15][17][18] |
| 逻辑层 | 理想主链是“目标资料→质量发布→学习现场→正式事实→模型→运行时/AI→提案/命令→课程改进”。前半段和中段已纵向收束，最后的课程反馈回流尚未完成。[30] | 这是最有潜力形成差异的层，但目前仍是“架构领先、结果未闭环”，不能对外宣称已经完成自进化课程。 |
| AI 能力层 | AI 负责生成候选、解释、提示、局部补充和动作提案；不能绕过事实、评分、诊断和命令直接写正式状态。[30][36] | 约束比“全自动 Agent”更克制，也更适合教育；代价是需要把确认、降级和失败恢复做得足够顺滑，否则用户只会感觉流程慢。 |

### 5.3 当前最强的五个基础，不应在融合中被破坏

1. **一个课程真源**：`CourseDocument / CourseBlock` 同时承载顺序、教学角色、媒体引用、证据引用和稳定锚点，外部 player、LMS 或白板不能再保存另一份课程正文。[30][32]
2. **正式事实与推断分离**：作答、记录、诊断是事实；掌握、偏好和下一步是可重算解释；AI 会话只是上下文，不是证据账本。[30][34][35][36]
3. **LearningRuntime 是无存储协调投影**：它把同一批课程、尝试、记录、模型和任务组成当前学习现场，而不成为第七套状态库。[30][35]
4. **AI 只提出、领域命令落地**：回答可以即时，正式课程变化、任务创建和状态变化必须经 typed proposal、确认、修订检查和回执。[30][36]
5. **AI 失效时课程仍可继续学习**：课程与正式任务不能依赖某个 provider 实时在线；生成、Tutor、白板和检索失败要有明确降级。[30][31][38]

这些基础看起来不如“30 种互动块”显眼，却决定灵知能否把多个开源能力组合起来而不崩成拼装系统。它们也是未来更换模型供应商、加入教师端或扩展新学科时最重要的可持续资产。

### 5.4 当前真正的短板：不是少一个功能，而是四个断点

**断点一：生成质量还缺 claim-level 证据语义。** 现有资料覆盖与发布质量门已经存在，但要达到 Lunaris 式可信生成，还需要明确哪些重要主张由哪些证据支持、哪里冲突、哪里只能降级或删除，并将这些结果绑定到 `CourseBlock` 和 objective/knowledge ID。[3][33][37][38]

**断点二：统一知识坐标尚未跨学科成熟。** 当前正式学科包主要覆盖线性代数，课程目标、正文、题目、错因、补救和 AI 上下文共享知识 ID 的方法需要先在一个学科做深，再复制到其他学科。[30][37] 过早引入一个通用 AI 生成知识图，只会用漂亮节点掩盖语义不稳定。

**断点三：学习者模型有正确边界，但还缺校准深度。** “从事实重算”解决了可信来源问题，却没有自动解决技能映射、先验、遗忘、提示折扣、跨技能题和开放任务评价。OATutor 的 BKT 可作为封闭技能的局部工具，不应成为全域模型。[5][24][35]

**断点四：课程质量反馈尚未闭合。** 当前系统可以生成课、学习、作答、建模和辅导，却还不能把同一锚点的反复困惑、错误和求助稳定聚合成课程问题，确认修改后也没有正式的前后效果窗口。[30] 这正是灵知与所有“一句话成课”产品拉开差异的关键一段。

### 5.5 必须诚实区分“架构领先”和“产品领先”

灵知目前可以合理声称：在内部设计上，已经明确课程、学习事实、学习者模型、AI 老师和课程修改之间的边界；这比许多公开产品说明更完整。[30][32][34][35][36] 但不能由此声称：生成质量已经优于 Coursebox/Sana、学习体验已经优于 A-Learn/Parlina、个性化效果已经优于 ALEKS/Khanmigo，或课程已经会根据群体反馈自动进化。前者需要同题生成盲测、真实产品走查和用户转化数据；后者需要近迁移、延迟保持、实施忠实度与受控改课实验。[9][11][17][18][23][28]

这份报告因此把灵知定位为：**拥有较强系统骨架、正在从“能力存在”跨向“闭环成立”的产品**。下一阶段的胜负不在继续扩大对象数量，而在把少数关键链路做成用户能感知、数据能复验、失败能恢复的完整体验。

**Confidence：High（项目边界与当前蓝图）/ Medium（市场相对领先程度）**

**Basis：** 项目状态以当前产品蓝图和主链实现为准；市场内部状态不可见，且本轮没有同题实测，因此相对判断保留边界。

## 6. 证据冲突与效果验证门

### 6.1 结构化学习事实优于原始聊天堆积

Khan Academy 在 2025-10 至 2026-04 的平台实验中发现：提供近期作答摘要，在 608,000 个 tutoring threads 中使同技能下一题正确率提高 3.4%；暴露未掌握先修技能并先做短复习，在 136 万个 threads 中提高 2.7%。原文没有说明这两个百分比是相对变化还是百分点变化。二者来自两个独立改动，不能把“合计 6.1%”误写成联合处理的单次效应。完整原始会话日志本身没有改善下一题正确率；调整格式并加入过去 24 小时同技能会话后，改善的是认知参与度，而不是已证明的长期迁移。[23]

这直接支持灵知当前方向：AI 老师应优先读取结构化的近期作答、正式前置缺口和当前课程位置，而不是扩大聊天记忆。[35][36]

### 6.2 AI 提示“可能有效”与“原始质量高风险”同时成立

OATutor 的随机实验只把经过人工质量筛选的 ChatGPT worked-solution 提示交给学习者。该组平均前后测差为 17.01 个百分点，无完整 worked-solution 提示的对照组为 1.85 个百分点，组间 `p=0.011`；ChatGPT 与人工提示组差异不显著，但研究并未做等效性检验。研究只有三道相同的前后测题，394 名众包成人中最终分析 274 人，约 30% 脱落，不能直接外推真实课堂。更关键的是，同一研究的离线质量评估中，原始单次 ChatGPT 提示有 32% 未通过答案正确、过程正确和语言适当性检查。[24]

所以结论不是“AI 提示有效，可以直接上线”，也不是“AI 会犯错，不能使用”，而是：生成式提示只适合在领域化离线集、正确性校验、答案泄露检测、失败降级和独立复验同时成立时逐步灰度。

### 6.3 自适应模型存在，不代表个性化真实发生

ALEKS 用 Knowledge Space Theory 表示可行知识状态，厂商将 Knowledge Checks 描述为用于确认和增强保持，这是成熟的可解释架构描述。[27] 但独立实施研究对 40 个 Algebra I 班级的观察显示：实际使用 ALEKS 的 31 个班中，29 个（约 94%）没有达到高个性化；按“推荐使用量和高个性化”同时满足的更严格标准，只有一个班达标。[28]

这意味着灵知不能只验收“LearnerModel 返回了 mastery probability”或“Runtime 生成了推荐”。还必须记录推荐是否真正曝光、用户是否采用、流程是否完成、独立任务是否改善。

### 6.4 知识图谱与知识追踪需要分层验证

CogGen 展示了“知识组件—BKT 状态—教学动作”的可控架构，但现有结果主要是知识抽取和专家评估，不是学生学习增益。[25] KG-RAG 在 76 人金融学习实验中显示即时后测优势，但缺少前测、长期保持和完整代码数据。[26] 2026 年医学教育研究给出结课和一个月保持的积极信号，但只有单校 6 个班整群随机，分析却按学生级 ANOVA 处理且未做聚类校正，又只分析 362 人中的 301 名完成者；因此只能把它视为需要独立复现的低至中等强度证据，不能据此拆分归因于 LLM、知识图或某个单一功能。[29]

灵知的效果门应分四层：

| 层级 | 需要证明什么 | 指标示例 |
| --- | --- | --- |
| 内容安全 | AI 产物可进入学习现场 | 事实/数学正确率、引用完整性、答案泄露、schema、延迟、降级 |
| 近迁移 | 辅导后能独立完成相近任务 | 下一题、同技能新题、零帮助通过率 |
| 延迟与跨情境 | 学习没有只停留在当前会话 | 7/30 天保持、跨技能题、真实任务迁移 |
| 实施忠实度 | 设计的个性化链真的被使用 | 曝光率、采用率、完成率、诊断/复验闭环率 |

**Confidence：Medium-High**

**Basis：** Khanmigo 和 OATutor 提供大规模平台实验或同行评审证据，知识图谱试验有积极信号；长期、跨学科和独立复制仍不足。

## 7. 灵知与市面产品的直接对比

### 7.1 与 Coursebox、Sana、Mindsmith 比：灵知内核更克制，产品化仍需追赶

| 维度 | 市面强项 | 灵知当前状态 | 结合点与取舍 |
| --- | --- | --- | --- |
| 课程创建 | Coursebox/Sana 已将多来源导入、候选大纲、可编辑内容和发布做成成熟工作流。[9][11] | 资料证据、生成工作区、质量门与 `CourseDocument` 内核更明确，但用户侧对解析、候选、预览和发布状态的感知仍可简化。[31][33][38] | 学习它们的前台流程，把灵知证据覆盖、阻断原因和恢复状态放在流程背后；不把用户变成块编辑器操作员。 |
| 多模态与互动 | Mindsmith 有 30 多种互动单元，Coursebox 支持视频、测验和图片。[9][14] | `CourseBlock.kind + role` 为多形态教学准备了更稳的内核，但形态数量和成品感未必领先。[30][32] | 优先补“教学目标→合适形态→可访问性→静态降级”，不做媒体数量竞赛。 |
| Tutor | Sana 对课程、目标、位置、进度的公开说明最完整；Coursebox 强调资料约束。[10][12] | `LearningRuntime + AIContextPackage` 的事实与上下文边界更明确，且禁止 Tutor 自有画像。[35][36] | 前台借 Sana 的选区解释、问题建议和课内 Tutor 交互；后台坚持最小证据和答案隔离。 |
| Analytics 与改课 | Mindsmith 可定位掉点、逐题问题并让 Agent 改内容。[15] | 灵知有修订、领域命令和正式学习事实，但 `CourseQualityIssue` 聚合尚未形成。[30][38] | 借分析入口，增加证据独立性、确认、撤销、历史冻结和效果窗口；这是最有价值的结合点。 |
| 企业交付 | 三者更成熟地覆盖作者、团队、发布、导出和管理。 | 灵知当前核心更偏个人持续学习，企业组织、权限、证书和 SCORM 不是主线。[30] | 短期不正面竞争完整 LMS；未来若做机构版，复用同一课程与事实真源，外加权限投影，不重建领域模型。 |

这组比较说明：灵知并不应该用“我们的架构更好”掩盖可见产品体验的差距。用户第一次进入时只会感知起课速度、首屏质量、是否能马上学、AI 是否理解当前位置、失败后能否继续。架构的价值是保证这些体验可以持续演进，而不是代替它们。

### 7.2 与 A-Learn、Parlina、Oboe、StudyFlow 比：灵知需要把复杂内核折叠成更轻的个人体验

这四个产品都在争夺“我现在想学一个东西，马上给我一条可以走的路”。A-Learn 强在节点图、断点、笔记与尝试历史；Parlina 强在课程地图、开放题反馈与语音 Tutor；Oboe 强在目标倒推和自动媒介选择；StudyFlow 强在资料到备考计划与复习循环。[17][18][21][22]

灵知可以把这些表面机制连接到更正式的底层：

| 用户看到的体验 | 市面参照 | 灵知应连接的真实对象 | 不能发生的退化 |
| --- | --- | --- | --- |
| 一句话后出现课程路线 | A-Learn、Parlina、Oboe | `CourseDocument` 目录 + `CourseKnowledgeMap` 投影 [30][32][37] | AI 自造一张与正式知识库无关的第二地图。 |
| 返回后从原位置继续 | A-Learn | `LearningRuntime`、稳定 block anchor、正式快照 [30][35] | 只在浏览器存一个 scrollTop，跨设备或修订后失真。 |
| 选区提问与局部解释 | A-Learn、Parlina | 当前 `CourseBlock` + 最小 `AIContextPackage` [32][36] | 把整课、全部笔记和全部聊天默认发送给模型。 |
| 系统自动选择图、音频、代码或题目 | Oboe | `CourseBlock.kind/role` + objective/knowledge ID + MediaAsset [21][32][37] | 用户手工拼装媒体；多媒体与课程目标漂移。 |
| 今日复习与考试计划 | StudyFlow | 正式作答、review task/checkpoint、知识状态和时间约束 [22][34][35] | 把固定 60/40 比例当成所有主题的通用算法。 |

灵知相对这组产品的机会在于：它不仅能显示“进度 60%”，还可以解释这个进度由哪些正式行为构成、哪些知识点只是接触过、哪些经过独立复验、为什么今天建议某个动作。它的风险也在这里：如果界面把这些证据、修订和置信度全部暴露给用户，体验会比竞品更重。正确做法是默认给一句清晰解释，例如“你在两个不带提示的新题中仍混淆了特征值与特征向量，所以先做一个 5 分钟补救”，详细证据只在需要时展开。

### 7.3 与 OATutor、ALEKS、Khanmigo 比：灵知有更完整的课程现场，但效果方法还需补课

| 维度 | OATutor / ALEKS / Khanmigo 已提供的参照 | 灵知的机会 | 灵知当前不能声称的事 |
| --- | --- | --- | --- |
| 技能与题目映射 | OATutor 把题目、技能、BKT 参数和提示结构化；ALEKS 用知识空间表示可学边界。[5][27] | 将课程块、正式题、错因、补救和知识 ID 贯通，不限于题库页面。[34][37] | 当前模型已经准确预测所有主题的掌握。 |
| Tutor 上下文 | Khanmigo 证明结构化近期作答和前置缺口比原始聊天更有用。[23] | 让 AI 老师同时理解课程现场、正式任务、学习事实与知识前置关系。[35][36][37] | 更多上下文必然带来更好学习效果。 |

| 课程演进 | 三者主要优化个人路径或 Tutor，公开证据未形成受控群体反馈改课链。 | 用跨学习者重复证据改进同一个课程块，并冻结历史事实。 | 课程已经可以“自动自我进化”。 |

灵知相对传统智能辅导系统的结构性优势，是课程并非只有题目序列：解释、例子、图示、代码、正式任务、个人记录与临时补救可以共享稳定锚点。它因此有机会把“为什么错—在哪里补—补什么形态—如何复验—是否应改原课”放在同一现场。但这只是机会；OATutor 与 Khanmigo 的价值恰恰是提醒灵知，教育效果必须用独立任务和长期数据证明，而不是由架构复杂度推导。[23][24]

## 8. 战略推演：从“一句话生成”走向持续学习基础设施

本节是基于前述证据的产品与架构推演，不是外部市场已经验证的事实。它的用途是帮助灵知选择更有长期复利的方向，并给后续用户研究和实验提出可证伪假设。

### 8.1 判断一：一句话生成是入口，不是产品本体

一句话的真正价值不是“少填几个表单”，而是把用户模糊意图转换为一个可以开始学习的承诺。这个承诺至少包含：我要解决什么问题、起点是什么、用哪些资料、先学什么、学到什么算完成。Coursebox、Sana、A-Learn、Parlina、UCHU 和 Oboe 已证明低门槛起课是普遍方向。[9][11][17][18][20][21]

灵知应保留一句话入口，但让系统在后台完成三个动作：解释意图、识别缺失约束、生成可审查蓝图。对低风险主题可直接给首课与默认路线；对高风险、强资料依赖或目标含糊的主题，最多追加一个真正改变课程结构的问题，而不是把一句话入口重新变成长表单。生成的结果不是“完成品”，而是进入持续学习循环的第一个稳定 `CourseDocument`。

### 8.2 判断二：课程应被理解为“可验证的编译产物”

传统在线课程像出版物，聊天 Tutor 像无限续写。灵知更适合第三种形态：**课程是由目标、资料证据、知识坐标和教学规则编译出的稳定产物；个人运行时可以在上面叠加临时解释与补救，群体证据可以触发受控重新编译，但正式课程不会因一次对话不断漂移。**

这个比喻带来明确分工：

```text
目标 / 资料 / 学科知识 / 教学标准
→ 课程编译与质量门
→ 稳定 CourseDocument / CourseBlock
→ LearningRuntime 叠加个人临时层
→ 正式学习事实与效果数据
→ 触发局部重新编译，而不是自由重写
```

Lunaris 的 typed draft、claim verification 和 coverage gate，[3] AI Course Builder 的 LO registry，[6] 以及灵知当前工作区、质量门和领域命令，[31][32][38] 都支持这一方向。它能同时避免两种极端：课程永远固定、不能适应；课程每次打开都被 AI 重写、无法复验。

### 8.3 判断三：正式学习事实是 AI 教育产品的内部 API

在普通 AI 产品里，memory 常被理解为“模型记住用户说过什么”；在教育产品里，更重要的是系统能否回答：用户完成了什么、是否用了提示、错误属于哪个知识点、补救后是否能独立完成、这个结论基于哪次修订。Khanmigo 的结果显示，结构化近期作答和先修缺口比原始会话日志更有用；OATutor 的骨架也建立在技能化作答之上。[23][5]

因此，灵知的正式学习事实不仅服务画像，也应成为所有上层能力的内部 API：复习调度、AI 老师、诊断、课程质量、教师视图、实验分析都通过同一事实协议读取数据。这样新增模型或功能时，不需要重新发明“用户状态”。这比保存越来越长的聊天记忆更可能形成数据复利。

### 8.4 判断四：AI 老师应该是“状态读取者和动作提出者”，不是状态拥有者

一个强 AI 老师需要理解当前块、学习目标、最近错误、提示历史和未解决问题，但不需要拥有一份私有画像。私有状态会造成三个问题：无法解释为什么得出结论、provider 或 prompt 改变后无法重算、与正式题目和课程修改发生冲突。

灵知应继续坚持：AI 临时读取 `AIContextPackage`，输出回答、解释或 typed proposal；正式状态由评分、诊断、记录、模型服务和课程命令更新。[35][36] 为了不让受约束 AI 显得笨重，产品层可自动完成低风险动作，例如打开相关正文、生成仅本次可见的反例或把用户带回一道未完成题；只有改变正式课程、创建长期任务或写入用户资产时才要求明确确认。

### 8.5 判断五：`CourseQualityIssue` 应成为课程的“结果模型”

学习者模型回答“这个人现在可能会什么”；课程质量模型应回答“这个课程位置对哪些人、在什么条件下、重复制造了什么问题”。Mindsmith 已经展示 analytics 到改课的商业价值，[15] 但灵知可以把它做得更严谨：

- 问题必须绑定稳定课程块、修订、知识点和证据类型；
- 区分内容错误、证据不足、先修断裂、例子不匹配、题目歧义、媒体失效和界面摩擦；
- 单个学习者的一次困惑默认只触发个人帮助，不直接升级为课程问题；
- 多个独立证据达到阈值后，AI 生成影响范围和最小修改候选；
- 修改后冻结历史事实，比较新修订下的独立通过率、求助率、停留和延迟保持；
- 结果无改善或护栏退化时撤销或重新设计。

如果这条链建立起来，灵知的课程资产会随真实使用积累“为什么这样教”的证据，而不是只积累更多 AI 生成文本。这比单纯模板库更难复制。

### 8.6 判断六：个性化的瓶颈不是算法，而是实施忠实度

ALEKS 的冲突证据非常关键：知识空间与 Knowledge Checks 可以存在，但真实课堂仍可能因为使用量不足、课程覆盖压力和教师流程不匹配而没有发生高个性化。[27][28] 对灵知而言，即使 `LearningRuntime` 生成了正确候选，只要用户看不到、不理解、不采用、没有完成复验，个性化仍然没有发生。

所以每个自适应功能都要记录一条实施漏斗：

```text
符合触发条件
→ 实际生成
→ 用户实际看到
→ 用户理解建议原因
→ 接受并开始
→ 完成补救
→ 完成独立复验
→ 后续保持或迁移改善
```

这条漏斗比“推荐算法命中率”更接近产品真实效果，也会反向要求界面保持单一主线、避免多个 AI 入口竞争注意力。

### 8.7 产品体验推演：用户只看见一条学习流，系统在背后维护五层状态

建议把完整体验收束为四个循环。

**起课循环**：用户输入一句目标，可选附资料；系统立即给出理解结果、候选路线和首个可学习单元。只有真正影响课程的缺口才追问。用户确认的是目标、深度、顺序和资料用途，不是内部块结构。资料证据不足时，系统明确标记“基于通用知识生成”或阻断高风险主张。[9][11][3]

**日常学习循环**：打开课程回到原正文锚点；稳定 `CourseBlock` 组成主流，临时 `AdaptiveBlock` 只在用户主动请求或强证据触发时出现，笔记/提问摘要作为正式记录投影贴近原文。AI 选区解释、例子、图示和局部测验都围绕当前内容，不要求用户切换到独立聊天产品。[30][32][35][36]

**诊断补救循环**：正式任务失败后，系统先区分知识缺口、误区、题目问题和偶然错误；补救只覆盖最小缺口，提示逐层展开；复验使用没有泄题的新题或新情境，成功后才形成较强掌握证据。[5][24][34]

**课程进化循环**：个人困惑先服务个人；当同一修订、同一锚点出现重复独立证据时，形成质量问题候选。作者或有权限用户看到“问题—证据—影响—修改—风险—观察指标”，确认后局部修改；旧作答不被改写，效果按新修订重新观察。[15][30][38]

### 8.8 市场切入口推演：先服务“带目标与资料的深度自学”，再扩展

从当前能力与竞品格局看，最匹配灵知的首个切入口不是泛化“任何人学任何东西”，也不是立刻成为企业 LMS，而是：**用户有一个具体目标，通常还带着教材、论文、文档、课程资料或考试范围，希望在数周内形成可验证掌握。** 这一人群同时需要资料保真、课程结构、连续学习、正式练习、AI 辅导和复习，能够感知灵知完整链路的价值。

可进一步分为两个场景假设：

1. **资料驱动的复杂主题自学**：例如一本教材、一个研究方向、一套技术文档。差异点是来源可追溯、知识路线、正文内 Tutor 和长期学习记录。
2. **有明确终点的能力/考试冲刺**：差异点是目标倒推、定位测评、正式作答、错因补救、复习与延迟验证；但需要更强题库质量与知识映射，因此应晚于第一个场景或只在线性代数等已具正式学科包的范围试点。

企业培训、教师创作、证书和 SCORM 未来可以成为分发与付费层，但不应现在反向重塑核心领域模型。更可持续的路线是先证明个人学习闭环，再把同一课程质量与事实协议暴露给教师/组织，而不是先复制 LearnHouse 或 Coursebox 的全部后台。[4][9]

### 8.9 护城河推演：不是模型，也不是内容数量，而是五种可积累资产

| 可积累资产 | 为什么有复利 |
| --- | --- |
| 结构化课程与稳定锚点 | 每次提问、作答、记录和质量问题都能回到同一个教学位置。 |
| 资料证据与知识坐标 | 课程、题目、错因、补救和多媒体可以共享语义，而不是每次重新 prompt。 |
| 正式学习事实协议 | 新 Tutor、新模型、新复习器和教师视图都能复用同一可信输入。 |
| 受控动作与效果回执 | 系统知道“建议了什么、用户是否执行、是否改善”，而不是只有聊天满意度。 |
| 课程质量问题与修订结果 | 真实学习不断提升课程资产，并保留为什么改、改后是否有效。 |

基础模型会让生成文本、音频、视频和图表越来越便宜；上述资产却依赖长期产品使用、稳定对象和治理协议。它们才是灵知应优先建设的长期复利层。

**Confidence：Medium**

**Basis：** 推演与已核验市场机制、效果证据和灵知当前边界一致，但切入口、付费意愿、用户理解成本与数据复利尚未通过访谈、可用性测试和真实留存验证。

## 9. 三阶段融合路线

### 9.1 阶段一：质量门与统一知识坐标（0—6 周）

目标不是立即增加新 UI，而是把外部项目最有价值的确定性约束收进现有主链：

- 参考 Lunaris，为重要正文 claim 增加证据绑定、冲突、覆盖和缺口状态；参考 AI Course Builder，让课件、讲稿、音频、题目共享稳定 objective/knowledge ID。[3][6]
- 补齐 CourseBlock、PracticeTask、MasteryCriterion、misconception、remediation 与 `knowledge_id/concept_ids` 的正式绑定。[32][34][37]
- 建立最小 `CourseQualityIssue`：只有重复、独立、版本一致的学习证据才能形成问题；AI 只能提出候选，经确认后用 CourseCommand 增量修改，再进入观察窗。[15][30][38]

验收门槛：不存在的 objective/knowledge/evidence ref 必须阻断发布；先修环、必用证据缺失和跨产物漂移进入同一质量报告；失败候选不得改变当前课程；AGPL/GPL 复制代码为零；MIT 移植保留原版权与许可声明并记录来源；现有课程生成与学习主链测试不退化。

### 9.2 阶段二：事实驱动知识追踪与教学动作（6—12 周）

在统一知识 ID 稳定后，才试验 OATutor BKT：

- 输入只来自正式 PracticeAttempt、LearningEvent 和 DiagnosticCase；输出只是 LearnerModel 中带 `source_revision_vector` 的知识点级派生投影。[5][34][35]
- 低掌握点只能成为 LearningRuntime 候选动作，不能绕过课程顺序、诊断确认和独立复验。[30][35]
- AI 老师读取 revisioned context，继续禁止直接修改掌握、画像、诊断和课程正文。[36]
- 同一事实批次全量重放必须得到相同模型修订和 Runtime 动作；scaffolded 作答不能计为独立掌握。

灰度必须同时观察近迁移、延迟、答案泄露和事实/数学错误。未达到预设胜出概率或任一护栏退化，就不扩大流量。[23][24]

### 9.3 阶段三：可操作可视化与课程质量回流（3—5 个月）

- 参考 ChatTutor，将白板建模为有稳定 page/object ID 的 `diagram/graph_embed` 块或 MediaAsset，脚本在沙箱运行，失败时回到静态表示。[7][32]
- 参考 Mindsmith，将跨学习者、跨时间的重复独立证据聚合为课程质量问题，但保留证据阈值、权限、冷却、确认、撤销和版本影响范围。[15][30][38]
- 参考 LearnHouse 的 plan/finalize 节奏，让结构级改课先停留在计划和候选，不引入其 LMS 实体。[4]
- 修改前后比较同技能独立题、延迟保持、求助率和同一课程位置的重复困惑；旧课程在改写失败时持续可用。

### 9.4 建议拆成五个可独立验收的交付包

| 交付包 | 核心产物 | 进入条件 | 通过标准 | 停止或回退条件 |
| --- | --- | --- | --- | --- |
| A. 证据化课程质量 | claim/evidence 关系、支持/冲突/缺口状态、coverage report | 现有资料解析与证据 ID 稳定 [33] | 标准课的重要 claim 可追溯；缺证据和冲突不被静默发布；旧生成链无回归 | 误报导致大部分课程无法发布，或只能靠第二套知识图工作 |
| B. 统一目标与知识 ID | objective registry、块/题/错因/补救/媒体引用校验 | 正式学科包与课程映射可用 [37] | 任一引用可定位；跨产物漂移被阻断；不存在 ID 不进入正式课 | 依赖 AI 临时生成 ID，或跨修订无法迁移 |
| C. 课程质量问题最小闭环 | `CourseQualityIssue`、人工分类、候选修改、确认、回执、观察窗 | 稳定锚点和正式学习事实已可聚合 [30][34] | 单次困惑不改课；重复独立证据可形成可解释候选；修改可撤销且历史事实冻结 | 证据无法区分内容、题目、前置与界面问题，或候选会绕过 CourseCommand |
| D. 知识追踪试验 | BKT 无状态函数、参数版本、离线回放、只读模型投影 | B 包已稳定，技能映射质量达到阈值 | 同批事实重放结果一致；提示折扣、独立复验和过期修订处理正确；校准优于简单基线 | 不能优于“最近独立正确率”等简单基线，或开放题映射制造伪精确度 |
| E. 教学对象与效果平台 | 稳定可视对象、离线质量集、线上实验与实施漏斗 | A—D 的对象和指标协议可复用 | 白板可降级；AI 动作同时监测质量、近迁移、保持与采用；护栏退化自动停止 | 视觉能力只能依赖不受控代码执行，或实验无法区分功能暴露与实际使用 |

执行顺序上，A/B/C 是产品差异化主线，D/E 是在主线稳定后增加的教学算法与表现力。尤其不建议把 BKT 或可视白板当成“先做一个很酷的功能”：没有统一 ID 和效果门，它们只会增加不可解释状态与内容风险。

### 9.5 每阶段都要保留三种对照

- **产品对照**：同一用户目标下，与当前灵知基线比较起课时间、首课可用率、继续学习率和任务完成率，而不是只看新功能是否出现。
- **教学对照**：与简单、可解释基线比较。例如 BKT 必须优于近期独立正确率；个性复习必须优于固定到期规则；AI 改课必须优于不改或人工小改。
- **系统对照**：AI provider 不可用、检索失败、白板执行失败和候选被拒绝时，当前课程、正式任务、断点和历史事实仍然可用。

进入正式实施前，每个交付包都应单独建立 OpenSpec，写清 proposal、design、tasks、spec 与验证样本；本报告只决定方向和边界，不代替具体变更规格。

**Confidence：High（路线顺序）/ Medium（周期估计）**

**Basis：** 顺序由许可证、真源依赖和当前缺口共同决定；周期尚未经过具体 OpenSpec 拆分和工程估时。

## 10. 产品定位与决策清单

### 10.1 建议定位

不建议继续使用：

> 一句话生成任何课程。

这一表达已经被 Coursebox、A-Learn、UCHU、Oboe 等大量产品占据，而且没有说明“生成以后为什么能学会”。[9][17][20][21]

建议正式表达：

> **灵知是一套以可追溯课程块为学习现场、以正式学习事实驱动可解释辅导，并把重复学习问题转成可确认课程改进的个人 AI 学习系统。**

面向用户可以进一步压缩为：

> **一句话开启一门课，真实学习证据持续帮助你学会。**

### 10.2 现在做、以后做、明确不做

| 决策 | 内容 |
| --- | --- |
| 现在做 | claim-level 质量门、统一知识 ID、CourseQualityIssue 最小闭环、AI 回答/提示离线质量集、近迁移与泄题护栏 |
| 条件成熟后做 | BKT 只读投影、间隔复习参数校准、稳定对象白板、多产物一致性构建器、跨学习者课程问题聚合 |
| 暂不竞争 | 模板数量、证书、排行榜、XP、播客数量、全自动 Agent 改课、企业 LMS 组织权限 |
| 明确禁止 | 第二课程库、第二学习者画像、第二 AI 会话、第二知识图谱；聊天直接改掌握；临时 AI 题改变正式证据；AGPL/GPL 代码未经审查并入主产品 |

这一路线不是放弃“一句话生成”，而是把它降为入口，把真正差异放在生成后的持续学习和受控演化上。

### 10.3 面向不同对象的三层表达

| 对象 | 建议表达 | 目的 |
| --- | --- | --- |
| 首次访问者 | **一句话开启一门课，真实学习证据持续帮助你学会。** | 先让用户理解“快速开始 + 持续变懂”，不展示内部术语。 |
| 深度学习者 | **从你的目标和资料生成可追溯课程；系统根据真实作答、问题和复验调整讲法与复习。** | 解释为何比通用聊天和普通生成课程更可靠。 |
| 合作者/投资人/团队 | **以稳定课程块、正式学习事实和受控 AI 动作为内核，把个人辅导与群体课程改进连接成可验证闭环。** | 说明系统壁垒、数据复利和治理边界。 |

不要在对外表达里先讲 `CourseDocument`、BKT、revision vector 或领域命令；这些是保证承诺成立的内部机制。也不要使用“自动掌握”“保证学会”“无限自进化”等超出证据的词。更可信的承诺是：来源可看、建议有因、正式变化可控、学会要靠独立任务验证。

**Confidence：High**

**Basis：** 市场能力和灵知当前模块缺口方向一致；定位仍需后续通过真实用户访谈和转化实验验证表达是否清晰。

## 11. 风险、未知与后续调研

- 本轮深读了开源仓库，但没有逐一部署、调用真实模型或生成同题课程；安装成功率、延迟、输出质量、资源成本和端到端 UX 尚未验证。
- 闭源产品依据官方页面和帮助中心，能证明公开功能和产品机制，不能证明内部数据结构、默认授权范围或真实学习效果。
- GitHub 提交时间和版本是 2026-07-14 快照，不代表真实用户数或生产成熟度。
- 许可证判断只覆盖仓库顶层许可证和 README 明示内容，没有完成全部依赖、模型权重、素材、字体、数据集和 API 条款审计，也不构成法律意见。
- BKT 对独立技能和封闭题型较合适，对开放题、跨技能任务和错误技能映射可能产生虚假精确度，必须用灵知真实历史数据回放校准。
- 现有效果证据集中在数学、技能题或单一学科；不能直接外推所有主题、开放课程和长期迁移。
- 报告给出的周期是产品优先级，不是正式工程估时；进入实施前仍需 OpenSpec、代码审计、许可清单和验收样本。

### 11.1 关键风险登记表

| 风险 | 早期信号 | 缓解方式 |
| --- | --- | --- |
| 过度工程化 | 用户起课慢、需要理解大量内部概念、确认弹窗过多 | 保持一句话入口和单一学习流；只对正式资产与高风险动作确认，低风险临时帮助自动完成。 |
| 知识坐标成本失控 | 每扩一个学科都需大量人工建树，AI 自动图又不稳定 | 先在线性代数做深；定义最小学科包、候选审核和版本发布；没有正式包时明确降级。 |
| 课程质量信号误判 | 停留、退出或单次错误频繁触发错误改课 | 使用稳定锚点、同修订、多种独立证据、人工分类和冷却；先提案不自动执行。 |
| BKT 伪精确 | 概率剧烈波动、与独立新题表现不一致、跨技能题难映射 | 只在封闭技能试验；参数版本化；与简单基线比较；界面优先显示解释性状态而非小数概率。 |
| AI 教学内容伤害 | 错解、泄题、引用错误或过度帮助进入正式学习 | 离线质量集、自一致性/规则校验、答案隔离、失败降级、抽检和近迁移复验。[23][24] |
| 开源许可污染 | AGPL/GPL 代码或 CC BY 内容混入主仓而无来源记录 | 默认 clean-room 式独立实现；MIT/内容分别登记；进入主链前做依赖与资产清单，必要时法律审查。 |
| 双真源复发 | 外部模块开始保存课程、画像、会话或图谱副本 | 六个接入契约、架构测试和领域命令门禁；任何新存储先证明现有真源无法承载。 |
| “能力存在但没人使用” | Runtime 有建议但曝光/采用/完成率低 | 为每个个性化动作记录实施漏斗，结合真实任务流程迭代，而不是只调算法。[28] |
| 效果叙事过度 | 用点赞、完成率、短期后测或厂商数据宣称长期学习 | 分开报告内容质量、近迁移、延迟保持、跨情境迁移和实施忠实度。 |

### 11.2 下一轮最值得做的实证调研

1. **真实账号走查**：用同一份资料和同一学习目标，在 Coursebox、Sana、Mindsmith、A-Learn/Parlina 中完成“创建—首课—提问—作答—恢复—修改”流程，记录生成时间、失败、引用、状态变化与实际付费边界。
2. **同题生成盲测**：让灵知、Lunaris、Coursebox/Sana 等对同一材料生成课程，以专家盲评目标覆盖、先修顺序、claim 支持、题目一致性和可学性，而不是比较页面截图。
3. **用户价值验证**：分别测试“一句话生成课程”“依据真实学习证据持续帮助学会”“课程会随真实问题受控改进”三种价值表达，观察理解、信任、起课和次日继续学习。
4. **机制 PoC 实验**：优先验证 claim verifier 与最小 `CourseQualityIssue`，因为二者最直接补当前断点；BKT 与白板排在统一知识 ID 之后。
5. **许可证与供应链审计**：对准备复制的 MIT 代码、CC BY 内容、模型、字体、媒体和 SaaS 条款单独建清单；AGPL/GPL 默认只做行为与架构研究。

**Confidence：High**

**Basis：** 限制直接来自各研究任务的未验证项和证据边界。
