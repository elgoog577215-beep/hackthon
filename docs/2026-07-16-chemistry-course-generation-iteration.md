# 迭代需求文档：真实生成一门化学课程后发现的问题（2026-07-16）

## 背景与方法

前两轮工作（见 `docs/2026-07-16-p0-p1-dev-plan.md`）都是基于代码走查和单测复现问题，本轮第一次配置了真实 `AI_API_KEY`（DeepSeek），跑通了从"创建生成任务"到"课程完成"的完整真实链路，用系统内置的 `natural_science`（自然科学）学科生成体系生成了一门真实课程：

- 主题：`高中化学·化学反应速率与化学平衡`
- 学习对象：高中生
- 要求：覆盖反应速率影响因素、化学平衡移动原理（勒夏特列原理）及相关计算，配合典型例题
- `pedagogy_mode: natural_science`

这门课程之前不存在，选它是因为要验证"内置的多套学科生成体系"里，除了已经用真实数据接进知识库的"辩论"课，自然科学类课程走完整生成流程时会不会暴露新问题——结果暴露了两个之前代码走查没发现的真实 bug，且第一个是阻断性的（会让任务直接 `failed`）。

## 发现1（P0，已修复并验证）：生成任务因缺少 max_tokens 而必然失败

### 现象
第一次生成请求跑到 `blueprint_generation` 阶段后反复出现"蓝图未通过格式或硬约束检查，正在重新规划"，最终整个任务 `failed`，报错：
```
课程蓝图两次未通过结构验收：模型没有返回可用的课程章节 JSON
```

### 根因（已用真实响应内容定位，非猜测）
`backend/ai_base.py` 的 `_call_llm`/`_stream_llm` 从未向 API 传递 `max_tokens` 参数，所有调用都在吃服务商的默认输出长度上限（DeepSeek 未指定时约为几千 token）。`natural_science` 模式的课程蓝图 schema 会在每个章节/小节下嵌套 `capability_points`/`mistake_points`/`improvement_points` 等结构，输出体量远超默认上限。实测抓到的两次真实原始响应：

- 第1次：在第657行、一个字符串中间被硬切断（`"计算转化` 后直接结束，没有闭合引号和括号），`json.loads` 报 `Unterminated string`。
- 第2次：换了个更谨慎的开头（多了个 ` ```json ` 围栏），但同样在生成到一半（`"直接将1代入分` 后）被切断，连闭合围栏都没来得及输出。

两次都不是"模型输出了错误格式的JSON"，而是**模型还没写完就被截断**——这是一个纯粹的长度预算问题，和之前 `_extract_json` 里做的 `_fix_invalid_escapes`（针对 LaTeX 反斜杠转义）完全是两类问题，此前代码里没有任何机制探测或应对"输出被截断"这种情况。

### 修复（已提交，已用同一次真实请求验证解决）
- `AIBase.__init__` 新增 `self.max_tokens`（环境变量 `AI_MAX_TOKENS`，默认 `8192`）。
- `_call_llm` 新增 `max_tokens` 参数，调用 API 时显式传递；同时通过 `finish_reason == "length"` 探测截断并打印明确的警告日志（`AI response truncated by max_tokens=...`），以后再出现同类问题能立刻在日志里看到，而不必像这次一样靠手工复现。
- 大纲生成这一步（全链路里 schema 最重的一次调用）显式传入 `max_tokens=16000`。
- 用同一个化学课程请求重新跑：第一次蓝图仍然没通过硬约束检查（下面发现2会解释为什么"没通过"不代表内容不完整），但第二次成功产出可解析 JSON，任务顺利 `completed`。

### 影响范围判断
这不是化学专属问题，是**任何会让模型输出较长结构化 JSON 的路径**都会碰到的系统性问题——`natural_science` 模式因为模块最重所以最容易触发，但只要材料多、要求细，其他学科模式理论上也会遇到。建议后续观察各学科模式实测的 `max_tokens` 消耗量，如有需要可以按 `pedagogy_mode` 做更精细的预算分级，而不是所有调用共用一个默认值。

## 发现2（P0/P1，未修复，需要产品决策）：课程蓝图完全没有"覆盖度"校验

### 现象
修复发现1后，任务变成 `completed`，`generation_quality_report.final_status = "passed"`，`publication_allowed = true`，`blocking_issues = []`。但实际生成的课程只有 **1个章节、1个小节**，内容是"反应速率的概念与表达"——用户在 `requirements` 里明确要求的"化学平衡移动原理（勒夏特列原理）及相关计算"**完全没有出现**，甚至连"反应速率的影响因素"（模型自己在第一次蓝图尝试里列出的5个学习目标之一）都没覆盖到。

这门课以"通过了全部质量闸门、可以发布"的状态交付给用户，而实际只完成了大约 1/5 的需求范围。

### 根因
`course_plan_constraint_report`（`brief_satisfaction.course_shape`）里 `expected` 字段恒为 `{}`——系统从来没有根据用户输入的 `requirements`/`subject` 反推过"这门课大概应该有几章几节"这类最低限度的覆盖度预期，所以无论蓝图多单薄，`course_shape.passed` 永远是 `true`，也就永远不会成为 `blocking_issues` 里的一项。

结合发现1看，这两个问题会互相放大：一旦某次蓝图生成因为截断失败，重试机制目前的策略是"换一次更保守/更简短的蓝图再试"，而这个更简短的蓝图无论多单薄都会顺利通过唯一的"格式检查"，最终呈现给用户的是一个"看起来完全正常、质量报告全绿"但内容严重不完整的课程——**用户完全没有办法从课程状态或质量报告里发现这一点**，只有打开正文才能发现。这比"生成失败"更危险，因为失败至少会被看到，静默阉割不会。

### 建议的迭代方向（供讨论，本轮未实现）
1. 最小方案：把用户 `requirements` 里出现的关键主题词/`learning_objectives` 数量作为覆盖度的粗粒度信号，在 `course_shape` 里加一条"声明的学习目标数 vs 蓝图实际覆盖的小节数"的比例检查，比例过低时至少打一个 `weak` 级别的 issue（不一定要 block，但不能是"零信号"）。
2. 更彻底的方案：蓝图生成后增加一次独立的"覆盖度自查"LLM调用——把用户原始 `requirements` 和生成的蓝图目录一起喂给模型，让它明确回答"以下要求点是否已被覆盖"，结构化输出缺口列表，接入现有的 `course_plan_constraint_report.issues`。
3. 无论选哪种方案，都应该和发现1的重试机制联动：当蓝图因截断失败被迫"降级重试"时，至少应该在 `generation_meta`/日志里标记"本次蓝图是降级重试产物"，供后续做覆盖度检查或人工复核时参考。

这一条本轮没有实现，因为"多单薄算单薄""该不该 block 发布还是只是警告"是产品口径决策，不是我能替你定的；上面三个方案的优先级需要你来选。

## 发现3（正向结论，非bug）：单节点内容质量本身是可信的

抽查了实际生成的 6887 字符正文（"1.1 反应速率的概念与表达"）：结构完整（引入问题→现象与问题→模型与规律→实验与证据），公式正确（$v(A) = -\frac{\Delta c(A)}{\Delta t}$、速率比例关系式等），难度契合"高中生"定位，还带了一个具体的实验数据表格。`generation_quality`（`course_quality_v8` 契约）对这一节的 `score` 是 `1.0`，各难度维度也都是 `1.0`。

**结论**：这次测试暴露的两个问题都是"结构/流程层面"（蓝图能不能生成完整、生成出来的蓝图覆不覆盖需求），不是"单节点内容生成质量"层面——一旦流程走到正文生成这一步，真实 LLM 产出的内容质量是达标的。这个区分很重要：不要把"结构性流程bug"和"内容质量问题"混为一谈去排期，它们需要不同的修复思路（发现1/2是流程和校验逻辑，发现3证明底层 prompt/生成质量目前没问题）。

## 和之前两轮发现的关系

这次是"资料生产"（材料驱动生成）理解的延伸验证：本次生成没有上传任何参考资料（`materials: []`），`coverage_plan.gaps` 里如实记录了"没有可用资料证据"并回退到 `general_knowledge` 策略——这部分行为符合预期，不是bug。但它印证了第一轮报告里的判断：`grounding_strategy="material_first"` 的价值目前完全依赖有没有真实材料输入，而材料输入路径（P0-1 证据入口、P1-7 知识库联动）在前两轮已经做了部分修复。**发现1/2这次暴露的截断和覆盖度问题，和材料驱动无关，是更底层、更通用的生成流程问题**——换句话说，即使材料/证据链路全部修好，只要蓝图生成还会截断、还没有覆盖度校验，"生成出一门内容单薄但看起来质量全绿的课程"这个风险依然存在，建议提高到和 P0-1/P0-3 同级的优先级。

## 建议优先级排序

1. **已完成**：修复 `max_tokens` 缺失导致的生成任务阻断性失败（发现1）。
2. **P0，待你决策具体方案**：蓝图覆盖度校验缺失（发现2）——建议至少先落地"最小方案"（学习目标数 vs 小节数比例检查），因为这直接决定"质量报告全绿"这句话是否可信。
3. **持续验证**：建议后续用其他 `pedagogy_mode`（如 `programming_engineering`、`life_medical`）各跑一门真实课程，确认 `max_tokens=16000` 对它们是否也够用，以及覆盖度问题是否同样存在——这次只验证了 `natural_science` 一种模式。
