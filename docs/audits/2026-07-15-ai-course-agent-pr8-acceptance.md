## 4. 阻断性问题

### P0-1：真实学习行为没有进入适配触发链路

#### 规格要求

学生与 AI 的对话、连续追问、重解释行为、错题、笔记、理解检查和课程内输入，都应形成可被适配系统消费的学习证据。

#### 当前实现

- AI 问答写入的事件类型是 `assistant_question_submitted`。
- 问答事件的 `node_id` 通常是课程小节 ID，而不是正文块 ID。
- `evaluate_and_propose_change()` 只读取 `event_type="learner_self_reported"`。
- 它还要求事件的 `node_id` 必须直接等于 `block_id`。
- 前端没有“完全看不懂”或“讲得更详细”的正式自我报告入口。

#### 实际影响

学生在真实 AI 助手里连续两次、三次甚至更多次要求详细解释，都不会进入这次新增的适配判断逻辑。开发规格中的最小演示从第一步就中断。

#### 代码证据

- `backend/routers/assistant.py:62-78`
- `backend/learner_model_service.py:158-163`
- `backend/learning_events.py:181-194`

#### 必须修复

建立统一证据适配器，把以下事件统一转成带有学习者、课程、小节、正文块和知识点定位的正式证据：

- AI 提问和连续追问
- 请求重解释或举例
- 错题与练习结果
- 笔记与问题记录
- 理解检查
- 显式“完全看不懂”反馈

触发服务不得只监听一个人为构造的 `learner_self_reported` 事件类型。

### P0-2：证据阈值测试与开发规格直接冲突

#### 规格要求

同一小节连续两次以上要求更详细解释时，系统必须形成 `AdaptationHypothesis`；达到证据门槛后生成课程变更集。

#### 当前实现

测试明确断言：

```text
两条“讲得更详细”弱证据 -> 不触发
三条弱证据 -> 才触发
```

当前实现也没有在两次追问后形成一个处于观察状态的适配假设。

#### 实际影响

单元测试虽然通过，但它验证的是与需求相冲突的行为。测试全绿不能证明规格已满足。

#### 代码证据

- `backend/tests/test_learner_evidence_proposals.py:35-47`
- `backend/learner_model.py:746-803`

#### 必须修复

将“形成假设”和“正式生成课程变更”拆成两个状态：

1. 连续两次明确重解释请求后形成可追溯的 `AdaptationHypothesis`。
2. 根据证据强度、相互一致性、覆盖范围、时效性、可逆性和误判成本，决定是否生成 `CourseChangeSet`。
3. 极强单一证据可以直接越过观察阶段。

### P0-3：没有正式适配假设，生成内容也不是真正的教学补充

#### 规格要求

证据聚合后应形成独立的适配假设，并据此生成当前块的补充说明以及后续相关章节的预防性补充建议。

#### 当前实现

- 生产代码中没有正式的 `AdaptationHypothesis` 实体或仓库。
- `_template_supplement_text()` 不调用 LLM。
- 它只是把原正文和学生反馈原句拼接在一起。
- 生成范围固定为当前一个 block。
- 没有分析第三、四、五节等后续依赖节点。

代码注释已经明确写明这是 `template_mvp`，后续才准备替换成 LLM。

#### 实际影响

即使人工写入正确事件并触发提案，得到的也不是一段真正有教学价值的解释，只是一段“学生说自己没懂”的重复记录，更不会产生跨章节预防性调整。

#### 代码证据

- `backend/learner_model_service.py:93-121`
- `backend/learner_model_service.py:180-210`

#### 必须修复

1. 增加正式的适配假设对象，记录支持证据、反向证据、置信度、影响节点和状态。
2. 调用现有课程 LLM 层生成教学补充，而不是拼接反馈原句。
3. 生成前读取课程依赖图和知识映射，输出当前块及后续受影响节点。
4. 每个生成结果都必须经过现有质量闸门。

### P0-4：拒绝和重新生成在前端实际不可用

#### 当前实现

点击拒绝或重新生成时，脚本写入：

```text
itemPromptOpen = proposalId:itemId
```

模板却判断：

```text
itemPromptOpen === item.item_id
```

两个值不可能相等，因此输入框永远不会出现，用户也无法点击确认按钮。

后端的重新生成接口同样没有执行生成，只是把旧 item 标记拒绝，再追加一个 `after: null` 的空 item。前端又把后端返回的完整 proposal 错当成单个 item 合并。

#### 实际影响

- 接受按钮可能可调用。
- 拒绝按钮点击后没有可完成的操作。
- 重新生成按钮点击后没有可完成的操作。
- 即使直接请求后端，也只会得到空白候选。

这不满足开发规格要求的“分别接受、拒绝或重新生成”。

#### 代码证据

- `frontend/src/components/SideAIPanel.vue:199-215`
- `frontend/src/components/SideAIPanel.vue:824-847`
- `backend/change_proposals.py:460-525`
- `frontend/src/stores/changeProposals.ts:119-130`

#### 必须修复

修正前端 prompt key，并让重新生成真正调用生成服务。后端应返回明确的完整 proposal 或新 item，前端类型和处理逻辑必须与真实响应保持一致。

### P0-5：新旧课程数据体系没有接好，大量现有课程无法使用新功能

#### 当前情况

当前浏览器打开的课程：

```text
a84fd008-b74d-47ae-a990-85dde04f08ca
```

其课程文档接口返回：

```text
source_format = legacy_projection
```

本地当前课程数据中：

```text
5 门 canonical
19 门 legacy
```

新提案接受接口要求课程必须已经是 canonical，否则返回 409。前端加载课程时只读取 legacy projection，没有调用正式课程迁移接口。

与此同时，旧版 AI 重写路由仍可对 legacy 课程直接写入；对于 canonical 课程，这些旧路由通常在生成后才因存储保护失败。

#### 实际影响

- 多数历史课程能看到旧功能，但不能使用新提案接受能力。
- 同一页面根据课程版本不同，出现不同的功能和行为。
- 新旧课程修改规则并存，正是当前“新旧功能交替出现”的结构性根因。

#### 代码证据

- `backend/routers/change_proposals.py:82-84`
- `backend/routers/courses.py:118-129`
- `frontend/src/stores/course.ts:177-217`
- `backend/routers/nodes.py:95-163`

#### 必须修复

1. 明确 canonical `CourseDocument` 是唯一正式写入模型。
2. 为历史课程提供可审阅、可重试、幂等的迁移流程。
3. 前端在进入需要新能力的课程时完成迁移检查。
4. 迁移完成后关闭旧 AI 直接写入路径，统一进入候选或变更集。

## 5. 主要功能缺口

### P1-1：个体化隔离没有成立

#### 当前实现

- proposal 顶层没有正式 `user_id`。
- proposal ID 只由 `course_id + request_id` 生成。
- proposal 列表只按 `course_id` 查询。
- 拒绝与重新生成接口不要求学习者身份。
- 接受接口只把当前请求用户作为操作 actor，没有验证其是否是 proposal 所属学习者。
- 拒绝原因回流时没有传递 `user_id`，因此写入默认用户。
- 路由没有完整校验 URL 中的 `course_id` 与 proposal 所属课程一致。

#### 实际影响

如果同一门课程被多个学习者使用，一个人的个性化提案可能被其他人看到或处理；拒绝原因还可能进入错误用户的学习记录。

#### 代码证据

- `backend/change_proposals.py:64-93`
- `backend/routers/change_proposals.py:44-50`
- `backend/routers/change_proposals.py:120-150`
- `backend/change_proposals.py:419-455`
- `backend/learning_events.py:66-72`

#### 必须修复

所有 proposal、证据、假设和变更集必须同时绑定 `user_id + course_id`。所有读写接口都必须校验学习者、课程和 proposal 三者一致。

### P1-2：多作用域目前只是枚举和显示标签

#### 当前实现

虽然定义了以下 scope：

```text
block / section / sections / chapters / book
```

但生产生成器只创建 `scope="block"` 的单块提案。`apply_item()` 也只调用 `replace_block`，无法表达：

- 新增块
- 删除块
- 移动块
- 拆分与合并
- 调整难度契约
- 修改章节结构

测试中的多节点提案是手工构造的数据，不是生产链生成的结果。

#### 实际影响

系统无法完成开发规格要求的“当前块 + 多个后续小节”变更，也无法对增加、删除、移动和难度调整进行待确认审核。

#### 代码证据

- `backend/change_proposals.py:30-39`
- `backend/change_proposals.py:268-307`
- `backend/learner_model_service.py:185-203`

#### 必须修复

将 `ChangeProposal` 升级为真正的 `CourseChangeSet`，每个 operation 明确操作类型、目标节点、基础版本、前后内容、证据和影响范围。

### P1-3：稳定节点标识只在局部成立，大纲重生成场景没有实现

#### 当前实现

- 小节 ID 由当前章节和小节顺序重新计算。
- 内容块 ID 由节点 ID、顺序和块类型计算。
- 当大纲重排、插入小节或块类型变化时，原有语义节点可能获得新 ID。
- canonical 课程的整课重新生成接口目前直接要求改用单块候选，不存在完整的大纲语义匹配与 ID 复用流程。

#### 实际影响

大纲重新生成后，笔记锚点、学习记录、知识映射和个性化内容都有失联风险，无法满足“语义节点仍存在时复用稳定 ID”的要求。

#### 代码证据

- `backend/course_generation_workflow.py:115-163`
- `backend/content_blocks.py:55-57`
- `backend/routers/course_versions.py:184-198`

#### 必须修复

大纲重生成必须先进行旧节点与新节点的语义匹配，再决定复用、拆分、合并或新增 ID；不能按照新列表位置直接重新编号。

### P1-4：前端不是正式的 PendingChangeOverlay

#### 当前实现

- 提案集中显示在右侧 AI 面板，没有出现在正文对应块的位置。
- `before` 和 `after` 的后端真实类型是对象，前端类型却声明为字符串。
- 当前展示不是文本级或块级 diff，只是在 `<p>` 中输出值。
- 没有展示具体证据列表和完整受影响节点。
- `pendingItemsByBlockId` 虽然存在，但没有用于正文行内渲染。
- 接受后前端只修改 proposal item 状态，没有刷新正式课程正文。

#### 实际影响

用户无法直观看到“正文哪里变了、为什么变、影响哪里”；接受后当前正文还可能保持旧状态，直到重新加载页面。

#### 代码证据

- `frontend/src/types/changeProposal.ts:15-23`
- `frontend/src/components/SideAIPanel.vue:138-160`
- `frontend/src/stores/changeProposals.ts:30-41`
- `frontend/src/stores/changeProposals.ts:92-102`

#### 必须修复

正文块应直接读取 pending overlay，在原位置显示新增、删除和修改差异；右侧 AI 面板只承担理由、证据、范围和操作控制。接受成功后必须使用后端返回的正式文档刷新课程状态。

### P1-5：提案缺少基础版本，所谓乐观锁不能防止过期提案覆盖新内容

#### 当前实现

proposal 创建时没有保存：

- `expected_document_revision`
- `expected_block_revision`

用户点击接受时，路由重新读取当前课程的最新 revision，再把这个最新 revision 传给 `replace_block`。

#### 实际影响

如果提案生成后正文已经被其他操作修改，接受旧提案仍可能把旧的 `after` 内容覆盖到最新正文上。当前乐观锁只防止“点击接受到提交之间的瞬时并发”，不能识别“提案本身已经过期”。

#### 代码证据

- `backend/change_proposals.py:182-245`
- `backend/routers/change_proposals.py:82-95`
- `backend/course_commands.py:16-51`

#### 必须修复

提案必须保存创建时的文档和目标块 revision。接受时先验证基础版本；不一致则进入冲突状态，要求重新生成或人工比较，不得自动覆盖。

### P1-6：来源证据没有完整展示和写入版本记录

#### 当前实现

学习证据 ID 只放在部分 proposal 的 `generation_meta` 中：

- 前端没有展示这些证据。
- `apply_item()` 写入课程时只传递 reason 和 actor。
- 课程操作日志没有保存来源证据引用。
- 拒绝证据写入使用 `except Exception: pass`，失败时会静默丢失。

#### 实际影响

用户看不到 AI 为什么提出修改，事后也无法从课程版本追溯这次修改使用了哪些学习证据。开发规格要求的“接受后记录版本与来源证据”没有满足。

#### 代码证据

- `backend/learner_model_service.py:203-210`
- `backend/change_proposals.py:297-307`
- `backend/course_repository.py:308-326`
- `backend/change_proposals.py:419-457`

#### 必须修复

证据引用必须成为 ChangeSet operation 和课程操作日志的正式字段。MUST 级别的证据回流不能使用无记录的静默失败，应持久化失败状态并支持重试。

### P1-7：知识库双向联动没有形成可用闭环

#### 当前实现

正文到知识库方向：

- 能创建 `target_kind="kg_node"` 的待处理条目。
- 但知识库是静态只读目录。
- 接受该条目固定返回 409。
- 前端直接隐藏接受按钮。

知识库到正文方向：

- 存在 `propose_content_linkage_from_kb_change()` 辅助函数。
- 当前仓库没有知识库修改路由，因此没有生产调用入口。
- 源码注释明确说明它只是未来接入点。

此外，当前连接的是学科级 `SubjectKnowledgeLibrary`，没有开发规格要求的课程级 `CourseKnowledgeBase`。

#### 实际影响

两个方向都不能完成“提出建议 -> 用户审核 -> 接受后写入”的闭环，不能判定双向联动已完成。

#### 代码证据

- `backend/course_knowledge_map.py:298-392`
- `backend/subject_knowledge.py:235-320`
- `backend/routers/change_proposals.py:71-81`

#### 必须修复

先建立可版本化、可写入的课程级知识库和知识库命令服务，再让两个方向都生成带基础版本的待确认 ChangeSet，并支持接受、拒绝和冲突处理。

## 6. 生成质量闸门验收

### 已经具备

1. 课程节点有确定性质量检查。
2. 节点质量不合格时能够调用模型进行一次修复。
3. 节点生成失败能够隔离，不必拖垮全部生成任务。
4. 最终课程存在质量报告和发布门槛。
5. 资料经过文档块和证据单元处理，优于直接硬切整份文本。
6. 已保存 `prompt_contract_version`。

### 仍然缺失

1. 课程蓝图 JSON 解析失败时直接使用 fallback，没有把校验错误反馈给模型受控重试。
2. fallback 被描述为“蓝图初稿已生成”，用户无法识别模型输出实际失败。

## 11. 可直接转发的验收结论

PR #8 的代码不是完全无效，变更提案仓库、逐 item 状态、动态证据评分和现有课程质量闸门都可以保留。但当前实现只完成了若干底层零件，没有满足开发规格规定的第一阶段唯一验收闭环。

最核心的问题不是界面没打磨，而是真实学习行为没有进入适配链路、没有正式适配假设、没有真实教学内容生成、没有跨章节变更、拒绝与重新生成不可用、历史课程无法接受新提案、学习者隔离缺失、知识库双向联动无法落库。现有测试主要验证了人工构造的局部模块，未验证真实产品路径，其中部分测试预期还与规格直接冲突。

因此本次提交应判定为“基础设施部分完成，第一阶段验收未通过”。后续应以开发规格第 8 节为唯一收束目标，先建立真实端到端测试，再按本报告的返工顺序补齐，不应继续以接口存在或单元测试全绿作为完成标准。
