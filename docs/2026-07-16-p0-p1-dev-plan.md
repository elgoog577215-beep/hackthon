## 4 个遗留问题的调查结论（2026-07-16 第二轮）

### 1.「卡在32%」根因定位（已定位，非纯UI显示问题）

`course_service.py` 的 `build_course_draft` 里进度上报顺序：
`requirement_analysis(5%) → material_processing(25%) → pedagogy_resolution(32%, course_service.py:373-378) → [此处调用大纲生成 LLM] → blueprint_generation(42%)`。

**32% 正好是"教学画像与难度契约已确定"上报之后、大纲生成 LLM 调用（`course_service.py:388-393`，`self._call_llm(..., enable_thinking=True)`）返回之前**——这段区间里没有任何中间进度上报，调用本身是一次同步 `await`。

关键：这次调用允许的最坏情况等待时间非常长。`ai_base.py` 的模型候选列表 `DEFAULT_SMART_MODELS` 有 **8 个候选模型**，每个模型 `retry_count=3` 次重试，每次请求超时 `AI_REQUEST_TIMEOUT_SECONDS=180`秒（默认）。如果 AI 服务商响应慢或对某类请求普遍降级，最坏情况下这一步可以同步阻塞长达 `8 × 3 × 180s ≈ 72 分钟`才失败转入兜底大纲（`_create_fallback_plan`）。`enable_thinking=True` 还会进一步拉长每次请求的实际耗时。

**结论**：这大概率不是"进度条显示错误"，而是**大纲生成这一步的 LLM 调用真的在等**，等待时长完全取决于当时 AI 服务商的响应状况——这也解释了为什么"昨天也是"（服务商当天响应普遍慢/降级会连续多天复现）。上传参考资料本身不是根因，但材料越多、`course_generation_brief`/证据目录越大，喂给大纲 prompt 的上下文越长，也会拉长这次调用的实际耗时，与"上传了参考资料"的观察相关但不是直接因果。

**建议后续动作**（本轮未改代码，涉及超时策略属于产品/成本决策，需要确认后再动）：
- 给这一步单独上报一个"正在生成课程大纲，可能需要较长时间"的中间态提示，而不是让 32% 停留到用户以为卡死；
- 评估是否要缩短 `AI_REQUEST_TIMEOUT_SECONDS` 或减少 `DEFAULT_SMART_MODELS` 候选数量，用"更快失败+更快兜底"换取更好的进度体感（代价是极端情况下可能提前放弃本能成功的慢请求）。

### 2. 知识库「当前学科还没有可用的正式知识库」——确认为数据缺口，不是 bug

- `backend/catalogs/subject_knowledge/` 目录下**只有一个**学科知识库文件：`math-linear-algebra-v1.json`（线性代数），`match_terms` 只匹配"线性代数/向量空间/矩阵/高斯消元/linear algebra"。
- `backend/subject_knowledge.py::resolve_subject_library` 对任何课程标题匹配不上这些关键词的情况，返回 `status="unavailable"`、`nodes=[]` 的空知识库——这是显式设计好的兜底，不是异常路径。
- 实测当前 `backend/data/courses/` 下**唯一一门课程**是《辩论：逻辑构建与实战技巧》，标题显然匹配不上线性代数的 `match_terms`。
- **结论**：图3截图的空状态 100% 是因为"辩论"这门课没有对应学科的知识库数据，而不是接受/展示链路坏了。要让知识库面板真正有内容，要么（a）新增"辩论/论证"学科的知识库 JSON 目录（工作量：编写一份新学科的知识点/关系/技能点数据，参照现有线性代数文件格式），要么（b）用一门线性代数课程做 demo。

同时验证了 P1-7 报的 409：直接调用 `apply_item` 的 kg_node 分支，返回体确认为：
```json
{"code": "kg_node_apply_not_supported", "message": "该条目目标是知识库节点，当前知识库为静态只读目录，暂不支持自动接受；请人工核对后在知识库目录中更新，或选择拒绝/暂不处理该条目。"}
```
这与验收报告描述的"固定返回409"一致；由于无法访问"同学版本"的实际代码（见下），无法验证图1所述"被当成课程正文块处理"是否为该分支的另一种措辞，还是真的存在别的报错路径——但当前 `main` 分支只有这一条 409 路径。

### 3. "同学版本"分支——无法在本仓库定位，需要你提供访问方式

检查了本地所有可见分支/远程（`origin/ljy_dev`、`origin/codex/ai-chain-cleanup`、`origin/master`、`fork/*` 等）：
- `origin/ljy_dev`（作者 Wayne Lee）**确实包含知识图谱相关组件**，但它最后一次从 main 合并是在改动引入"变更提案系统"（change_proposals.py 等）**之前**——`git diff` 显示它完全没有 `change_proposals.py`/`course_knowledge_map.py` 这些文件，说明它是基于更早期的 main 分叉出去的独立版本，不是"在当前 PR#8 基础上新增5项"的那个"同学版本"。
- 其余远程分支（`codex/ai-chain-cleanup`、`master`、`fork/*`）与当前 `main` **没有共同祖先**（`git merge-base` 报错），是完全独立的历史，同样不像是"基于当前主干的同学分支"。
- **结论**：这个仓库里找不到与需求文档描述吻合的"同学版本"分支。它大概率存放在协作者本地或另一个未推送到这个远程的仓库/fork 里。**需要你提供实际的分支名/远程地址，或者让协作者把它推送上来**，否则无法核实"3个TS错误"和"被当成课程正文块处理"的具体报错这两点。

### 4. 是否值得先投入两周做「路径B最小闭环」——这是产品决策，我只给分析不替你定

基于本轮实测：
