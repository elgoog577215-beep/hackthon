## 二、产品核心原则

### 原则一：结构化同源

课程不是互不相干的多份页面，也不是"先生成大纲、再生成正文、再生成 PPT"各自独立的多份副本，而是一棵拥有统一语义源的结构化课程树。

- 大纲、章节、知识点、概念、推导、例子、练习、检查、总结等都是具有稳定标识与关系的课程节点。
- 目录、正文、知识地图，以及未来的 PPT，都应是同一课程源的不同视图/投影。
- 一处变化发生时，系统必须能识别其影响的上游、下游及相关节点，形成可审阅的联动变更，而不是产生互相失真的多个版本。

### 原则二：个体化生长

课程不是生成后固定不变的成品，而是随单个学生学习证据持续演化的个人课程。

- 公共课程保持稳定；个人学习流根据正式学习证据动态增加解释、例子、练习和补救。
- AI 综合学生与 AI 的对话、追问、重解释行为、错题、笔记、理解检查和课程内输入，判断学生当前理解状态。
- 证据足够时，AI 可对当前内容及未来章节提出增加、修改、替换、删除、移动、难度调整等建议，均以「待确认变更」形式进入课程，由学生接受/拒绝/重新生成并可填写理由；变更范围可为局部块、多个小节、多个章节或全书。

## 三、核心系统模型

### 1. 三层知识结构

```
SubjectKnowledgeLibrary（学科知识库，可复用学科级底座）
        ↓ 选取、组织、约束
CourseKnowledgeBase（课程知识库，生成课程时建立，可持续细化/扩充/修正）
        ↓ 映射到具体课程结构
CourseKnowledgeMap（课程知识地图，表达知识节点与章节/课程块的映射）
```

### 2. 证据到课程变化的决策链

```
EvidenceItem（原子学习证据）
        ↓ 聚合与解释
AdaptationHypothesis（适配假设）
        ↓ 达到动态证据门槛
CourseChangeSet（课程变更集）
        ↓ 学生审阅确认
CourseDocument / Knowledge Base（正式版本）
```

证据门槛不可写死为固定次数触发，需综合证据强度、一致性、覆盖范围、时效性、可逆性和误判成本；单条强证据可立即触发，多条弱证据可累积触发。

### 3. 课程运行时结构

```
CourseDocument（已确认课程）
  + PendingChangeOverlay（待确认 AI 变更层）
  + AdaptiveBlock（可演化课程块）
  + LearningRecord（学习记录与证据）
```

待确认内容不得直接覆盖正式课程，需以高亮、差异标记、"AI 生成"标签长期保留，直到学生接受或拒绝。

## 四、产品与权限边界（不可偏离）

1. AI 操作权限高，但正式写入权限低：AI 可提出任何合理课程操作，但不能替学生确认。
2. 所有 AI 变化必须可解释：至少展示调整理由、使用的证据、影响范围。
3. 所有未确认变化必须可见：不能静默修改正式课程，也不能因用户暂不处理而自动消失。
4. 证据判断必须动态：不将固定次数作为唯一触发条件，也不能凭一句模糊反馈全局改课。
5. 支持局部与全局范围：当前块、当前小节、多个小节、多个章节、全书均需有明确作用域。
6. 当前仅在当前课程内生效：暂不将单课程的个性化调整自动传播到其他课程或教师端。
7. 课程与知识结构需双向联动：正文变化可提出知识库变化，知识库变化也可提出课程受影响节点的联动变化。
8. 启智联动、PPT 双向联动、视频讲解暂不属于本轮实现范围（架构上预留扩展接口，不扩大当前开发范围）。

## 五、参考实现：当前"启智"生成链路审计结论（技术基线，供设计参考）

以下内容摘自对启智（教师端）现有 dev 分支源码的只读审计，反映当前一套可运行的"结构化提示词驱动生成"实现方式，可作为灵知在课程生成/多产物一致性设计上的参考基线，不代表灵知需照搬。

### 1. 生成能力总览

全部生成能力基于 OpenAI 兼容的 vLLM/Qwen 调用层，提示词由 system message、user message、历史消息与结构化 Schema 共同决定输出。

| 功能 | 提示词资产 | 调用形态 | 主要产出 |
|---|---|---|---|
| 教学大纲 | outline/prompt.md + analyze/verify/refine | 4 步多次调用 | Markdown 大纲 |
| 教案 | teaching_plan/prompt.md + analyze/verify/refine | 4 步多次调用 | Markdown 教案 |
| 题库 | question_bank/prompt.md | 单次流式调用 | Markdown 题库 |
| PPT | slides.py 内 4 个硬编码 Prompt | 页序 + 并发逐页填充 | HTML + .pptx |
| 通用助手 | assistant/prompt.md | Plan + ReAct 循环 | 回答与工具操作 |
| 视频分析 | video_analyzer/prompt.md | ReAct + 转录查询工具 | JSON 评价 |
| 文档/PPT 分析 | analyzer.py 内嵌 Prompt | Map-Reduce 并发分析 | 六维报告 |

共享调用层：
- PromptManager 递归读取 agents 目录下的 .md 文件，以文件夹名或文件名作为 key，并进程内缓存。
- 通用 LLM 层采用 system + history + user 的 Chat Completions 结构，默认关闭 thinking。
- PPT 自建同步 Runner：强制 JSON object、以 Pydantic 校验、失败时携带校验错误重试。

### 2. 大纲/教案的四步质量管线

分析 → 初稿 → 核查 → 修订，四步分别独立成 Prompt，职责单一，便于替换与测试：

| 步骤 | 模板 | 输入的核心信息 | 产出与约束 |
|---|---|---|---|
| 1. 分析 | _pipeline/analyze.md | 任务描述、课程表单或已有大纲、用户要求 | 不写正文，给出目标/章节/篇幅/文献方向/易错点，限 400 字 |
| 2. 初稿 | outline/prompt.md | 真实表单字段 + 第1步分析 + 用户要求 | 完整 Markdown 大纲，含课程信息/目标/考核/教学安排/参考资料 |
| 3. 核查 | _pipeline/verify.md | 已生成初稿 | 严格 JSON issues；检查参考文献、客观事实、字段一致性 |
| 4. 修订 | _pipeline/refine.md | 初稿 + 核查 issues + 用户要求 | 仅按问题清单修改，输出完整定稿 Markdown |

题库为单次流式生成，无四步质量管线，质量门槛低于大纲/教案（已在审计中列为差异项，见"风险清单"）。

### 3. PPT 生成链路

PPT 文本提示词位于 `server/agents/ppt/slides.py`，不由 PromptManager 管理，模型只生成 deck 数据（JSON），最终文件由固定渲染器（HTML + python-pptx）创建。

真实调用顺序：`教案/大纲 + 用户要求 → OUTLINE_PROMPT 生成 DeckPlan → 对每个 content/qa/summary 页并发调用对应填充 Prompt → 统一 deck → HTML 与 python-pptx 渲染`

输入限制：
- 页序阶段仅传入教案前 16,000 字；逐页填充每页仅传入教案前 6,000 字。
- 页数默认 12–24 页，由模型在合法页数内裁定；封面、提纲、小结页数固定，其余上限裁剪。
- 用户输入仅拼入页序 Prompt 的"用户额外要求"部分，不会自动传入每张内容页的填充 Prompt；用户要求主要影响结构与标题，逐页文本主要受教案正文约束。

### 4. 结构化约束与兜底机制

| 机制 | 实现 | 效果 | 边界 |
|---|---|---|---|
| JSON Schema | PPT DeckPlan/SlideFill、文档分析结果采用 Pydantic | 减少格式漂移，便于程序消费 | 只验结构，不验事实与教学质量 |
| 纠错重试 | JSON 解析失败后将错误追加到 user prompt 并降低温度 | 提高可用率 | 可能增加时延和成本 |
| PPT 兜底页序 | 按 Markdown 标题自动生成封面/提纲/内容/讨论/小结 | 页序调用失败也能产出文件 | 不是语义完整课件 |
| 单页隔离 | 某页填充失败时返回空 bullets/notes/terms | 不拖垮整套 PPT | 可能产生空白页 |
| 固定渲染 | python-pptx 与 HTML 使用同一 deck | 样式稳定、可下载 | 视觉多样性有限，未做渲染后质检 |

### 5. 已识别风险与优先级建议（用于灵知规避同类问题）

| 优先级 | 问题 | 现状 | 建议 |
|---|---|---|---|
| P0 | 默认密钥逻辑位于源码 (agents/llm.py) | 硬编码 | 迁移至环境变量或密钥服务，轮换已泄露密钥 |
| P1 | "事实核查"无联网或检索能力 (verify.md 明示无联网检索能力) | 只能做自洽性判断 | 对参考文献/外部引用接入外部检索、来源链接与引用可验证性 |
| P1 | PPT 用户要求只影响页序，未逐页贯彻 (slides.py 仅 OUTLINE_PROMPT 拼接 user prompt) | 逐页填充不感知用户要求 | 将用户意图提炼为结构化 deck policy，传给每页填充 |
| P1 | 超长教案前级截断 (16,000/6,000 字固定上限) | 可能丢失关键内容 | 按章节语义检索或分块，显式提示已截断范围 |
| P1 | 缺少最终 PPT 内容与视觉质检 | 无渲染截图/空页检测链路 | 生成后检查空页、重复标题、密度、文字溢出与覆盖度 |
| P2 | 模型版本不固定（运行时取模型列表第一项） | 结果不可复现 | 固定正式模型 ID，写入生成记录 |
| P2 | 提示词版本不可追溯（仅进程缓存，无版本 receipt） | 无法追责/复盘 | 保存模板 hash、模型 ID、参数、截断与调用统计 |
| P2 | 生成核心缺少对应单元测试 | tests 中未见 PPT/prompt/pipeline 专项测试 | 增加 fixture、Schema、兜底、渲染与回归样本测试 |

**重点判断**：正确演进方向不是单纯"继续加 prompt"，而是把 prompt 视为可版本化资产，并补齐上下文检索、外部事实验证、生成后质检和固定评测集，才能把"偶尔生成得不错"升级为可稳定交付。灵知在设计课程生成、AI 变更审阅链路时应直接规避以上 P0/P1 问题（密钥管理、事实可验证性、用户意图逐层贯彻、生成后质检、模型与提示词版本可追溯）。

## 七、研发启动建议

1. 以统一的实施变更单（如 OpenSpec 变更 `build-structured-adaptive-course-ai`）承载本轮开发，避免需求碎片化。
2. 先完成实体、版本、证据、变更集与状态机等底座，再接入 AI 推理与前端交互。
3. 先跑通一条最小纵向闭环：学生留下证据 → AI 生成局部变更 → 页面高亮 → 学生接受/拒绝 → 正式课程更新 → 记录可追溯。
4. 再扩展到未来章节预调整、跨章节作用域、知识库双向联动、难度组合调整。
5. 每阶段均对照场景验收，不以"接口可调用"代替产品闭环完成。

## 八、最小演示闭环（首选展示链路）

```
学生在第二节连续要求更详细解释
→ 系统积累对话与重解释证据
→ AI 判断当前课程的推导颗粒度偏粗
→ 在当前块补充一段解释
→ 对第三、第四、第五节提出预防性补充建议
→ 新增内容和行内修改均高亮并标记"AI 生成"
→ 学生查看理由与范围
→ 学生分别接受、拒绝或重新生成
→ 接受项写入正式课程，拒绝原因回流为新证据
```

该链路同时展示结构化编辑、证据驱动、课程主动生长、未来章节联动、人机共治与可追溯性，是本项目最具辨识度的产品能力，建议作为第一阶段 Demo 目标。
