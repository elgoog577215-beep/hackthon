# 外部 PPTX 再导入评估

> 评估日期：2026-07-19
> 对应 OpenSpec：`build-same-source-teaching-representations` 任务 4.6
> 结论：完成方案评估；当前版本不开放任意 `.pptx` 直接回写课程真源。

## 1. 评估问题

灵知当前的 PPT 闭环是：

```text
CourseDocument / CourseKnowledgeBase / PracticeTask
→ SlideDeckSpec + SourceBinding
→ 浏览器预览 / 可编辑 PPTX
→ 工作台结构化编辑
→ 影响分析与教师确认
→ 课程命令与精准重建
```

外部 PPTX 再导入要解决的不是“能否读取文本框”，而是能否证明文件里的每项修改
仍然对应某个课程来源、某个课件单元和某个可安全修改的字段。缺少这层证明时，直接把
外部文件内容写回 `CourseDocument` 会绕过来源绑定、影响分析、质量门和教师确认。

## 2. 当前实现事实

- 灵知可以从同一份 `SlideDeckSpec` 生成浏览器预览和可编辑 `.pptx`。
- 导出文件保留可见文本、版式对象和讲者备注，但没有嵌入可校验的同源 manifest。
- 当前没有 `.pptx` 上传入口、PPTX 解析服务、外部对象到 `unit_id` 的映射器或回导冲突模型。
- `python-pptx` 能读取常见文本、形状和备注，不代表它能无损恢复 SmartArt、图表、
  动画、母版、组合对象、嵌入媒体、公式对象和第三方扩展。
- 外部编辑可能改动对象 ID、拆分或合并文本框、复制页面、替换版式；仅靠页面序号、
  标题或文本相似度无法建立可信的 `SourceBinding`。

因此，当前导出的 PPTX 是正式交付物，不是可无损往返的课程数据文件。

## 3. 方案比较

| 方案 | 能力 | 同源安全性 | 结论 |
|---|---|---:|---|
| 任意 PPTX 直接解析并覆盖课程 | 读取页面文本后猜测课程位置 | 低 | 拒绝；会产生静默误绑定 |
| 任意 PPTX 作为资料导入 | 抽取文本、图片和备注，进入资料证据链 | 中 | 可作为独立需求建设，但不得称为回导 |
| 灵知导出件受控回导 | 依赖导出时嵌入的 manifest，对受支持字段生成差异 | 高 | 推荐的后续实现方向 |
| 只在工作台内编辑 | 直接编辑 `SlideDeckSpec` 并沿现有影响链确认 | 高 | 当前正式路径 |

## 4. 决策

### 4.1 当前版本

1. 继续以 PPT 工作台的结构化编辑作为唯一同源反向编辑入口。
2. 不展示“导入 PPTX 并同步课程”按钮，不接受按页码或文本相似度自动回写。
3. 用户提供的任意外部 PPTX，未来可以走“导入为课程资料”流程；抽取内容必须带
   `external_material` 来源，不能继承当前课程的知识、目标或题目身份。
4. PPTX 导出失败或重新生成失败时，保留注册表中的上一可用 `SlideDeckSpec`；未通过
   质量门的临时页面不得替换正式版本，也不得导出。

### 4.2 后续受控回导

只对带有效灵知 manifest 的导出件开放受控回导。导出时至少嵌入：

```text
course_id
representation_id / representation_revision
representation_compiler_version / slide_compiler_version
source_revision_vector
slide.unit_id
supported_field_hashes
manifest_schema_version
manifest_signature
```

回导流程必须是：

```text
上传隔离副本
→ 校验文件、manifest 与签名
→ 读取受支持字段并按 unit_id 对齐
→ 与导出基线做三方差异
→ 分类为 presentation / equivalent_semantic / semantic / ambiguous
→ 展示冲突和影响范围
→ 教师确认
→ 复用课程命令或 presentation_overrides
→ 精准重建并通过质量门
```

受支持字段第一阶段只考虑标题、副标题、核心信息、讲者备注和已登记版式；新增/删除
页面、任意形状变化、动画、媒体、图表和公式对象都进入显式冲突，不自动写回。

## 5. 必须拒绝或降级的情况

- 文件不是合法 OPC/PPTX，包含宏、外部链接、嵌入包或超过大小/解压预算。
- manifest 缺失、签名错误、课程不匹配或编译器版本不兼容。
- 课程当前修订与导出基线不同，且三方合并不能唯一确定结果。
- `unit_id` 缺失、重复，页面已删除或被复制到多个位置。
- 修改涉及不支持的对象，或被判定为 `ambiguous`。
- 导入结果未通过来源完整性、跨产物一致性或课件质量门。

以上情况不能静默忽略。系统应提供“作为资料导入”“保留为独立 PPT”或“返回工作台
手动修改”的明确选择。

## 6. 实施前置与验收门

后续实现前必须先完成以下基础能力：

- 导出 manifest 和签名协议；
- 导出基线的不可变保存与按修订读取；
- 受支持字段的三方差异与 `override_conflicts`；
- 文件安全扫描、解压预算和资源隔离；
- 回导候选的幂等、撤销、失败恢复与审计记录。

最小验收矩阵应覆盖：未修改往返、仅表现修改、明确语义修改、课程先变化、页面删除/
复制、对象重排、manifest 篡改、超大文件、质量门失败，以及失败后上一可用版本仍可用。

## 7. 与当前 Demo 的关系

答辩 Demo 使用“新建课程 → 同源 PPT → 工作台语义编辑 → 精准同步 → PPTX 导出”，
不包含外部 PPTX 回导。录屏中不得把导出的 PowerPoint 原生可编辑性描述为“可回导”；
准确表述是“可在 PowerPoint 继续使用，课程内的同源修改请回到 PPT 工作台完成”。
