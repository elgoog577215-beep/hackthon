# 个人学习操作系统升级规划

> 历史规划说明：本文只保留早期 V1 推演，用于追溯为什么曾出现 `LearnerState -> TeachingDecision -> LearningOSSnapshot`。这条链及其 profile、导师卡、legacy signals、默认用户兼容均已退出生产路径，本文任何“当前状态”表述都不得作为实现依据。当前唯一依据是 `docs/product-blueprint.md`、`docs/architecture/course-generation.md` 和当前 OpenSpec。

本文是一份面向下一阶段升级的前期规划。它不从现有功能出发，而是先定义理想型目标，再反推系统应具备的能力、架构层次和推进路线。

## 1. 北极星目标

这个产品不应只被定义为“AI 课件生成工具”，也不应只是“课程树加聊天框”。

它更大的目标是：

**成为一个能持续理解学习者、动态组织知识、主动调整教学策略的个人学习操作系统。**

它最终要给用户的感觉不是：

```text
我输入一个主题，AI 给我生成一门课。
```

而是：

```text
我拥有一个越来越懂我的 AI 老师。
它知道我正在学什么、卡在哪里、该怎么讲、什么时候该练、什么时候该复习。
```

系统的智能感不应只来自回答漂亮，而应来自连续性、判断力和教学闭环：

- 它记得用户过去问过什么。
- 它知道用户在哪些概念上反复出错。
- 它能根据课程结构判断当前问题依赖哪些前置知识。
- 它不会只回答问题，还会判断下一步应该讲解、追问、练习、复习还是回到前置知识。
- 它能解释为什么建议用户走下一步。

## 2. 当前框架的核心限制

现在系统已经具备几个重要基础：

- 课程生成链路已经收拢到 `CourseService`。
- 学习者信息已经通过 `LearnerContext` 聚合。
- AI 助手已经通过 `assistant_context` 统一构造上下文。
- `tutor_service` 已经有错题、掌握度、目标和主动建议雏形。

但它仍然容易停留在“功能模块拼装”阶段：

```text
课程生成模块
+ AI 问答模块
+ 笔记错题模块
+ 导师建议模块
```

这套组合可以做出一个有用的学习工具，但还不足以成为真正的智能老师。

真正缺的是一个更高层的“教学决策系统”：

```text
系统不只是知道有哪些内容和记忆，
还要知道此刻应该采取什么教学动作。
```

如果继续在旧框架里补功能，风险是：

- 课程越做越复杂，但学习闭环没有变强。
- 记忆越塞越多，但 AI 不知道该用哪种记忆。
- 助手回答越来越长，但不会判断用户是否真的掌握。
- 导师功能越来越多，但和课程、问答、练习之间缺少统一调度。

所以下一阶段不能以“再加一个 AI 功能”为主线，而要以“形成完整学习闭环”为主线。

## 3. 理想系统的核心闭环

理想系统应该围绕一条闭环运转：

```text
学习目标
→ 知识结构
→ 学习者状态
→ 教学决策
→ 教学动作
→ 学习反馈
→ 记忆更新
→ 下一步行动
```

每一次用户学习、提问、做题、保存笔记或复习，都应该推动这个闭环前进。

### 3.1 学习目标

系统首先要知道用户为什么学：

- 是入门理解？
- 是考试复习？
- 是掌握技能？
- 是准备项目实践？
- 是快速建立全局地图？

同一门课程，目标不同，教学策略就不同。

例如“线性代数”：

- 为考试学：强调定义、定理、题型、易错点。
- 为机器学习学：强调向量空间、矩阵运算、特征值、几何直觉。
- 为工程应用学：强调计算流程、库函数、数据表示和实践案例。

所以课程生成不能只问主题，还要逐步理解学习目标。

### 3.2 知识结构

课程不能只是目录树。

目录树回答的是：

```text
内容按什么顺序展示？
```

知识结构还要回答：

```text
概念之间如何依赖？
这个题考了哪些概念？
这个误区来自哪个前置断点？
当前节点不会，是因为当前节点没懂，还是前面没搭好？
```

未来的课程结构应逐步从“章节树”升级为：

```text
章节树 + 概念图谱 + 前置依赖 + 误区索引 + 练习映射
```

### 3.3 学习者状态

学习者不只是有画像。

画像通常是静态描述：

```text
数学基础一般，喜欢例子，偏好循序渐进。
```

但真正有用的是动态状态：

```text
用户刚刚在左右极限上出错。
用户能复述定义，但不能迁移到题目。
用户对图像直觉有帮助，但对形式定义抗拒。
用户当前卡点不是极限本身，而是量词顺序。
```

下一阶段要把学习者状态拆得更清楚：

- 掌握度：用户对某个概念掌握到什么程度。
- 置信度：系统对这个判断有多确定。
- 卡点类型：概念不懂、推理断裂、记忆遗忘、题型迁移失败、表达模糊。
- 学习偏好：例子优先、形式定义优先、图像直觉优先、练习驱动。
- 风险误区：反复出现的错误理解。
- 近期状态：当前是否困惑、是否连续提问、是否刚刚做错题。

### 3.4 教学决策

这是下一阶段最关键的层。

系统每次面对用户时，都应该先判断：

```text
此刻最应该做什么？
```

可选教学动作包括：

- 直接解释
- 换一种讲法
- 给例子
- 画结构图
- 追问理解
- 出一道小题
- 回到前置知识
- 提醒旧错题
- 总结成笔记
- 安排复习
- 推进下一节

当前 AI 助手主要还是“回答问题”。理想助手应该先做教学决策，再生成回答。

也就是说，未来的调用链应从：

```text
问题 + 上下文 → AI 回答
```

升级为：

```text
问题 + 上下文 + 学习者状态
→ 教学决策
→ 选择教学动作
→ 生成具体内容
→ 更新学习状态
```

### 3.5 学习反馈

没有反馈，系统就不知道教学是否有效。

反馈不应只来自显式测验，也可以来自多种行为：

- 用户说“不懂”“懂了”“再举个例子”。
- 用户连续追问同一概念。
- 用户保存笔记。
- 用户做题正确或错误。
- 用户跳过某节内容。
- 用户在某个节点停留很久。
- 用户要求重写或简化内容。

这些都应该被看作学习事件。

### 3.6 记忆更新

记忆不是把所有东西塞进 prompt。

记忆应该分层沉淀：

- 事实记忆：用户做过什么、问过什么、错过什么。
- 能力记忆：用户在哪些概念上强或弱。
- 偏好记忆：用户更适合什么讲法。
- 教学记忆：哪些解释方式对这个用户有效。
- 风险记忆：哪些误区反复出现。

每一层记忆的用途不同，不能混成一坨。

### 3.7 下一步行动

系统每次交互后都应能产生一个合理的下一步：

- 继续学习下一节。
- 回到某个前置知识。
- 做一道理解检查题。
- 复习一个薄弱点。
- 总结当前节点。
- 把一个概念加入错题/误区追踪。

真正的智能感来自这里：用户不需要自己管理整个学习过程，系统会带着他往前走。

## 4. 目标架构

下一阶段可以把系统目标架构定义为七层。

```text
目标层：用户为什么学
知识层：知识如何组织和依赖
状态层：用户现在懂到哪里
决策层：此刻应该采取什么教学动作
执行层：生成课件、回答、练习、复习
反馈层：用户产生了什么学习证据
记忆层：哪些状态需要长期沉淀
```

### 4.1 目标层

目标层负责保存用户本轮学习的意图。

它不需要一开始很复杂，但至少要能表达：

- 学习主题
- 学习目的
- 目标水平
- 时间约束
- 偏好风格
- 是否面向考试、项目、兴趣或工作

目标层会影响课程大纲、节点难度、练习密度和 AI 助手回答方式。

### 4.2 知识层

知识层负责把课程从“内容集合”变成“可推理结构”。

它应逐步维护：

- 概念列表
- 概念依赖
- 节点到概念的映射
- 题目到概念的映射
- 常见误区
- 前置断点

当前正式基础是包含知识、能力、易错与提升的版本化统一知识库，以及只保存课程覆盖和顺序的 `CourseKnowledgeMap`。个人状态只作为覆盖层叠加，不改写正式库。

### 4.3 状态层

状态层负责描述学习者当前状态。

它可以从 `LearnerContext` 和 `tutor_service` 继续演进，但要比现在更结构化。

建议未来形成统一的 `LearnerState`：

```text
用户是谁
当前课程是什么
当前节点是什么
每个概念的掌握度
近期错题和误区
当前困惑程度
适合的讲解方式
下一步风险
```

### 4.4 决策层

决策层是未来的核心新增能力。

它不直接写长文，而是输出一个教学动作。

例如：

```text
动作：回到前置知识
原因：用户问题暴露出对“函数邻域”的理解不足
依据：连续两次在极限定义中混淆自变量趋近和函数值趋近
执行：先用图像解释邻域，再回到 epsilon-delta
```

这比直接让 AI 回答更稳定，也更像老师。

### 4.5 执行层

执行层负责把决策变成具体内容。

当前已有能力可以归入执行层：

- `CourseService`：生成和重写课程内容。
- `AIQAService`：流式回答问题。
- 测验服务：生成练习。
- 图表服务：生成图表。
- 摘要服务：生成复盘。

未来执行层不应自己决定“为什么做”，而应执行决策层给出的教学动作。

### 4.6 反馈层

反馈层负责把用户行为转化为学习证据。

建议统一抽象为学习事件：

```text
用户提问
用户回答题目
用户保存笔记
用户标记不懂
用户完成节点
用户请求简化
用户复习完成
```

每个事件都应能更新状态层和记忆层。

### 4.7 记忆层

记忆层负责长期沉淀。

它不应该直接等于 prompt 文本，而应该先是结构化事实，再按场景被转成 prompt。

未来记忆层可以分为：

- 事件账本：发生过什么。
- 状态快照：当前判断是什么。
- 教学经验：哪种讲法有效。
- 长期画像：稳定偏好和学习风格。

## 5. 模块升级方向

### 5.1 课程生成升级

目标：从“生成课程内容”升级为“生成可教学的知识结构”。

要提升：

- 大纲中明确每节学习目标。
- 每节绑定关键概念。
- 每节声明前置依赖。
- 每节列出常见误区。
- 每节有最小验收标准。
- 练习题能映射到概念和误区。

判断标准：

```text
不是看内容写得多不多，
而是看系统能不能用这些结构做教学决策。
```

### 5.2 AI 助手升级

目标：从“问答助手”升级为“教学动作执行者”。

要提升：

- 回答前先判断用户问题类型。
- 判断是否需要回到前置知识。
- 判断是否应该追问而不是直接讲完。
- 回答后给出下一步动作。
- 把问答结果转成学习事件。

未来 AI 助手的输出不只是正文，还应包括：

```text
本次教学动作
使用的依据
建议更新的学习状态
建议生成的下一步任务
```

### 5.3 记忆升级

目标：从“上下文拼接”升级为“学习状态资产”。

要提升：

- 笔记、错题、提问、测验、复习都进入统一事件账本。
- 每个事件都能关联课程、节点、概念和时间。
- 记忆能区分事实、推断和稳定偏好。
- prompt 中只注入本次需要的记忆。

核心原则：

```text
记忆不是越多越聪明。
能选对记忆，才聪明。
```

### 5.4 评估升级

目标：从“生成题目”升级为“判断掌握”。

要提升：

- 学前诊断。
- 节点理解检查。
- 随堂小题。
- 错题归因。
- 阶段测评。
- 复习效果追踪。

每次评估都应该回答：

```text
用户掌握了什么？
没有掌握什么？
问题属于概念、推理、记忆还是迁移？
下一步该怎么教？
```

### 5.5 导师系统升级

目标：从“提醒和建议”升级为“学习调度器”。

要提升：

- 根据学习者状态生成今日学习建议。
- 根据错题和遗忘曲线安排复习。
- 根据目标和进度调整路径。
- 根据学习阻塞推荐前置节点。
- 能解释每个建议的原因。

### 5.6 产品体验升级

目标：从“看课程和聊天”升级为“学习驾驶舱”。

未来界面应让用户清楚看到：

- 我现在学到哪。
- 我为什么学这一节。
- 当前知识点依赖什么。
- 我哪里掌握得不好。
- AI 为什么建议我做下一步。
- 我最近有哪些进步和反复错误。

产品不应只展示内容，也要展示学习过程。

## 6. 关键数据对象建议

为了避免继续在旧框架里乱长，建议未来逐步明确这些对象。

### 6.1 学习事件

用于记录用户发生过什么。

字段方向：

- 用户
- 课程
- 节点
- 概念
- 事件类型
- 事件内容
- 结果
- 时间
- 置信度

### 6.2 概念状态

用于描述用户对某个概念的掌握。

字段方向：

- 概念 ID
- 掌握度
- 置信度
- 最近证据
- 常见误区
- 最近一次复习时间
- 下一次建议动作

### 6.3 教学动作

用于表达系统决定做什么。

字段方向：

- 动作类型
- 动作原因
- 依据证据
- 目标概念
- 执行方式
- 预期结果

### 6.4 记忆条目

用于沉淀长期记忆。

字段方向：

- 记忆类型
- 来源事件
- 稳定程度
- 适用场景
- 摘要
- 最近更新时间

## 7. 推进路线

### 阶段一：统一学习事件账本

先不要急着重做全部架构。

第一步应该把所有学习行为统一成事件：

- 提问
- 保存笔记
- 做错题
- 测验完成
- 节点完成
- 请求简化
- 请求扩展
- 复习完成

意义：

```text
没有统一事件，就没有统一状态。
没有统一状态，就没有教学决策。
```

### 阶段二：建立学习者状态模型

在事件基础上，形成可查询的状态：

- 当前课程进度
- 节点掌握度
- 概念掌握度
- 薄弱点
- 错题原因
- 学习偏好
- 当前阻塞

这个阶段的重点不是 AI 生成，而是状态可信。

### 阶段三：增加教学决策层

先做一个轻量规则版，不必一开始就复杂。

例如：

- 连续问“不懂” → 简化解释 + 给例子。
- 同一概念做错两次 → 回到前置知识 + 生成相似题。
- 用户保存笔记 → 更新兴趣和关注点。
- 节点测验低分 → 标记薄弱点并安排复习。

之后再让 AI 辅助生成更细的教学策略。

### 阶段四：让 AI 助手执行教学动作

把 AI 助手从“直接回答”改成：

```text
读取教学动作
→ 生成对应回答
→ 输出 metadata
→ 产生学习事件
```

这样助手会更稳定，也更容易解释。

### 阶段五：升级课程知识结构

把课程节点进一步绑定概念、前置依赖、练习和误区。

这会让系统能做更强的判断：

- 为什么用户卡在这里。
- 应该回到哪个前置节点。
- 应该出什么题。
- 当前回答应该引用哪段原文。

### 阶段六：形成学习驾驶舱

当状态和决策稳定后，再升级前端体验。

否则 UI 只会展示一堆不可靠的指标。

学习驾驶舱应展示：

- 当前路径
- 掌握地图
- 薄弱点
- 下一步建议
- 建议原因
- 学习趋势

## 8. 下一阶段最小可行目标

建议下一阶段不要一口气做完整理想系统，而是先做一个可验证的闭环。

最小闭环可以定义为：

```text
用户学习一个节点
→ 用户提问或做题
→ 系统记录学习事件
→ 系统更新节点/概念状态
→ 系统生成下一步教学建议
→ AI 助手根据建议执行一次教学动作
```

这条闭环跑通后，系统才真正从“生成内容”进入“推动学习”。

## 9. 架构判断规则

后续每个新功能都要问五个问题：

1. 它属于学习闭环的哪一环？
2. 它产生了什么学习证据？
3. 它会更新什么学习状态？
4. 它会影响什么教学决策？
5. 它是否让系统更像一个长期老师，而不是一次性内容生成器？

如果一个功能回答不上这五个问题，就不应该优先做。

## 10. 不应该走的路

为了避免重新走偏，需要明确几个不要做：

- 不为了显得高级而引入复杂多 Agent。
- 不把所有历史都塞进 prompt。
- 不让 AI 在没有课程事实约束时自由编课。
- 不只做更长、更漂亮的回答。
- 不把画像当成学习状态。
- 不把测验当成孤立功能。
- 不让前端堆指标，而后端没有可信状态。

系统的复杂度必须服务学习闭环，而不是服务概念包装。

## 11. 一句话总结

下一阶段的方向不是继续把“课件生成器”做强，而是把它升级成：

**以学习目标为起点，以知识结构为地图，以学习者状态为依据，以教学决策为核心，以内容、问答、练习和复习为执行手段的个人学习操作系统。**

这才是后续所有架构升级应该对齐的目标。

## 12. 历史方案终止点

本文原第 12、13 节曾把旧 V1 方案描述为“当前落地主链”，这会误导后续开发继续接入已经退役的对象，因此删除。2026-07-14 起，正式主链固定为：

```text
正式领域对象 + LearningEvent
→ 可重算 LearnerModel
→ LearningRuntime
→ AIContextPackage 按意图读取最小证据
→ AI 回答或白名单领域命令
→ 新事实再触发模型与运行时重算
```

任何旧名称只用于阅读本文前十一节的历史推演，不得恢复为生产服务、前端 Store、兼容路由或事实写入口。
